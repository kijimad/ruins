# ゲームログのグローバル変数廃止とECSシングルトン化

## 背景

`gamelog.GameLog` がパッケージレベルのグローバル変数として定義されており、約50箇所から直接参照されている。テストの独立性がなく、暗黙的な依存関係になっている。

```go
// var.go
var GameLog = NewSafeSlice(GameLogMaxSize)

// 利用側（約50箇所）
gamelog.New(gamelog.GameLog).PlayerName("Hero").Append("が攻撃した").Log()
```

将来的には `Dungeon`, `GameProgress` など Resources の動的データも同様にECSシングルトンへ移行し、Resources は静的設定のみに絞りたい。

## 設計

### ログストレージの保持方式

キューを1エンティティに持たせる方式を採用する。

1メッセージ1エンティティ方式も検討したが、以下の理由で不採用とした。

- ログは「集合として意味がある」データで、操作の単位がコレクション全体（最新N件取得、FIFO削除、カウント）である。個々のメッセージを独立操作するユースケースがない
- 毎フレーム描画で最新5件を取得する処理が、キュー方式はスライス末尾取得で済むが、1メッセージ方式はJoin+ソートが必要になる
- ターン別フィルタ、カテゴリ検索など ECSクエリの恩恵を受ける場面が現状存在しない

### ECSシングルトンエンティティ

1つのシングルトンエンティティに、ワールド状態を表すコンポーネントをすべて付与する。GameLog, Dungeon, GameProgressなどは同じライフサイクル（InitWorldで生成、セッション終了で破棄）を共有するため、1エンティティで管理するのが自然である。

シングルトンごとに別エンティティにする方式も検討したが、個別に生成・破棄するユースケースがないためmapによるID管理は不要な複雑さとなる。

```go
// components.go
type GameLog struct {
    Store *gamelog.SafeSlice
}

// Components に追加
GameLog *ecs.SliceComponent
```

### シングルトンアクセスの最適化

初期のヘルパー関数方式（`worldhelper.GameLog(world)`）では、毎回 `Manager.Join().Visit()` で全エンティティを走査してシングルトンを探す必要があった。シングルトンに対してJoinクエリを使うのは意味的にも効率的にも不適切であるため、エンティティIDキャッシュとWorldメソッドを導入する。

#### 検討した方式

| 方式 | アクセス | 利点 | 欠点 |
|------|----------|------|------|
| A. Worldメソッド+エンティティキャッシュ | `world.GameLog()` | ECS一貫性を保ちつつO(1)アクセス。メソッドなので「導出アクセス」であることが明示的 | シングルトンが増えるとWorldにメソッドが増える |
| B. 型安全なレジストリ（Generics） | `singletons.Get[T](world)` | 型ごとに1つの制約が自然 | Go的でない。リフレクション使用 |
| C. Worldに直接フィールド追加 | `world.GameLog` | 最もシンプル | Resourcesにフィールドを追加するのと本質的に同じ。World構造体が雑多になる |
| D. Resourcesに追加 | `world.Resources.GameLog` | 既存の仕組みを活用 | Resourcesの責務が曖昧になる。ポインタ経由でアクセスしにくい |
| E. Singletons構造体を新設 | `world.Singletons.GameLog` | 名前空間で整理 | C/Dと構造的に同じ |

**A案を採用する。**

#### エンティティIDのキャッシュ場所

シングルトンエンティティのIDは初期化時に設定され、以降は変更されない静的データである。このためResourcesに `Singleton` フィールドとして保持する。

1エンティティに全シングルトンコンポーネントを載せるため、キャッシュするのはエンティティID1つだけでよい。mapやキー定数は不要になる。

```go
// resources.go
type Resources struct {
    // 静的
    ScreenDimensions *ScreenDimensions
    SpriteSheets     *map[string]components.SpriteSheet
    Fonts            *map[string]Font
    Faces            *map[string]text.Face
    UIResources      *UIResources
    RawMaster        *raw.Master
    Singleton        ecs.Entity // シングルトンエンティティIDキャッシュ

    // 動的
    Dungeon      *Dungeon
    GameProgress *GameProgress
}
```

Worldメソッドはこのエンティティからコンポーネントを取得する。

```go
// world.go

// GameLog はGameLogシングルトンのストアを返す
func (w World) GameLog() *gamelog.SafeSlice {
    return w.Components.GameLog.Get(w.Resources.Singleton).(*gc.GameLog).Store
}
```

InitWorldでのエンティティ生成時にキャッシュする。

```go
func InitWorld(c *gc.Components) (World, error) {
    // ...
    world := World{
        Manager:    manager,
        Components: c,
        Resources:  resources.InitGameResources(),
        // ...
    }

    // シングルトンエンティティを生成してキャッシュする
    // ワールド状態を表すコンポーネントをすべてこのエンティティに付与する
    singleton := manager.NewEntity()
    singleton.AddComponent(c.GameLog, &gc.GameLog{
        Store: gamelog.NewSafeSlice(gamelog.GameLogMaxSize),
    })
    // 将来: singleton.AddComponent(c.Dungeon, &gc.Dungeon{...})
    world.Resources.Singleton = singleton

    return world, nil
}
```

#### 設計の利点

- **O(1)アクセス**: エンティティIDフィールド参照でJoin/Visit不要
- **ECS一貫性**: データはECSエンティティとして保持される
- **単純さ**: mapもキー定数も不要。フィールド1つとメソッドだけで完結する
- **拡張性**: シングルトン追加時はコンポーネントを同じエンティティに付与し、Worldにアクセサメソッドを追加するだけ
- **メソッドによる導出アクセス**: `world.GameLog()` は内部のECSエンティティから取得していることが明示的になる

### 利用側の変更

ヘルパー関数 `worldhelper.GameLog(world)` は廃止し、Worldメソッドに置換する。

```go
// Before（ヘルパー関数方式）
gamelog.New(worldhelper.GameLog(world)).PlayerName("Hero").Log()

// After（Worldメソッド方式）
gamelog.New(world.GameLog()).PlayerName("Hero").Log()
```

`worldhelper` パッケージへの依存が不要になるため、importの整理が発生する。`worldhelper/game_log.go` と `worldhelper/game_log_test.go` は削除する。

GameLogのテストは `world/world_test.go` に移動し、Worldメソッド経由でのアクセスをテストする。

### Resources の整理方針

Resources の中身を性質で分類し、ゲーム状態のみECSシングルトンへ移行する。

| データ | 現在 | 移行先 | 理由 |
|--------|------|--------|------|
| `GameLog` | ECSシングルトン（済） | 維持 | ゲーム状態。ライフサイクルがセッションに紐づく |
| `Dungeon` | Resources（動的） | ECSシングルトン | ゲーム状態。フロア進行・マップデータ |
| `GameProgress` | Resources（動的） | ECSシングルトン | ゲーム状態。セーブ対象 |
| `ScreenDimensions` | Resources（静的） | 維持 | 描画インフラ。エンティティとしての意味がない |
| `SpriteSheets` / `Fonts` / `Faces` | Resources（静的） | 維持 | 静的アセット。ゲームロジックと無関係 |
| `UIResources` | Resources（静的） | 維持 | UI描画用。ECSクエリの対象にならない |
| `RawMaster` | Resources（静的） | 維持 | マスターデータ定義。読み取り専用 |
| `Singleton` | Resources（静的） | − | エンティティIDキャッシュ。イミュータブルな参照 |

### テスト

gamelog パッケージ内のテストは既にローカルストアを使っているため変更不要。

```go
// 既存のテストパターン（変更不要）
store := NewSafeSlice(GameLogMaxSize)
New(store).Append("テスト").Log()
```

利用側のテストはWorld生成時にGameLogエンティティが自動生成されるため、テスト間で状態が共有されなくなる。

## コメント

- まずGameLogから移行し、パターンが確立したらDungeon, GameProgressに広げる
- gamelog パッケージから `var.go`（グローバル変数定義）を削除するのがゴール
- Worldメソッド方式はヘルパー関数方式よりも簡潔で、`worldhelper` への依存も減る
- シングルトン追加時は (1) コンポーネントを同じエンティティに付与 (2) Worldにアクセサメソッド追加 の2ステップで完了する
