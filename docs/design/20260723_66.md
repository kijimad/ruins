# オーバーワールドのチャンク類型化と街・建物の決定的生成

## 背景

現在の街はオーバーマップの地物になったが、実装は仮である。`internal/overworld/town.go` の
`spawnTown` が開始チャンクの中心へ、会話NPC 3体と収納 prop を**ハードコードの相対座標で直接配置**する。
呼び出しは帯生成の外、`driver.go` の `startNewBand` からの一発で、`stage.Bind` でオーバーワールド帯へ
束縛するだけ。結果として街は開始チャンクにしか存在せず、帯を東西へストリーミングしても以降のチャンクには
街も建物も現れない。

本ドキュメントは、この仮実装を「**チャンクごとに類型を決定的に定義し、街・建物をチャンク生成の一段として
生成する**」機構へ置き換える設計を残す。以前メモした「チャンク類型の将来課題」の実装にあたる。

参考として `./Cataclysm-DDA` の overmap 都市生成を調査した。採るべき設計語彙と、あえて捨てる部分を
本設計の判断根拠にする。

## スコープ

本設計は**オーバーマップの街・建物生成に限定する**。オーバーワールドのチャンクに街と建物を決定的に
生成する、その1点に絞る。遺跡・ダンジョン側の生成、および遺跡入口をチャンク類型で帯全域へ撒く案は
**本ドキュメントの対象外**とする。遺跡入口の全域配置は Phase 8 の `PortalConnection` 結線と、各入口への
決定的な遺跡定義割り当てを別途要するため、将来の別ドキュメントで扱う。設計本文が挙げる `ChunkRuinField`
類型は、末尾の追記でスコープ限定に合わせて削る。

## 前提の再確認

### ruins の現状

- **チャンク生成は決定的**。`internal/overworld/chunkgen.go` の `NewChunkGen` が返す closure は、
  `chunkIndex` ごとに `ChunkSeed(runSeed, chunkIndex)` から splitmix64 系でチャンク seed を導き、
  `mapplanner.Plan` → `mapspawner.SpawnAt` で決定的に地形を生成する。
- **街は生成の外にある**。`spawnTown` は `startNewBand` から開始チャンクにだけ呼ばれ、チャンク生成
  パイプラインに乗っていない。
- **帯は1次元**。`chunkIndex` に沿った東西方向のみストリーミングし、`chunkH` は固定。帯シフトで
  西端を捨て東端を生成し、再訪すると同一 seed から再生成される。

### Cataclysm-DDA の都市生成、調査からの事実

- **2段階**。`overmap::place_cities` で都市中心を決め、`build_cities` → `build_city_street` で
  街路と建物を展開する。`src/overmap_city.cpp`。
- **都市の数と配置**。被覆率 `1 / 2^city_spacing` から `roll_remainder` で都市数を確率的に決め、
  デフォルト地形の候補タイルから選び、半径2以内を候補除去して最小間隔を作る。中心は必ず十字路。
  サイズは tiny/small/large/huge の分布で振り、`[2, 55]` にクランプ。
- **街路成長**。中心から4方位へ、都市サイズ `cs` を再帰ごとに `rng(1,3)` 減衰させながら伸ばし、
  左右へ枝道を再帰生成、`cs < 2` で自然停止する。
- **建物の種別切替**。道路沿いに 3/4 の確率で配置し、中心からの正規化距離 `town_dist` が 0〜100。
  `shop_radius`・`park_radius` を正規分布で揺らした閾値と比較し、**中心寄りは shop、中間は park、
  外周は house** を選ぶ。各テーブルは `[建物ID, 重み]` の重み付き。houses は概ね重み 50、
  shops は grocery=1500 と一桁大きい。`region_settings_city` と `regional_settings.cpp`。
- **配置制約は2系統**。都市内は重みテーブル、都市外の特殊構造物は `overmap_special` が
  `city_distance [min,max]`・`city_sizes [min,max]`・`occurrences [min,max]`・`locations` で宣言する。
- **決定性の要点、最重要**。CDDA で座標のみから決まる純粋決定なのは地形の simplex ノイズだけ。
  **都市・街路・建物はその overmap を生成する時点のグローバル乱数状態に依存**し、per-overmap の
  再シードは無い。生成順序が変わると同一 seed でも街が変わりうる。

## 要件

- チャンクごとに類型を決定的に定義する。同じ `runSeed`・`chunkIndex` なら常に同じ類型、同じ街配置に
  なる。帯シフト・再訪・reload をまたいで不変。
- 街・建物の配置は `ChunkSeed(runSeed, chunkIndex)` を唯一のエントロピー源にする。CDDA のような
  グローバル乱数状態への依存、すなわち生成順序依存は持ち込まない。
- 街内の建物は、中心からの距離で種別を切り替え、重み付きテーブルから選ぶ。CDDA の翻案。
- 街機能である店・雇用・合成は prop 化した相互作用で提供する。既存 `town.go` の方針を一般化する。
  収納 stash は設けない。末尾の追記、課題1の決定を参照。
- 仮の `spawnTown` を置き換える。開始チャンクの街も、この一般機構の1インスタンスとして表す。
- 共存方式・serde と整合する。街の地物は `StageBound` でオーバーワールドへ束縛され、遺跡進入で退避、
  帰還で復元される。既存の束縛方針を踏襲する。

## 設計

### 全体像 — チャンク生成に「地物レイヤ」を挿す

`NewChunkGen` の closure に、地形生成の後段として「地物 feature の配置」を足す。段は次の順になる。

1. 地形を生成する。`mapplanner.Plan` → `mapspawner.SpawnAt`。現状のまま。
2. **チャンク類型に応じて地物を配置する。街、遺跡入口など。新設。**
3. フィールドエンティティをオーバーワールド帯へ束縛する。`stage.Bind`。現状のまま。
4. 継ぎ目を再計算する。`RecalcSeamAutotile`。現状のまま。

地物レイヤは `ChunkSeed(runSeed, chunkIndex)` から派生した seed だけを使い、チャンク単体で完結させる。
これで CDDA が持てなかった「座標のみで決まる街」を、帯の東西ストリーミングと再訪に耐える形で実現する。

### 1. チャンク類型 `ChunkKind` を決定的に導く

> **注記(改訂)**。この節の `ChunkKind` 3値 enum は初稿。末尾の追記(1)課題2で述語 `IsTownChunk` に、
> 追記(2)で2次元 `ChunkCoord` に改めた。**実装は追記側に従う**。この節は判断の経緯を残すための初稿記録。

`chunkIndex` から類型を決める純関数を置く。乱数状態でなく `ChunkSeed` だけに依存させ、生成順序から
独立させる。ここが CDDA との最大の分岐点になる。

```go
// ChunkKind はオーバーワールドの1チャンクが担う役割。地形の上に何を配置するかを決める。
type ChunkKind int

const (
	ChunkWilderness ChunkKind = iota // 原野。地物を置かない
	ChunkTown                        // 街を生成する
	ChunkRuinField                   // 遺跡入口を置く
)

// ChunkKindAt は chunkIndex の類型を決定的に返す。ChunkSeed のみに依存し、生成順序に依存しない。
// density は town の出現頻度で、CDDA の city_spacing の翻案。1次元帯では被覆率でなく
// 「およそ density チャンクに1つ town」という頻度で表す。
func ChunkKindAt(runSeed uint64, idx consts.Chunk, cfg ChunkKindConfig) ChunkKind

// ChunkKindConfig は類型付けのパラメータ。マスタ設定として持つ。
type ChunkKindConfig struct {
	TownEveryN     int // town のおおよその出現間隔。CDDA city_spacing 相当
	TownMinGap     int // 隣接 town を避ける最小 index 間隔。CDDA の最小間隔2の翻案
	RuinFieldEveryN int // 遺跡入口の出現間隔
}
```

`TownMinGap` は近傍 index の類型を見て隣接 town を抑制する。CDDA が候補除去で作る最小間隔を、
1次元では index 距離のしきい値で表す。判定は決定的なハッシュのしきい値比較にし、乱数状態を持たない。

### 2. 街レイアウトの決定的生成

`ChunkTown` のチャンク内に街を生成する。`ChunkSeed(runSeed, idx)` から派生した seed で、中心・建物枠・
種別・prop を順に決める。すべて決定的。

CDDA の再帰的な街路成長は 180x180 の overmap 向けで、ruins の 50x50 タイル1チャンクには過剰である。
まずは簡易レイアウトで始める。中心に広場を置き、その周囲に建物枠を格子状に取り、枠ごとに地物を選ぶ。
本格的な街路成長は後続の拡張とし、常に動く状態を保つ。

建物の種別は、中心からの距離で切り替え、重み付きテーブルから引く。CDDA の `town_dist` と
`shop_radius`/`park_radius` の翻案。ruins では簡略化し、中心寄りは service、外周は house の2段でよい。

```go
// TownTable は街に置ける地物の重み付き分布。CDDA の houses/shops/parks テーブルの翻案。
// 中心からの距離帯ごとに別テーブルを引く。
type TownTable struct {
	Services []WeightedFeature // 中心寄り。店・雇用・合成・収納など機能を持つ地物
	Houses   []WeightedFeature // 外周の装飾建物。相互作用なし。初期実装では持たず、追記(3)Houses の扱いに従い後で足す
}

// WeightedFeature は1つの地物候補と、その相対的な出やすさ。
type WeightedFeature struct {
	Name   string // prop 名またはNPC定義名
	Weight int    // 重み。大きいほど出やすい。CDDA の [id, weight] 翻案
}

// TownLayout は決定的に確定した街の配置結果。生成の入力でなく出力。
type TownLayout struct {
	Center   consts.Coord[consts.Tile]
	Features []PlacedFeature
}

// PlacedFeature は確定した1地物の配置。
type PlacedFeature struct {
	Name string
	Pos  consts.Coord[consts.Tile]
}
```

### 3. 街機能の prop 化

既存 `town.go` の NPC・収納の方針を `TownTable` の Services 枠へ一般化する。店は商人NPC、雇用は
酒場の主人、合成は怪しい科学者、収納は木箱。相互作用は既存の `InteractionTalk`・`InteractionStorage`
をそのまま使う。街を専用ステージからオーバーマップ地物へ移す方針は Phase 8 で確定済みで、本設計は
その地物を「チャンク生成が決定的に置く」段へ進めるだけ。

### 4. `spawnTown` の置換と開始チャンク特例

`driver.go` の `startNewBand` からの `spawnTown`・`SpawnDungeonEntrance` の直接呼びを撤去し、
地物レイヤへ寄せる。新規ゲームの開始点には街が要るので、開始チャンクは必ず `ChunkTown` にする特例を
`ChunkKindAt` 側で表す。CDDA のメガシティが中心配置を乱数でなく固定点にする、あの決定的特例に相当する。

### 5. 遺跡入口の類型化

現状 `SpawnDungeonEntrance` も開始チャンクへの一発配置である。これを `ChunkRuinField` 類型として
地物レイヤへ寄せ、帯全域に遺跡入口が点在するようにする。街と入口が同じ「チャンク類型 → 地物」機構に
乗る。

## データ定義

変更の骨子は「生成の外の一発置き」から「チャンク生成内の決定的な類型 → 地物」への移行である。

```go
// 現状: 生成の外。開始チャンクにハードコード配置
//   driver.startNewBand:
//     spawnTown(world, center)              // NPC/収納を相対座標で直接置く
//     SpawnDungeonEntrance(world, cx+2, cy) // 入口も一発

// 目標: 生成の内。チャンク seed から決定的に
//   NewChunkGen closure(chunkIndex, offsetX):
//     地形生成
//     switch ChunkKindAt(runSeed, chunkIndex, cfg) {
//     case ChunkTown:       placeTown(world, ChunkSeed(runSeed, chunkIndex), ...)
//     case ChunkRuinField:  placeRuinEntrance(world, ChunkSeed(runSeed, chunkIndex), ...)
//     }
//     stage.Bind / RecalcSeamAutotile
```

## 採用しなかった方法

- **CDDA のグローバル乱数方式をそのまま移植する**。都市・街路・建物を生成時のグローバル乱数状態から
  引く。生成順序に依存するため、帯の東西ストリーミング・再訪・reload で街が変わる。ruins の per-chunk
  決定シードを捨てることになり、共存・serde と最も相性が悪い。ruins の `ChunkSeed` は CDDA の
  この弱点の上位互換なので、決定性の作り方だけは踏襲しない。
- **街を専用ステージにする**。街を遺跡と同様の別ステージにし入口から入る。Phase 8 で街をオーバーマップ
  地物にした方針に逆行する。街は地物のまま、生成段だけを進める。
- **2次元オーバーマップ化する**。CDDA の 180x180 OMT グリッド相当を導入し南北もストリーミングする。
  ruins は1次元帯で足り、南北ストリーミングは要件外。過剰で侵襲的。
- **街路成長を最初から完全移植する**。`build_city_street` の再帰分岐は overmap 向けで、50x50 の1
  チャンクには過剰。簡易レイアウトで始め、必要になったら `street.go` で本格化する段階導入にする。
- **配置を runSeed だけで決める**。`chunkIndex` を混ぜないと帯全域で同じ街になり、位置の決定性も
  失われる。`ChunkSeed(runSeed, chunkIndex)` を必須にする。

## コメント

- CDDA から採るのは設計語彙である。中心距離で種別を切り替える重み付きテーブル、出現頻度と最小間隔の
  パラメータ、`locations`・`city_distance` 相当の配置制約。捨てるのはグローバル乱数依存の決定性で、
  ここは ruins の `ChunkSeed` に置き換える。
- まず `ChunkKindAt` と `TownTable` の骨格を入れ、簡易街で常に緑を保つ。街路成長は後続に回す。
- 段階は Phase 8 と同じ方針を守る。代替を入れてから撤去し、常にゲームが動く状態を保つ。仮 `spawnTown`
  を壊さずに `TownTable` ベースへ置換し、類型付けで帯全域へ広げ、遺跡入口も類型へ寄せ、最後に街路成長を
  足す。

## 追記(2026-07-23): 批判レビューで判明した課題と素案

初稿を実コードに照らして批判的にレビューし、6点の課題が出た。スコープをオーバーマップに限定した
うえで素案を検討し、設計を改める。以下は初稿の該当箇所を上書きする。

### スコープ限定の反映

- 本設計はオーバーマップの街・建物生成に限定する。冒頭「スコープ」節のとおり。
- 遺跡入口の類型化 `ChunkRuinField` は対象外とし、設計本文の3値 `ChunkKind` から落とす。これにより初稿
  §5「遺跡入口の配線が既存 enter/exit と未接続」は本ドキュメントの課題から外れる。

### 課題1(最重要): 街の永続状態と帯シフトの破棄・再生成が両立しない

- **事実**。帯シフトはエンティティを退避でなく破棄する。`band.go` の `ShiftEast`/`ShiftWest` が
  `RemoveEntitiesInXRange` で端チャンクの街 NPC・prop を削除し、再訪時に `gen` が新品を再生成する。
  初稿の「`StageBound` で退避・復元」は遺跡進入時の suspend/restore の話で、帯シフトには当てはまらない。
  suspend/restore と destroy/regenerate は別寿命の2機構で、街は後者に晒される。
- **帰結**。収納 prop に預けたアイテムや倒した NPC など、プレイヤーが加えた変更がシフト往復で失われる。
  開始チャンクの収納も、帯が K チャンク東進すれば西端で破棄される。初稿の「serde と整合」は表層的で、
  決定的な街を serde するのか seed から再生成するのかを決めていなかった。
- **素案。推奨は案A**。

  ```text
  案A(推奨): サービスを状態の有無で二分する。
    - 無状態 store/hire/craft: 帯の街で自由に再生成してよい。会話メニューを開くだけで
      インスタンス状態を持たないため、破棄・再生成しても損失がない。
    - 有状態 収納 stash: per-chunkIndex の可変オーバーレイとして serde し、再生成時に
      seed 由来の幾何へ差し戻す。幾何は決定的、可変差分だけ永続。

  案B(簡素): 帯の街に永続機能を置かない。収納など永続が要る機能は帯シフトの破棄対象で
    ない開始拠点だけに持たせ、帯の街は一時サービス(店・雇用・合成)に限る。実装は最小だが
    遠征先で預けられない。

  案C(不採用寄り): 街を帯シフトの破棄から除外し真の退避にする。だが帯はリベースで全
    エンティティを平行移動するため、除外した街は帯外へドリフトする。リベース機構と噛み
    合わず侵襲的。
  ```

- **決定(2026-07-23)。案B を採る。stash は設けない**。街は永続状態を持たない通過点とし、フィールドに
  アイテムを保持しない。収納 stash 機能を街から外し、サービスは店・雇用・合成の無状態メニューだけにする。
  これで serde すべき可変状態が無くなり、seed からの決定的再生成と完全に整合する。案A の per-chunkIndex
  オーバーレイは不要になり、課題1は最も単純な形で解ける。案C も検討不要になった。

### 課題2: `ChunkKind` enum は却下方針の再来 → 述語へ畳む

- doc 65 は却下案で「enum で種別を持つ。Phase F で撤去した `StageKindDungeon` 系の再来」と明記した。
  初稿の `ChunkKind int`(iota)はこの反パターンの再導入だった。
- スコープをオーバーマップ限定にすると類型は「街か否か」の2値で足り、enum は不要になる。3値 `ChunkKind`
  を廃し、純関数の述語にする。

  ```go
  // IsTownChunk は chunkIndex が街チャンクかを決定的に返す。ChunkSeed のみに依存する。
  func IsTownChunk(runSeed uint64, idx consts.Chunk, cfg TownGenConfig) bool

  type TownGenConfig struct {
      Density int // 街の出やすさ。おおよそ Density チャンクに1つ街。CDDA city_spacing の翻案
      MinGap  int // 隣接街を抑制する窓半径。CDDA の最小間隔の翻案
  }
  ```

- 閾値の定め方。`townScore` は `[0, maxUint64]` の一様値とみなし、`townScore(c) < maxUint64 / Density` を
  街候補の閾値にする。これで平均しておよそ `Density` チャンクに1つが候補になる。候補のうち窓 `MinGap` で
  最大の1つだけを街にして隣接を抑える。`Density` の単位はチャンク間隔、`MinGap` は窓半径のチャンク数。

- これで種別 enum の分岐を持たず、§2 と §5 が同時に解ける。

### 課題3: 隣接抑制の決定的アルゴリズムを具体化する

- ストリーミング生成は overmap 全体を持てないので、CDDA の候補除去(`points_in_radius` による除去)は
  使えない。局所ハッシュの窓最大で表す。

  ```go
  // townScore は idx の「街らしさ」を表す生ハッシュ。ChunkSeed 派生で決定的。
  func townScore(runSeed uint64, idx consts.Chunk) uint64

  // idx が街 ⇔ townScore(idx) が閾値超え、かつ窓 [idx-MinGap, idx+MinGap] で最大。
  // 近傍は生スコアを引き直すだけで比較でき、他チャンクの最終判定に依存しない。
  // よって1枚生成中でも O(MinGap) で決定的に判定でき、再帰も全域スキャンも不要。
  ```

- 閾値と `MinGap` は `cfg`(TownGenConfig)が持つ。`townScore` は生ハッシュに徹し、`IsTownChunk` が cfg の
  閾値超え判定と窓最大判定を行う。ゆえに `townScore` は unexported でよく、閾値を知らなくてよい。
- 帯端の扱い。`eastIndex=0` 付近は窓が片側だけになるが、開始チャンク特例(課題4)が優先するので破綻しない。
  Y 方向の端は追記(3)で扱う。

### 課題4: 開始チャンクの必須サービスを保証する

- 開始チャンクは「必ず街」かつ「店・雇用・合成を確定配置」にする。weighted 抽選に委ねると開始の
  街に必須サービスが揃わない恐れがあるため、開始の街だけは core サービスを固定枠で置く。CDDA のメガ
  シティが中心配置を乱数でなく固定点にする決定的特例を、サービス構成にまで及ばせる。
- 収納 stash は課題1の決定により設けないので、必須サービスからも外す。

### 課題5: 遺跡入口の配線が既存 enter/exit と未接続 → スコープ外で解消

- 初稿 §5 の課題は、スコープをオーバーマップに限定したことで本ドキュメントの対象から外れた。遺跡入口の
  類型化と `PortalConnection` 結線は別ドキュメントで扱う。番号を飛ばさないためここに明示する。

### 課題6: 「ChunkSeed が唯一のエントロピー源」を不変条件として強制する

- 街生成の内部でパッケージレベルのグローバル `math/rand` を使わないことを不変条件にし、テストで固める。
  ChunkSeed から `*rand.Rand` を作り引数で通し、グローバル乱数を禁じる。
- **実装前に確認**。既存の `lifecycle.Spawn*`・`mapspawner` がグローバル乱数を使っていないか。使っていれば
  そこが決定性の穴なので、街生成に載せる前に塞ぐ。皮肉にも CDDA が踏んだ罠と同型で、シード純粋性は
  経路全体で守らないと崩れる。

### 妥当だった判断(維持)

- **地物レイヤを `NewChunkGen` の中に置く**。帯シフトが再訪時に `gen` で再生成する以上、街生成も `gen` の
  中でなければ辻褄が合わない。破棄・再生成の寿命に自然に乗る。維持。
- **CDDA のグローバル乱数依存を採らない**。維持。ただし ruins 側の決定性も**幾何に限る**話で、エンティ
  ティの可変状態は課題1で別途永続化する。初稿はこの限定を書かず決定性を過大に述べていた。

## 追記(2026-07-23): 重大要件 — 縦のチャンク行数を可変にする

新たな重大要件が出た。**現在1のオーバーマップの縦チャンク行数を、近く3や5へ増やす**。プレイヤーが東西
一本道でなく、南北に分かれた複数レーンから経路を選べるようにするため。これは街生成の骨格に効くので、
行数が1のうちから2次元前提で設計し、増やしても作り直さないようにする。

### 現状と要件

- **現状**。帯は縦1行。`ChunkGen(chunkIndex, offsetX)` は X だけの1次元で、`ChunkSeed(runSeed, chunkIndex)`
  も1次元キー。`consts.AbsTileX` はあるが `AbsTileY` は無い。南北はストリーミングしない横ストリップ。
- **要件**。縦チャンク行数 `Rows` を 1→3/5 と増やせるようにする。街生成は最初から2次元チャンク座標
  `(cx, cy)` を前提にし、`Rows` の変更で街判定・配置ロジックを作り直さない。東西は従来どおり無限
  ストリーミング、南北は `Rows` に**有界**。全域2次元ストリーミングは要らない。

### 初稿「却下: 2次元オーバーマップ化」の訂正

初稿は2次元化を「1次元帯で足り、南北ストリーミングは要件外。過剰」として却下した。**この判断は本要件で
覆る**。ただし覆るのは「固定高さ `Rows` の複数行帯」までで、CDDA の 180x180 相当の無限2次元 overmap は
依然要らない。区別を明記する。

- **採用に転じる**。固定小行数(3/5)の2次元チャンク格子。X は無限ストリーミング、Y は有界。
- **非採用のまま**。Y も無限にストリーミングする全域2次元。要件外で過剰。

皮肉にも、2次元にすると CDDA の都市配置が**より素直に翻案できる**。CDDA は2次元 OMT 格子で距離・半径・
候補除去を使う。1次元へ潰す無理が消える。

### 街判定を2次元チャンク座標へ

追記(1)の1次元版 `IsTownChunk(idx)` を次で置き換える。

```go
// ChunkCoord はオーバーマップのチャンク座標。X は絶対で東西無限、Y は [0, Rows) に有界。
type ChunkCoord struct {
    X consts.Chunk // 東西。eastIndex 系の絶対インデックス
    Y consts.Chunk // 南北。0..Rows-1
}

// IsTownChunk は (cx, cy) が街チャンクかを決定的に返す。ChunkSeed のみに依存する。
// cy は現在 0 のみだが、Rows を増やしても呼び出しと判定ロジックは不変。
func IsTownChunk(runSeed uint64, c ChunkCoord, cfg TownGenConfig) bool

// townScore も2次元キーへ。窓最大は2次元近傍の Chebyshev 距離 MinGap で取る。
func townScore(runSeed uint64, c ChunkCoord) uint64
```

隣接抑制は2次元窓の最大で表す。CDDA の `points_in_radius(center, 2)` の候補除去に、1次元より近い翻案に
なる。

### 経路選択の設計含意

- 街・地物を複数行に散らすと、プレイヤーは「北レーンの街に寄るか、南レーンで前線を避けるか」を選べる。
  街配置は行にまたがる到達性を意識する。
- 前線(寒波前線)は現在 X 一軸で東進する。複数行でも前線は全行を貫く帯として X で進めばよく、行ごとに
  地形が違うことでレーンごとの難易度差が生まれる。前線ロジック自体は X 一軸のままでよい。

### 帯インフラ側の依存(本ドキュメントの範囲外だが前提)

街生成を2次元前提にしても、実際に `Rows>1` を動かすには帯インフラの変更が要る。これらは別作業とする。

- `Band` に行数 `Rows` と Y 方向のチャンク配置。`ChunkGen` を `(cx, cy, offsetX, offsetY)` へ拡張。
- `AbsTileY` 相当を足すか、Y 有界前提で据え置く。マップ高さ = `Rows * chunkH`。
- シフトは X のみ。X シフトで列(全行)を破棄・生成。Y は据え置きなので破棄・再生成の寿命は課題1の分析の
  ままで、南北移動では街は破棄されない。

- [ ] (別作業・前提) 帯インフラの `Rows` 対応。本ドキュメントは2次元前提の街生成に限る

## 追記(2026-07-23): PR レビュー反映

PR #511 のレビューで、実装が一意に定まらない箇所が指摘された。詰める。

### ChunkSeed の2次元拡張

`chunkgen.go` の `ChunkSeed(runSeed, chunkIndex consts.Chunk)` は1次元。`townScore` を2次元化するには
`Y` を混ぜた2次元シードが要る。別関数として明示し、街判定のシード源を一本化する。

```go
// ChunkSeed2D は runSeed と (cx, cy) から決定的なシードを導く。既存 ChunkSeed の2次元版。
// cx と cy を別々の奇数定数で混ぜてから splitmix64 で撹拌し、隣接や転置((x,y)と(y,x))でも散らす。
func ChunkSeed2D(runSeed uint64, cx, cy consts.Chunk) uint64

// townScore は ChunkSeed2D の値をそのまま返す薄い別名。ChunkSeed2D が既に撹拌済みなので
// 二重には混ぜない。閾値超え判定と窓最大は IsTownChunk が持つ。
func townScore(runSeed uint64, c ChunkCoord) uint64 // = ChunkSeed2D(runSeed, c.X, c.Y)
```

地形生成の `ChunkSeed(runSeed, chunkIndex)` も、帯が `Rows>1` になった時点で `ChunkSeed2D` へ寄せる。
これは帯インフラ側の変更で本ドキュメントの範囲外だが、シード導出を `ChunkSeed2D` に一本化する方針は
ここで示す。**後方互換は保証しない**。`Rows` を増やす移行は世界形状そのものの変更なので、全チャンク
再生成を許容する。ゆえに `cx`・`cy` を独立に混ぜてよく、`cy=0` で従来値と一致させる制約は課さない。
互換を約束すると定数設計が加算方式に縛られ自由度を失うため、あえて保証しない。

### 2次元窓最大の Y 端

`Y` は `[0, Rows)` に有界なので、`Y=0` や `Y=Rows-1` のチャンクは窓が片側だけになる。窓
`[cy-MinGap, cy+MinGap]` を `[0, Rows)` でクランプすればよい。近傍が少ないだけで、窓最大の判定は決定的な
まま成立する。X は無限なのでクランプ不要、Y だけ有界クランプ、という非対称を実装で固定する。

### 近傍評価のコスト

2次元化で窓は `(2*MinGap+1)²` の格子になる。`MinGap=3` なら最大 49 セル。各セルは `ChunkSeed2D` を1回
呼ぶだけの生ハッシュ計算で、地形生成の1チャンク分に比べ無視できる。チャンク生成のたびに `O(MinGap²)` 回の
`ChunkSeed2D` 呼び出しに留まることを実装で保つ。

### Houses の扱い

案B、stash なしでは街の機能は Services の無状態メニューに尽きる。`Houses` は相互作用を持たない外周の
装飾建物で、街を街らしく見せる視覚的変化のためのもの。初期実装では装飾建物 prop が無いので `TownTable`
は `Services` のみで始める。装飾建物 prop が揃った時点で `Houses` を足し、中心距離での種別切替を導入する。
空フィールドを先取りせず、必要になってから型を広げる。YAGNI に従う。

- [ ] `ChunkSeed2D` を追加し `townScore` の唯一のシード源にする。転置・隣接で散ることをテスト。後方互換は保証しない
- [ ] 2次元窓最大の Y 端クランプ。`Rows=3` の `Y=0/2` で片側窓でも判定が決定的なことをテスト
- [ ] 実装順序。`ChunkSeed2D` を先に入れ街判定から使う。地形生成の `ChunkSeed` は帯インフラの `Rows` 対応後に寄せる
