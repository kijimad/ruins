# オーバーワールド生成アーキテクチャ

## 背景

生成ロジックの豊かさはゲームの面白さのキモである。街生成 [20260723_66.md](./20260723_66.md) と
縦チャンク任意数化 [20260723_67.md](./20260723_67.md) は個別機能だが、豊かさ・拡張性・表現性に耐える
横断アーキテクチャが要る。

doc 66 の初稿は生成の seam を「街」という具体に置いていた。地物を種類ごとの述語 `IsTownChunk` で表すと、
街・遺跡・資源・ハザードと増やすたびに `IsXChunk` が並列に生え、合成も優先順位も共有ルールも無い。種類を
増やすほど分岐が増える天井があった。

本ドキュメントは生成を**層のパイプライン**と、**データ駆動に移行できる FeatureSpec レジストリ**として設計し、
街・遺跡・資源・ハザードを同型のインスタンスにする。doc 66 はその最初の実装、Settlement に位置づけ直す。

参考は CDDA と Minecraft。CDDA から中身の重み・配置制約の語彙、Minecraft から決定的チャンク生成とリージョン
配置を採る。各々の強い部分だけ翻案する。

## 最終目的地

生成の射程を確定させる。以降の設計はここへ向かう。

- **バイオーム第一級**。region ごとに地形・敵・気候が変わる。境界は noise で連続に滲む。
- **地物4カテゴリ**。拠点系、遺跡入口、資源・採集ノード、危険地帯・POI・イベント。すべて器に載せる。
- **難易度勾配2軸**。東進距離で基礎難度と報酬、前線距離で寒冷と危険。いずれも座標由来で決定的。
- **クロスチャンク構造**。街をつなぐ道、region をまたぐ大型構造。決定的に協調する。
- **動的生成は射程外**。生成は `(seed, 座標)` の純関数を保つ。プレイヤー状態を入力にしない。難易度勾配も
  座標由来に限る。これが決定性・serde・再訪一致を守る背骨になる。
- **コード先行、データ化可能**。最初は Go 定義、器が固まったら TOML/raw へ移せる形にする。

## 前提の再確認

- doc 67 が2次元帯の座標基盤を提供する。`ChunkCoord`(X 絶対・Y 有界)、`ChunkSeed2D`。
- doc 66 が Settlement の配置(リージョン方式)と中身(`TownTable`)を提供する。
- 既存の temperature/front 系が気候・寒冷を扱う。Biome の気候はここに接続し二重管理を避ける。
- 生成は `NewChunkGen` の closure で1チャンク単位に走る。層はこの中に順に並ぶ。

## 設計

### 1. 生成パイプライン（層）

`NewChunkGen(coord)` を層の列にする。各層は前層の出力に作用し、`ChunkSeed2D` と座標から決定的。

```text
GenerateChunk(coord):
  prog    := ProgressionAt(coord)        // 1. 難易度場（東進・前線の2軸）
  biome   := BiomeAt(coord)              // 2. バイオーム（noise で連続）
  terrain := GenTerrain(coord, biome)    // 3. 地形（biome でパラメータ化）
  PlaceFeatures(coord, biome, prog)      // 4. 地物（FeatureSpec レジストリ）
  DrawConnections(coord)                 // 5. 道・大型構造（近傍を算出して結ぶ）
  Decorate(coord)                        // 6. 継ぎ目・小物
```

各層を差し替え・合成・単体テスト可能な段にする。単層に全部を詰めない。

### 2. Progression 場（難易度2軸）

座標から静的に難易度を導く純関数。動的入力なし。

```go
// Progression は座標由来の難易度。プレイヤー状態に依存しないので決定的。
type Progression struct {
	East  int // 開始からの東進距離由来。基礎難度・報酬 tier
	Front int // 前線からの距離由来。寒冷・危険
}

func ProgressionAt(c ChunkCoord) Progression
```

Biome の気候、敵スケール、Feature の距離帯 Constraints、報酬 tier がこれを読む。

### 3. Biome 層（第一級・noise）

```go
// Biome はステージでなくオーバーワールドの地帯。地形・敵・気候・許可地物を束ねる第一級概念。
type Biome struct {
	Name         string
	Planner      mapplanner.PlannerType   // 地形生成のプランナ
	EnemyTable   string                   // 敵の湧き表
	TempOffset   int                      // 気候。既存 temperature 系へ接続
	AllowFeature func(FeatureKind) bool   // 許可する地物カテゴリ
}

// BiomeAt は noise で連続的に biome を決める。region 離散でなく境界が滲む。
// noise は ChunkSeed2D 派生、または既存の overworld ノイズ相当を用いる。
func BiomeAt(c ChunkCoord) *Biome
```

### 4. Feature 層（FeatureSpec レジストリ）

地物を種類ごとの述語でなく、統一 spec のインスタンスにする。ここが拡張性の芯。

```go
type FeatureKind int

const (
	FeatureSettlement   FeatureKind = iota // 拠点。doc 66 の実装
	FeatureRuinEntrance                     // 遺跡入口
	FeatureResourceNode                     // 資源・採集
	FeatureHazard                           // 危険地帯・POI・イベント
)

// FeatureSpec は1種類の地物の「どこに・どの条件で・何を」を1つにまとめた宣言。
// 種類を増やすのは spec のインスタンスを足すこと。分岐を増やさない。
type FeatureSpec struct {
	Kind        FeatureKind
	Placement   Placement    // どこに撒くか
	Constraints Constraints  // どの biome・距離帯で許すか
	Content     ContentGen   // 何を置くか
}

// Placement は配置ルール。疎な構造は Minecraft のリージョン方式、密な散布は別モード。
type Placement struct {
	Spacing    consts.Chunk // 疎: リージョンに高々1つ。拠点・遺跡
	Separation consts.Chunk // 疎: 最小間隔
	Salt       uint64       // 種類ごとに相関を切る
	// 密な散布（資源）は Density モードを別に持つ
}

// Constraints は配置可否。CDDA の locations/city_distance/occurrences の翻案。
type Constraints struct {
	Biomes      []string // 許可 biome
	EastBand    [2]int   // 東進距離帯 [min,max]。難易度勾配で出現を絞る
	Occurrences [2]int   // 出現回数域
}

// ContentGen は確定した地物の中身を配置する。biome と prog を受け表現を変える。
type ContentGen func(world w.World, c ChunkCoord, seed uint64, b *Biome, p Progression) error
```

- 拠点は `FeatureSpec{Settlement, リージョン Placement, Constraints{平地 biome, ...}, TownContent}`。doc 66 の
  実装が `Placement` と `Content` を埋める。
- 遺跡入口・資源・ハザードは別 spec。当面はコード、将来は TOML で足すだけ。

### 5. Connection 層（クロスチャンク）

`Placement` が純関数なので、あるチャンクは近傍リージョンの拠点位置を**生成せずに算出**できる。道はその位置を
結び、region 大型構造も決定的に配置する。Minecraft が構造物を生成せず座標算出するのと同じ原理。

```go
// NearbyFeatures は生成を伴わず、近傍にある指定種の Feature 位置を決定的に算出する。
func NearbyFeatures(c ChunkCoord, kind FeatureKind, radius consts.Chunk) []ChunkCoord
```

### 6. 決定性とコード→データ

- 全層は `(ChunkSeed2D, 座標, Progression)` の純関数。プレイヤー状態を入力にしない。動的生成は射程外。
- `FeatureSpec`・`Biome` は Go の登録表で始める。フィールドは TOML スキーマへ一対一で写せる形にし、レジストリ
  の interface を安定させる。器が固まったら `raw`/TOML へ移す。

## データ定義

上記の struct 群が骨子。`Progression`・`Biome`・`FeatureSpec`・`Placement`・`Constraints`・`ContentGen` が
生成の語彙で、種類の追加はこれらのインスタンスを足すことに還元される。

## 採用しなかった方法

- **種類ごとの述語** `IsTownChunk`/`IsRuinChunk`…。分岐が増え合成しない。地物を `FeatureSpec` の別インスタンス
  にすれば、種類は同型のデータになる。doc 66 初稿の天井をここで外す。
- **region 離散の biome**。境界が四角く不自然。noise で連続に滲ませる。前線の気候勾配とも馴染む。
- **最初からデータ駆動**。器が固まる前に TOML スキーマを凍結すると手戻りが大きい。コード先行で抽象を固め、
  安定したら移行する。
- **動的生成、プレイヤー反応**。生成が純関数でなくなり、決定性・serde・再訪一致がすべて崩れる。難易度は座標
  由来の Progression に限る。
- **単層生成**。全部を1フックに詰めると太る一方。層に分け、各層を差し替え・テスト可能にする。

## コメント

- 街 doc 66 を最初の FeatureSpec にし、器を1インスタンスで検証してから種類を増やす。常に緑を保つ。
- Biome を第一級にしたので地形・敵・気候が region で変わる。気候は既存 temperature/front に接続し二重管理を
  避ける。
- クロスチャンクは `Placement` の純関数性が支える。生成せず算出できることが道・大型構造の鍵。決定的配置は
  Minecraft の実績ある王道で、CDDA のグローバル乱数依存の弱点を踏まない。
- 拡張は「分岐を増やす」でなく「同型のインスタンスを増やす」で伸ばす。seam を具体でなく抽象に置くのがその条件。

## 追記(2026-07-24): 最終的な方向性の確定

doc 66・67・68 が積み上げた機構は詳細だが、どのゲーム体験へ向かうかの上位像が未確定だった。
その方向性を対話で詰め、以下に確定する。本追記は最終像の宣言で、各 doc の機構はこの像へ奉仕する。

### コア体験 — ゆるやかな制約下のオープン探索

- 主軸はオープンな探索。明確な東ゴールへプレイヤーを追い立てる構造にはしない。
- 前線すなわち寒波は「東ほど強まる難易度勾配」の一つとして据える。探索を方向づけるが縛らない。
  doc 68 の Progression 2軸のうち前線軸は、この勾配として読む。
- **前線ロジックの X 一軸は据え置く**。変えるのはフレーミングだけで、「前線が追い立てる」から
  「東ほど寒冷・高難度の勾配」へ意味を寄せる。doc 67 が前提していた「レーンは並走しゴールが東で
  共通、前線が追う」の追い立て色を薄める。

### 場所の類型 —— いずれも特別扱いせず同型の地物にする

現実寄りの世界観に合わせ、街を一様な補給点とはしない。`lore.md` の集落モデルに従い、規模と危険で分ける。

- **小集落・一軒家**: 生存者がいる交易・補給・改造・雇用・情報・休息の場。村・農家・峠の一軒家など。
  規模が小さく比較的安全で、生き残りと商うキャラバンの相手。無状態で stash なし。開始拠点もこの類型。
  doc 66 案B の無状態・stash なしを最終形として確定する。seed からの決定的再生成と完全に整合する。
- **市街地・大都市**: 地表の高リスク・高リターン地帯。**規模が大きいほどモンスターも戦利品も多い**。
  複数チャンクに及ぶ地表の攻略対象で深さ y は持たない。補給の場ではない。以前の追記で「基地」と呼んだ
  地表大型構造を、市街地・大都市として一般化した本カテゴリ。自衛隊基地跡や研究施設もこの危険帯に属する。
- **遺跡・ダンジョン**: 深さ y を持つ階層ステージ。入口から地下へ潜る。y を足すだけの一類型で特別ではない。
  doc 65 Phase 8 の「遺跡は隣のステージにすぎない」統合方針の延長に置く。

これにより FeatureKind は最低でも `Settlement`（小集落）・`UrbanRuin`（市街地・大都市）・
`RuinEntrance`（遺跡）・`Resource`・`Hazard` に増える。市街地は地表の大型 POI なので、§4 の Feature 層に
spec を1つ足しつつ、§5 の Connection 層が扱うクロスチャンク大型構造として決定的に配置する。規模で
リスクと報酬が変わる点を、その `Content` と `Placement` が表す。小集落と市街地を分けるのが本改訂の要で、
「補給の安全な小集落」と「高リスク高報酬の大都市」を別カテゴリにする。

### レーンはリスク/リワードの明確な分岐にする

- 南北 Rows>1 は空間的な広がりに留めず、リスク/リワードの明確な分岐にする。あるレーンは報酬が多いが
  危険、別のレーンは安全だが乏しい、のようにレーンごとに難易度と報酬を差別化する。
- **含意**: これには Progression と Biome を Y に効かせる必要がある。doc 67 の控えめな「行ごとに地形が
  違うだけ」から一歩踏み込み、`ProgressionAt(ChunkCoord)` にレーン軸を持たせ、Biome 割当も Y に依存
  させる。前線は X 一軸のままでよい。レーン差は東進軸でなく南北軸の Progression と Biome が担う。

### アーキテクチャの確定

- doc 68 の層パイプライン + FeatureSpec を最終目的地として確定する。小集落をその第一インスタンスにし、
  市街地・遺跡入口・資源・ハザードを同型の FeatureSpec として増やす。種類の追加は分岐でなくインスタンスの
  追加に還元する、という本 doc の芯を最終像として据える。

### 各 doc への含意まとめ

- **doc 66**: 小集落の無状態・stash なしを最終形として維持。街を一様な補給点とせず、小集落＝補給と
  市街地・大都市＝高リスク高報酬に分ける。
- **doc 67**: レーン差別化のため Progression/Biome の Y 依存を要件へ加える。前線フレーミングを追い立てから
  勾配へ改める。帯・シフト・serde の機構自体は不変。
- **doc 68**: FeatureKind に `UrbanRuin`（市街地・大都市）を追加。`Progression` にレーン軸を持たせ Y 依存にする。
