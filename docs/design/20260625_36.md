# 小部屋マップの入口にランダムにドアを生成する

## 背景

小部屋タイプのマップビルダー（`NewSmallRoomPlanner`）で生成されるダンジョンには、部屋と廊下が存在するが、部屋の入口にドアがない。ローグライクとして、閉じた扉が部屋の入口にあることで探索の緊張感や戦術性が生まれる。

### 要件

- 部屋の入口にランダムな確率でドアを配置する
- 3タイル幅の廊下が部屋との接続部で1タイル幅に狭まるようにする
- 既存の扉システム（開閉、ロック、BlockPass/BlockView）をそのまま活用する

## 設計

### 方針: タイルパターン検出によるドア配置

digger_rsの方式を参考に、全タイルを走査して「床タイルの左右が壁かつ上下が床」または「上下が壁かつ左右が床」のパターンを検出し、ドアを配置する。部屋境界の走査やグループ化は不要で、タイルパターンのみで判定する。

廊下が部屋との接続部で1タイル幅に狭まるのは `LineCorridorPlanner` が担当する。部屋に隣接するサイドタイルを床に変換しないことで、自然に1タイル幅の入口が生まれる。

### PlannerChain内の位置

```
RectRoomPlanner        // 部屋を計画
FillAll("wall")        // 全体を壁で埋める
RoomDraw               // 部屋を描画
LineCorridorPlanner    // 廊下を作成（部屋境界で1タイル幅に狭まる）
DoorPlanner            // 入口にランダムにドアを配置  ← 新規
ConvertIsolatedWalls   // 床に隣接しない壁をvoidに変換
EnvironmentPlanner     // 環境情報を計算
```

### データ構造

```go
// DoorSpec はドア配置仕様を表す
type DoorSpec struct {
    consts.Coord[int]
    Orientation gc.DoorOrientation // 計画フェーズで確定済み
}
```

### スポーン

`mapspawner/spawner.go` の `spawnDoors` で `DoorSpec.Orientation` をそのまま使用する。向きは計画フェーズで確定済みのため、スポーン時の再計算は不要。

### 採用しなかった方法

| 方法 | 不採用の理由 |
|------|-------------|
| 部屋境界走査→グループ化→狭め方式 | 4辺走査、グループ化、壁戻しなどロジックが複雑。タイルパターン検出で十分 |
| LineCorridorPlanner内で入口座標を記録する | 廊下生成ロジックと入口検出ロジックが混在して責務が曖昧になる |
| 廊下を全体的に1タイル幅にする | 見た目の開放感が失われる。部屋境界のみ狭めれば十分 |

## コメント

- `DoorChance: 0.8` でドア配置確率を80%に設定している。ゲームバランスに応じて調整可能
- DoorPlannerのRNG消費により、固定シードを使うテスト（`TestHostileNPCPlanner_PlanMeta`）のシードを99999→100に変更した
- `DoorSpec` に `Orientation` を持たせることで、計画フェーズとスポーンフェーズの二重計算を排除している
