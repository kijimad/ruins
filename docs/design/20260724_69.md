# オーバーワールド生成の統合設計

旧 `20260723_66`(街生成)・`20260723_67`(縦チャンク任意数化)・`20260724_68`(生成アーキテクチャ)を1枚へ
統合する。本 doc が最新の結論で、反復の経緯 —— 批判レビュー、PR レビュー反映、Minecraft リージョン方式への
改良、世界観の全面刷新 —— は3枚分の git 履歴に残る。本文は結論だけを、決定理由と却下案を添えて示す。

## 背景と世界観

生成ロジックの豊かさはこのゲームの面白さのキモである。ゆえに拡張性と高い表現性に耐える横断設計が要る。
初稿は生成の seam を「街」という具体に置いており、種類を増やすたびに `IsXChunk` の分岐が増える天井があった。
本 doc は生成を層のパイプラインとデータ駆動可能な FeatureSpec レジストリに置き、街・市街地・遺跡・資源・
ハザードを同型のインスタンスにする。

世界観は現実寄り。2026年日本、未知災害後の世界。詳細は `docs/plan/lore.md`。コア体験は**ゆるやかな制約下の
オープン探索**である。

- 主軸はオープンな探索。明確な東ゴールへプレイヤーを追い立てる構造にはしない。
- 寒波前線は**西から東へ**進み、呑まれた土地は凍てつく。**寒いのは背後の西**である。プレイヤーは前線に
  背を押されるが、追い立てられはしない。前線は「いずれ確実に呑まれる」ゆるやかな制約として探索を方向づける。
  東へ進むほど厳しくなる難度は前線でなく**東進距離の軸**が担う。2軸を混同しない。前線ロジックの X 一軸は
  据え置く。
- 生成は `(seed, 座標)` の純関数を保つ。難易度もプレイヤー状態でなく座標由来にする。これが決定性・serde・
  再訪一致を守る背骨になる。動的生成は射程外。

参考は CDDA と Minecraft。CDDA から中身の重み・配置制約の語彙、Minecraft から決定的チャンク生成とリージョン
配置を採る。各々の強い部分だけ翻案し、CDDA のグローバル乱数依存は採らない。

## 最終目的地（射程）

- **バイオーム第一級**。region ごとに地形・敵・気候が変わる。境界は noise で連続。
- **地物5カテゴリ**。小集落、市街地・大都市、遺跡入口、資源、ハザード。すべて同型の FeatureSpec。
- **難易度勾配3軸**。うち生成が読むのは座標由来の2軸、東進(基礎難度・報酬)とレーン(南北 Y、リスク/リワード
  分岐)。前線(西からの寒冷)は時間依存なので生成には入れず、実行時オーバーレイが担う。
- **クロスチャンク構造**。街をつなぐ道、region をまたぐ大型構造。決定的に協調。
- **決定的・静的**。プレイヤー状態を入力にしない。
- **コード先行、データ化可能**。最初は Go 定義、器が固まったら TOML/raw へ。

## 場所の類型 —— いずれも特別扱いせず同型の地物にする

現実寄りの世界観に合わせ、街を一様な補給点とはしない。規模と危険で分ける。

- **小集落・一軒家**（`Settlement`）。生存者がいる交易・補給・改造・雇用・情報・休息の場。村・農家・峠の
  一軒家など。規模が小さく比較的安全。**無状態で stash なし**。開始拠点もこの類型。
- **市街地・大都市**（`UrbanRuin`）。地表の高リスク・高リターン地帯。**規模が大きいほどモンスターも戦利品も
  多い**。複数チャンクに及ぶ地表の攻略対象で、深さ y は持たない。補給の場ではない。自衛隊基地跡や研究施設も
  この危険帯に属する。
- **遺跡・ダンジョン**（`RuinEntrance`）。深さ y を持つ階層ステージ。入口から入り階を進む。**地下とは限らない**。
  廃ビルは上へ、地下室・地下墓・自然洞窟は下へ。y の向きは構造しだいで、y を足すだけの一類型。Phase 8
  (`20260720_65`)「遺跡は隣のステージにすぎない」統合方針の延長に置く。

小集落と市街地・大都市を**別カテゴリ**にするのが要。「補給の安全な小集落」と「高リスク高報酬の大都市」を
分ける。市街地は地表の大型 POI なので Feature 層に spec を1つ足し、Connection 層が扱う複数チャンクの大型
構造として決定的に配置する。規模でリスクと報酬が変わる点を、その `Content` と `Placement` が表す。

## 設計

### 1. 生成パイプライン（層）

`NewChunkGen(coord)` を層の列にする。各層は前層の出力に作用し、`ChunkSeed2D` と座標から決定的。単層に全部を
詰めず、各層を差し替え・合成・単体テスト可能な段にする。

```text
GenerateChunk(coord):
  prog    := ProgressionAt(seed, coord)  // 1. 難易度場（東進・レーンの2軸。前線は実行時）
  biome   := BiomeAt(seed, coord)        // 2. バイオーム（連続 noise、Y にも依存）
  terrain := GenTerrain(coord, biome)    // 3. 地形（biome でパラメータ化）
  PlaceFeatures(coord, biome, prog)      // 4. 地物（FeatureSpec レジストリ）
  DrawConnections(coord)                 // 5. 道・大型構造（近傍を算出して結ぶ）
  Decorate(coord)                        // 6. 継ぎ目・小物
```

### 2. Progression 場（生成が読むのは2軸）

座標から静的に難易度を導く純関数。動的入力なし。**前線軸は生成に入れない**。前線位置は `TotalTurns` から
導出される時間の関数であり、生成入力に混ぜると同じチャンクの再生成結果が時刻で変わり、再訪一致が壊れる。
前線の寒冷・危険は既存の temperature / frost 系が**実行時オーバーレイ**として重ねる。「生成は座標純関数、
時間依存の効果はその上に実行時で乗る」という分離を不変条件にする。

レーン軸は南北 Rows>1 を空間的な広がりでなく**リスク/リワードの明確な分岐**にするために持たせる。あるレーンは
報酬が多いが危険、別は安全だが乏しい。**レーンのリスク配分は seed 依存**にし、ランごとに変える。固定配分の
「北は常に危険」は周回で読みが固定化し単調になるため採らない。可読性は biome の見た目と、集落の情報サービス
(§6)が担う。

```go
// Progression は座標由来の難易度。プレイヤー状態にも時間にも依存しないので決定的。
type Progression struct {
	East int // 開始からの東進距離。基礎難度・報酬 tier
	Lane int // 南北レーン。リスク/リワードの分岐。配分は seed 依存
}

func ProgressionAt(runSeed uint64, c ChunkCoord) Progression
```

Biome の敵スケール、Feature の距離帯 Constraints、報酬 tier がこれを読む。前線は X 一軸のまま全行を貫き、
レーン差は南北軸の Progression と Biome が担う。

### 3. Biome 層（第一級・連続 noise・Y 依存）

```go
// Biome は地形・敵・気候を束ねる第一級概念。どの地物を許すかは FeatureSpec 側の
// Constraints.Biomes だけが持ち、Biome 側には持たない。判定の二重管理を避け、TOML 化も阻害しない。
type Biome struct {
	Name       string
	Planner    mapplanner.PlannerType // 地形生成のプランナ
	EnemyTable string                 // 敵の湧き表
	TempOffset int                    // 気候。既存 temperature 系へ接続し二重管理を避ける
}

// BiomeAt は連続 noise で biome を決める。region 離散でなく境界が滲む。Y にも依存させ
// レーンごとに地形を差別化する。
//
// noise は runSeed だけでシードし、絶対座標で引く連続場にする。ChunkSeed2D は隣接チャンクで
// 無相関になるよう撹拌した per-chunk ハッシュなので、バイオーム場には使えない。使うと
// チャンク境界で biome が割れる。ChunkSeed2D は per-chunk の地物ロール専用。
func BiomeAt(runSeed uint64, c ChunkCoord) *Biome
```

粒度はハイブリッドにする。地形タイルのパレットは per-tile の連続場から引き、「滲む境界」を実際に作る。
Feature の Constraints 判定と敵テーブルは per-chunk の代表値、すなわちチャンク中心のサンプルで足りる。
チャンク単位の1値だけにすると 50x50 ごとの角張った境界になり、滲む境界にならない。

### 4. Feature 層（FeatureSpec レジストリ）

地物を種類ごとの述語でなく、統一 spec のインスタンスにする。ここが拡張性の芯。種類の追加は分岐でなく
インスタンスの追加に還元する。

```go
type FeatureKind int

const (
	FeatureSettlement   FeatureKind = iota // 小集落・一軒家。補給・安全
	FeatureUrbanRuin                        // 市街地・大都市。地表の高リスク高報酬・複数チャンク
	FeatureRuinEntrance                     // 遺跡入口。深さ y の階層へ
	FeatureResource                         // 資源・採集
	FeatureHazard                           // 危険地帯・POI・イベント
)

// FeatureSpec は1種類の地物の「どこに・どの条件で・何を」を1つにまとめた宣言。
type FeatureSpec struct {
	Kind        FeatureKind
	Placement   Placement    // どこに撒くか
	Constraints Constraints  // どの biome・距離帯で許すか
	Content     ContentGen   // 何を置くか
}

// PlacementMode は配置方式。ゼロ値の暗黙でなく明示の enum で区別する。
type PlacementMode int

const (
	PlaceRegion  PlacementMode = iota // リージョンに高々1つ。小集落・遺跡・市街地
	PlaceScatter                      // 密度散布。資源など
)

// Placement は配置ルール。疎な構造は Minecraft のリージョン方式、密な散布は Scatter。
type Placement struct {
	Mode       PlacementMode
	Spacing    consts.Chunk // Region: リージョンの一辺
	Separation consts.Chunk // Region: 最小間隔
	Salt       uint64       // 種類ごとに相関を切る
	Density    int          // Scatter: 散布密度
}

// Constraints は配置可否。CDDA の locations/city_distance/occurrences の翻案。
type Constraints struct {
	Biomes      []string // 許可 biome
	EastBand    [2]int   // 東進距離帯 [min,max]。難易度勾配で出現を絞る
	Occurrences [2]int   // 出現回数域
}

// ContentGen は確定した地物の中身を配置する。biome と prog を受け表現を変える。
type ContentGen func(world w.World, c ChunkCoord, seed uint64, b *Biome, p Progression) error
```

- **Settlement**(小集落)は最初の実装インスタンス。§5 で詳述。
- **UrbanRuin**(市街地・大都市)は複数チャンクの大型 POI。規模が大きいほどモンスターも戦利品も多い点を
  `Content` と `Placement` が表す。Connection 層がクロスチャンク配置を担う。
  - **RuinEntrance を内包できる**。lore の市街地は駅・地下街・ビル群を含み、これらは多層＝遺跡の素材である。
    市街地の中心部に階層への入口を置き、「地表の危険帯を突破して奥の入口に潜る」二段のリスク構造を作る。
    合成は Content が他 Feature の入口 prop を置けることで表現し、レジストリ機構は変えない。
  - **チャンク片の導出不変条件**。市街地の各チャンク片は `(citySeed, 相対offset)` の純関数で、**隣接チャンクを
    生成せずに**自分の断片を描けること。帯ストリーミングでは都市の一部だけが先に生成されるため、この不変条件を
    破ると生成順序依存が再発する。
- 遺跡入口・資源・ハザードは別 spec。当面はコード、将来は TOML で足すだけ。

### 5. 小集落の生成（Settlement の実装）

`Settlement` FeatureSpec の中身。旧 doc 66 の設計を最終形として据える。

- **配置は Minecraft のリージョン方式**。世界を `Spacing` チャンク四方のリージョンに分け、リージョンごとに
  `(seed, regionX, regionY, salt)` で seed した値が `[0, Spacing-Separation)` のオフセットを引き、そのチャンクを
  候補にする。「1リージョンに高々1つ」「最小 `Separation`」を O(1) で決定的に保証する。窓最大方式の帯端問題も
  消える。`townScore = ChunkSeed2D` の別名で、閾値と窓判定は不要になった。
- **中身は規模で gate した重み付き施設テーブル**。CDDA の city_size と houses/shops の翻案。小さな集落は
  基本施設のみ、規模が大きいほど施設が多様になりレアも出る。中心からの距離で種別を切り替える。施設は現代
  日本の意匠(コンビニ・薬局・宿・骨董品店ほか)。武器・防具は骨董品店で扱う。
- **無状態・stash なし**（旧 doc 66 案B を確定）。街に永続状態を持たせず、フィールドにアイテムを残さない。
  serde すべき可変状態が無く、seed からの決定的再生成と完全に整合する。
- **開始チャンク特例**。新規ゲームの開始点は必ず小集落にし、店・雇用・合成の必須サービスを確定配置する。
- `ChunkSeed2D` を唯一のエントロピー源にし、グローバル乱数を使わない。
- **経済はスコープ外**。集落ごとの産物・不足プロファイル、価格差、在庫・枯渇の記憶は本設計では扱わない。
  本設計はマクロ地図の生成に集中する。交易の経済設計は別ドキュメントで行う。その際は「無状態の表層」と
  「在庫・枯渇の記憶」の緊張、design.md の逗留の有限性と低い床・高い天井、を正面から扱う必要がある。
- **雇用候補の実装注意**。無状態の集落は、雇用済みの候補を再訪時にそのまま再提示しうる。候補の identity を
  `(集落seed, スロット)` から決定的に導き、生成時に隊ロスターと照合して雇用済み候補をスキップすれば、
  追加の永続状態なしで二重化を防げる。

### 6. Connection 層（クロスチャンク）

`Placement` が純関数なので、あるチャンクは近傍リージョンの拠点位置を**生成せず算出**できる。道はその位置を
結び、市街地・大都市のような region 大型構造も決定的に配置する。Minecraft が構造物を生成せず座標算出するのと
同じ原理。

```go
// NearbyFeatures は生成を伴わず、近傍にある指定種の Feature 位置を決定的に算出する。
func NearbyFeatures(runSeed uint64, c ChunkCoord, kind FeatureKind, radius consts.Chunk) []ChunkCoord
```

`NearbyFeatures` は道の結線だけでなく、**集落サービス「情報」の実装基盤**を兼ねる。lore の小集落は交易・雇用と
並んで情報を商う。配置が純関数なので、集落で噂を買う＝`NearbyFeatures` を引いて未踏の地物をミニマップへ印す、
が**追加の永続状態ゼロ**で実装できる。seed 依存にしたレーンのリスク配分(§2)の可読化も、この情報サービスと
biome の見た目が担う。

### 7. 帯インフラ —— 縦チャンク数を任意数にする

生成が走る座標基盤。旧 doc 67。現状の帯は縦1行だが、南北レーンのため行数 `Rows` を任意数にする。東西は無限
ストリーミング、南北は `Rows` に有界で非ストリーミング。`Rows=1` は現状と挙動同値。

```go
type Band struct {
	chunkW    consts.Tile
	chunkH    consts.Tile  // 縦スロット配置に要る
	k         consts.Chunk // X方向のチャンク数
	rows      consts.Chunk // Y方向のチャンク数。1 で現状
	eastIndex consts.Chunk
}
func (b *Band) Rows() consts.Chunk  { return b.rows }
func (b *Band) Height() consts.Tile { return b.rows.Tiles(b.chunkH) } // = rows*chunkH

// ChunkGen は2次元。ChunkCoord は X 絶対(東西無限)、Y 有界[0,Rows)
type ChunkCoord struct{ X, Y consts.Chunk }
type ChunkGen func(c ChunkCoord, offsetX, offsetY consts.Tile) error

// ChunkSeed2D は runSeed と (cx,cy) から決定的シードを導く。cx,cy を独立に混ぜる。後方互換は保証しない
func ChunkSeed2D(runSeed uint64, cx, cy consts.Chunk) uint64
```

- `generateBandChunks` は `rows × k` の2重ループ。`Level.TileHeight = rows*chunkH`。
- **シフトは X のみ・列(全行)単位**。`RemoveEntitiesInXRange` で端列の全 Y を破棄、`TranslateAllEntities(±chunkW,0)`、
  新端列は `rows` 行を生成。南北移動は破棄・再生成を伴わない。
- **`AbsTileY` は追加しない**。Y は非ストリーミングの有界ローカル座標で足りる。
- **縦境界の継ぎ目** `RecalcSeamAutotileY(world, boundaryY, totalHeight)`。上下端は自己スキップ。
- **serde に `Rows`**。`SeamlessBand.Rows` を追加。`startNewBand` で `sb.K` の隣に `sb.Rows = band.Rows()` を書く。
  `restoreFromSave` で読み `Rows=0→1` 正規化。旧セーブ互換。
- **前線は X 一軸のまま**。全行を貫く。行ごとに地形が違いレーン差が生まれる。

### 8. 決定性とコード→データ

- 全層は `(ChunkSeed2D, 座標, Progression)` の純関数。プレイヤー状態を入力にしない。動的生成は射程外。
- `FeatureSpec`・`Biome` は Go の登録表で始める。フィールドは TOML スキーマへ一対一で写せる形にし、レジストリ
  interface を安定させる。器が固まったら `raw`/TOML へ移す。

## 採用しなかった方法

- **種類ごとの述語** `IsTownChunk`/`IsRuinChunk`…。分岐が増え合成しない。`FeatureSpec` の別インスタンスにする。
- **CDDA のグローバル乱数方式**。生成順序に依存し、帯シフト・再訪・reload で世界が変わる。`ChunkSeed2D` の
  決定的配置に徹する。Minecraft の実績ある王道。
- **窓最大の隣接抑制**。`O(MinGap²)` で帯端の片側窓問題を抱える。Minecraft のリージョン方式に置き換え O(1)。
- **region 離散の biome**。境界が四角く不自然。noise で連続に滲ませる。
- **バイオーム場を ChunkSeed2D から引く**。per-chunk 無相関ハッシュでは境界が割れる。連続 noise を runSeed で
  シードし絶対座標で引く。
- **前線軸を生成入力に入れる**。前線位置は時間の関数で、混ぜると同じチャンクの再生成が時刻で変わり再訪一致が
  壊れる。前線は実行時オーバーレイに限る。
- **`Biome.AllowFeature` による許可判定**。`Constraints.Biomes` と二重管理になり、func フィールドはデータ駆動化を
  阻害する。spec 側に一本化。
- **レーンリスクの固定配分**。周回で読みが固定化し単調。seed 依存にし、可読性は biome の見た目と情報サービスで
  担保する。
- **全域2次元ストリーミング(Y も無限)**。過剰。固定行数 `Rows` の帯で足りる。南北シフトも不要。
- **`AbsTileY` 追加**。Y は有界なので不要な抽象。X の絶対軸だけ持つ。
- **街に stash・永続状態**。帯シフトで破棄・再生成されるため保持できない。無状態(案B)に確定。
- **街を一様な補給点**。規模と危険で分け、小集落(補給・安全)と市街地・大都市(高リスク高報酬)を別カテゴリに。
- **前線が追い立てる構造**。オープン探索を主軸にし、前線は難易度勾配へフレーミングを寄せる。ロジックは不変。
- **最初からデータ駆動**。器が固まる前に TOML スキーマを凍結すると手戻り。コード先行→移行。
- **動的生成(プレイヤー反応)**。決定性・serde・再訪一致が崩れる。難易度は座標由来の Progression に限る。

## 実行計画

各段は単独で `make check` 緑を保つ。`Rows=1`・`cy=0` を現状同値に保つのが全段の退避路で、大改造を小さな緑の
積み重ねに分解する。

1. **共有前提**。`ChunkCoord`・`ChunkSeed2D` を入れる。転置・隣接で散ること、後方互換を保証しないことをテスト。
2. **帯インフラ**。`Band` に `chunkH`・`rows`。`ChunkGen` を2次元化。`generateBandChunks` 2重ループ。シフトを列
   単位に。縦継ぎ目・`serde Rows`・マスタ行数。`Rows=1` 同値、`Rows>1` の通し。
3. **パイプライン骨格**。層の順序と純関数境界を固定。`ProgressionAt`(東進・レーン2軸、配分 seed 依存)。
   `BiomeAt`(連続 noise・ハイブリッド粒度)。前線が生成入力に混ざらないことをテストで固定。
4. **FeatureSpec レジストリ**。Settlement を最初の1インスタンスに。仮 `spawnTown` をその `Content` へ再構成、
   収納撤去、無状態、開始チャンク特例。`ChunkTown` 判定をリージョン方式で。
5. **カテゴリを増やす**。市街地・大都市(`UrbanRuin`)、遺跡入口、資源、ハザードを FeatureSpec として順次。
6. **Connection 層**。道・大型構造の `NearbyFeatures`。市街地のクロスチャンク配置。
7. **レーン差別化**。`Progression`/`Biome` の Y 依存でレーンをリスク/リワードに分ける。
8. **データ化**。器が固まったら `FeatureSpec`・`Biome` を `raw`/TOML へ移す。
