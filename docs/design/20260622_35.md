# アイテム個数変化時のスタック整理

## 背景

現在、同種Stackableアイテムの統合（MergeInventoryItem）はPickup時にのみ呼ばれている。職業選択経由でアイテムが増えたとき、収納から取り出したとき、クラフトで生成したときなどには統合されず、同名アイテムがバックパック内に分散したまま残る。

### 再現手順

1. バックパックに「回復薬 x3」を所持
2. 収納から「回復薬 x2」を取り出す
3. バックパックに「回復薬 x3」と「回復薬 x2」が別エンティティとして存在する（期待: 「回復薬 x5」に統合）

### 現行の統合経路

| 経路 | 統合 | 関数 |
|------|------|------|
| Pickup | あり | `activity/pickup.go` → `MergeInventoryItem()` |
| Storage取り出し | **なし** | `states/storage_menu.go` → `MoveToBackpack()` のみ |
| クラフト生成 | **なし** | `worldhelper/craft.go` → `SpawnBackpackItem()` のみ |
| ショップ購入 | あり | `worldhelper/shop.go` → `ChangeStackableCount()` で既存に加算 |
| 職業選択 | 問題なし | `SpawnBackpackItem(name, count)` で1エンティティにcount設定 |

### 方式

**追加時の即時統合**

`inventory::add_item(should_stack=true)` で、アイテム追加時に既存スタックを走査し `stacks_with()` でマッチするものがあれば `merge_charges()` で統合する。追加の入口で統合するため、呼び出し元は統合を意識しない。

## 設計

### 方針: バックパック追加の入口で統合する

バックパックへの追加は全て `MoveToBackpack()` を経由するため、ここで統合処理を呼べば全経路をカバーできる。

### 変更箇所

`worldhelper/location.go` の `MoveToBackpack()` に統合処理を追加する。

```go
func MoveToBackpack(world w.World, entity ecs.Entity, owner ecs.Entity) {
    setLocation(world, entity, &gc.LocationInBackpack{Owner: owner})
    owner.AddComponent(world.Components.StatsChanged, &gc.StatsChanged{})
    owner.AddComponent(world.Components.WeightDirty, &gc.WeightDirty{})

    // Stackableアイテムならバックパック内の同名アイテムと統合する
    if entity.HasComponent(world.Components.Stackable) {
        name := world.Components.Name.Get(entity).(*gc.Name)
        MergeInventoryItem(world, name.Name)
    }
}
```

`activity/pickup.go` の個別統合呼び出しは不要になるため削除する。

```go
// 削除対象（pickup.go の collect() 内）
if entity.HasComponent(world.Components.Stackable) {
    if err := worldhelper.MergeInventoryItem(world, name); err != nil {
        return fmt.Errorf("インベントリ統合エラー: %w", err)
    }
}
```

`worldhelper/craft.go` の `SpawnBackpackItem()` は内部で `MoveToBackpack()` を呼ぶため、自動的に統合される。追加の変更は不要。

### MoveToBackpackの返り値変更

`MergeInventoryItem` はエラーを返すため、`MoveToBackpack` も `error` を返すように変更する。

```go
func MoveToBackpack(world w.World, entity ecs.Entity, owner ecs.Entity) error {
    setLocation(world, entity, &gc.LocationInBackpack{Owner: owner})
    owner.AddComponent(world.Components.StatsChanged, &gc.StatsChanged{})
    owner.AddComponent(world.Components.WeightDirty, &gc.WeightDirty{})

    if entity.HasComponent(world.Components.Stackable) {
        name := world.Components.Name.Get(entity).(*gc.Name)
        if err := MergeInventoryItem(world, name.Name); err != nil {
            return fmt.Errorf("バックパック内のアイテム統合に失敗: %w", err)
        }
    }
    return nil
}
```

### 影響範囲

`MoveToBackpack` の呼び出し元すべてでエラーハンドリングを追加する。

| 呼び出し元 | ファイル |
|-----------|---------|
| Pickup処理 | `activity/pickup.go` |
| Storage取り出し | `states/storage_menu.go` |
| 装備解除 | `worldhelper/equip.go` |
| SpawnBackpackItem | `worldhelper/spawn.go` |
| テスト | 各種 `_test.go` |

### 採用しなかった方法

| 方法 | 不採用の理由 |
|------|-------------|
| 各呼び出し元で個別に `MergeInventoryItem()` を呼ぶ | Pickup と同じパターンの繰り返しになる。呼び忘れが起きやすく、今回のバグと同じ問題を再発させる |
| Stackableを廃止して全アイテムを個別エンティティにする | UI表示やインベントリ操作が煩雑になる
| SpawnBackpackItem内で統合する | SpawnBackpackItemは内部でMoveToBackpackを呼ぶので、MoveToBackpackで統合すれば自動的にカバーされる。より低レイヤーで統合する方が漏れが少ない |

## コメント

- `MoveToBackpack` の返り値変更は破壊的変更だが、コンパイラがすべての呼び出し元を検出するため安全に移行できる
- `MergeInventoryItem` は `LocationInBackpack` を持つアイテムのみを対象とするため、Storage内やフィールド上のアイテムには影響しない
