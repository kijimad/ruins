# マクロ移動（キャラバンのルート網踏破）の実装設計

## 背景

54.md（§11・§12）でゲーム構造を **B1＝多地点キャラバン交易ローグライク／骨格グラフ→シームレス埋め込み** に確定した。その中核である **マクロ移動 ── キャラバンが凍原のノード網（ルートグラフ）を片道で踏破し、どのノードを経由するか選ぶ層 ──** を、現行コードにどう載せるかを設計する。

参照モック（既存・確定イメージ）:

- `docs/plan/mock/b1_caravan.html` ── マクロ層マップ（ルート網）＋キャラバン状態（食料・燃料・体温・積載）＋各ステップ選択（交易→ルート→潜行→引き際）。上部に**寒波前線が背後まで接近**・**目的地まで N ターン**。
- `docs/plan/mock/quest_flow.html` ── 遠征選択（背骨）→ ルート網（分岐→隊商宿で合流→再分岐）→ ノード型のリスク/メリット凡例。

現行構造（本設計の土台。Explore 調査より）:

- シーンは `internal/engine/states` の汎用ステートマシン（`State[T]`／`Transition[T]`／`TransPush/Pop/Switch/Replace`）上に、`internal/states` の各ステートが載る。**Town すら `DungeonState`（`BuilderType="Town"`）の一種**で、マップは常に 50×50 タクティカル（`consts.MapTileWidth/Height=50`）。
- run レベルの状態は**シングルトンエンティティ上のコンポーネント**（`Dungeon`＝現在階・深度・探索済み・定義名、`GameProgress`＝クリア済み・イベント、`TurnState`）として持つ idiom。
- ECS は Ark。コンポーネントは登録表＋`make generate`（`add-component` skill）。
- 既存 stats 系に `hunger`・`tile_temperature`・`health_status` がある（食料・体温の土台）。燃料・積載・寒波前線・ルートグラフはまだ無い。

**現状はマクロ層が存在しない**（ダンジョン選択→単発ダンジョン潜行のみ）。ここを新設する。

## 要件

- 54.md §11–§13 の確定方針を壊さない: 骨格グラフ／分岐→合流／ノード型／辺＝面数＋逆向き利得／**一方向（寒波前線が背後を封じる）**／段階実装（入口＝グラフ明示表示、発展＝シームレス埋め込み）。
- **選択感が最重要**: 「次どこへ」がリスク/メリットを見て腑に落ちる判断になること。最短一択にしない。
- **段階移行で捨て駒を出さない**: Phase 1 で作る骨格データ構造を、Phase 2 のシームレス化でそのまま使えること。
- **テスタビリティ**: グラフの生成・踏破・消費計算は ECS/Ebiten 非依存の純ロジックとして単体テストできること（CLAUDE.md「シンプルさとテストしやすさ最優先」）。
- 既存 `DungeonState`（タクティカル・レグ）と滑らかに往復できること。
- **交易は自動精算せず能動売却**（54.md §12/§13 確定）: 道中マーケットの売却＝通貨即 bank／Goal＝背骨ターンイン（精算場所でない）／前哨＝最終売却点／価格は噂・斥候で開示。交易は本業（潜行サルベージ＋納品）に対するオプションの金策（低い床・高い天井）。

## 設計

### 用語

| 語 | 意味 |
|---|---|
| ルートグラフ | 母港→目的地の**層状 DAG**。ノード＋有向辺。分岐がしばらく続き合流点で再合流し再分岐する |
| ノード | 交易/潜行/野営/合流/目的地などの地点。型ごとにリスク/メリット |
| 辺（レグ） | ノード間の移動。**面数 N** のコスト（供給消費・寒波接近）と型（近道/迂回/危険路）を持つ |
| キャラバン | プレイヤーの隊。ルートグラフ上の**現在ノード**に居る。供給（食料/燃料）と積載を持つ |
| 寒波前線 | 背後から前進するスカラー位置。追いつかれたらラン失敗＝一方向性の源 |

### 1. データモデル ── 純パッケージ `internal/route`

グラフの**モデル・生成・踏破コスト計算**を ECS/Ebiten 非依存の純パッケージに置く。ここが本機能のテスト可能な心臓部。

```go
// internal/route  （新規・純ロジック）
type NodeID int

type NodeType int
const ( NodeHome NodeType = iota; NodeMarket; NodeShop; NodeRuin; NodeCamp; NodeOutpost; NodeJunction; NodeGoal )

type Node struct {
    ID    NodeID
    Type  NodeType
    Layer int        // 列（母港=0、目的地=最終層）。生成と描画の骨格
    Label string
    // リスク/メリットは「見えている情報」。噂・斥候で開示される値は別途 Fog で管理
    Merit string
    Risk  string
}

type EdgeType int
const ( EdgeNormal EdgeType = iota; EdgeShortcut; EdgeDetour; EdgeDanger ) // 凍える近道/暖かい迂回/危険路

type Edge struct {
    From, To NodeID
    Type     EdgeType
    Faces    int   // 面数。距離＝供給消費・寒波接近の量
    // 逆向き利得（面数だけを差にしない）は EdgeType から導く or 明示フィールド
}

type Graph struct {
    Nodes  []Node
    Edges  []Edge
    Home   NodeID
    Goal   NodeID
}

// 現在ノードから進める辺
func (g *Graph) Outgoing(from NodeID) []Edge
```

**辺の踏破コスト**（面数×重量で運搬役が積荷を食う＝§14 の物量的頂点）も純関数に:

```go
// 1 面あたりの供給消費。積載（重量）が重いほど1面の消費が増える
func LegCost(e Edge, load Weight) Supply
```

### 2. run 状態の持ち方 ── シングルトンコンポーネント `CaravanRun`

現行 idiom（`Dungeon`/`GameProgress` をシングルトンエンティティに載せる）に合わせ、**run 単位のマクロ状態を1つのシングルトンコンポーネント `CaravanRun`** に持たせる。グラフ本体は §1 の純型を参照するだけ。

```go
// internal/components/caravan_run.go （新規・add-component 手順で登録）
type CaravanRun struct {
    Graph       *route.Graph   // 今回生成されたルート網（serde 除外 or ID で再構築）
    Current     route.NodeID   // キャラバンの現在ノード
    Visited     []route.NodeID // 通過済み（戻れない）
    FrontLayer  float64        // 寒波前線の到達層（現在ノードの Layer を超えたら失敗）
    ExpeditionID string        // 選んだ遠征（背骨・§13）
}
```

供給（食料・燃料・体温・積載）は **既存 stats を優先活用**:

- 食料 → 既存 `hunger` を隊レベルに集約（or 隊アイテムとしての糧食在庫）。
- 体温 → 既存 `tile_temperature` の露出をレグ単位で適用。
- 燃料（炉・凍晶）・積載合計 → **新規が必要**。`CaravanSupply`（燃料残・積載上限/現在）を新設 or `CaravanRun` に内包。

> 判断ポイント（要確認）: グラフを**シングルトンコンポーネントに内包**するか、**Resources に持たせる**（`resources.Resources` に `CaravanRun *route.Run`）か。前者は既存 idiom と統一・save 系に乗る／後者は ECS 非依存で最もテストしやすい。**推奨は前者**（idiom 統一・セーブ統合）。ただしグラフ型自体は純パッケージなので、どちらでもテスト容易性は §1 で担保される。

### 3. シーン設計 ── 新ステート `MacroMapState`（3案比較）

マクロ層の「マップを見て次ノードを選ぶ」画面をどう作るか。

| 案 | 内容 | 選択の明示性 | 実装コスト | シームレス移行 |
|---|---|---|---|---|
| **A. 専用グラフステート**（採用/Phase1） | `MacroMapState` を新設。SVG モック相当のノード網を描画し、進める辺を選ぶ | ◎ モック通り | ○ 新ステート1つ＋描画 | ○ 骨格データを流用 |
| B. `DungeonState` に統合 | マクロを「オーバーワールドのタクティカルマップ」として `DungeonState` の一変種に | △ タイル上に溶けやすい | △ 既存に相乗り | ◎ そのまま連続化 |
| C. ハイブリッド | Phase1=A、Phase2 で A の描画を連続マップに差し替え | ◎→◎ | Aと同等 | ◎ |

**採用＝C（入口は A、骨格を共有して B 方向へ）**。54.md §11「骨格グラフ→シームレス埋め込み・段階移行（捨て駒なし）」に一致。

`MacroMapState`（`internal/states/macro_map.go`）の責務:

- `Draw`: `CaravanRun.Graph` をノード網として描画（モック `b1_caravan.html` の SVG 相当を Ebiten で）。現在ノード強調・寒波前線オーバーレイ・各辺の面数/型・各ノードのリス/メリット（未開示は霧）。
- `Update`: 進める辺（`Graph.Outgoing(Current)`）から1つ選ぶ。選択 UI は既存 `widgets/tabmenu` を流用可。
- 選択→ §4 の踏破解決→ 到達ノードの型に応じ §5 でサブステートへ遷移。

遷移スタック（既存 `TransPush/Pop`）:

```
DemoStart → (遠征選択) → MacroMapState  ←──────────────┐
                              │ 辺を選ぶ                │ 戻る(TransPop)
                              ▼                         │
                 ┌── Ruin  → DungeonState(潜行) ────────┤
                 ├── Market→ ShopMenu/TavernMenu ───────┤
                 ├── Camp  → 野営(休息/加工) ────────────┤
                 └── Goal  → 背骨ターンイン・ラン終了(遠征達成)
```

MacroMapState は**スタックに残す（Push でサブステートを重ねて Pop で戻る）**。これで「潜行から戻ったら同じマップの続き」が自然に成立。

### 4. 辺の踏破解決 ── 面数・供給・寒波前線

辺を選んだら:

1. `LegCost` で **面数 × 現在積載**ぶんの供給（食料/燃料）を消費、体温を辺の型に応じ変動（近道＝寒い）。
2. **寒波前線 `FrontLayer` を面数ぶん前進**。到達層がキャラバンの層に追いつけばラン失敗（呑まれる）。
3. `Current` を更新し `Visited` に追加（**戻れない＝一方向**。off-route→再合流の短い寄り道のみ Outgoing で許可）。

**「面」を Phase1 で何にするか**（2択）:

- **(a) 抽象 tick ＋任意の単発遭遇**（推奨/Phase1）: 各面は供給消費・寒波接近・低確率の遭遇イベント（襲撃＝指示で捌く）を**ログとダイスで抽象処理**。フルマップは開かない。まずループを立ち上げる。
- (b) 面ごとに軽量トラベル面（既存50x50を travel 用に流用）: 手触りは濃いが、長ルートが全部フルマップだと作業感（§11「面の粒度」の警告）。**遺跡ノードのみフルタクティカル**、travel 面は (a) の抽象で始め、後で肉付け。

→ **Phase1＝(a)、遺跡ノード到達時のみ既存 `DungeonState` を開く。**

### 5. ノード到達ディスパッチ

到達ノードの `NodeType` でサブステートを Push:

| NodeType | 遷移先 | 既存資産 |
|---|---|---|
| Market/Shop | **集落マップ（`NewMarketState`）に入場**し、商人に話しかけて `ShopMenuState`（需給プロファイルで価格差）。ShopMenu は即出さない | **あり（集落マップ＝town_plaza の門を帰還ゲートに差し替え。売却タブ・`SellItem` 実装済）。価格に土地係数を追加** |
| Ruin | `NewDungeonState(...)`（潜行・頂点で撤収） | あり（既存タクティカル） |
| Camp | 野営（休息・暖・加工） | 一部（休息/加工の軽ステート新設） |
| Outpost（前哨） | 目標地点手前の**最終売却点**（Market 相当の sell）＋最終補給 | Market 流用 |
| Junction | 交易＋全ルート合流の結節（Market 相当＋再分岐） | Market 流用 |
| Goal | **背骨をターンインして遠征達成**（スコア＝富＋希少財＋生存者）→ ラン終了。一般交易品を精算する場所ではない（売りは道中で能動的に済ませる） | 達成/納品処理を新設 |

### 6. グラフ生成 ── 層状 DAG（分岐→合流を保証）

`route.Generate(expedition, rng) *Graph`（純関数）:

- **層（列）で構成**: Layer 0＝母港、最終層＝目的地。中間層に数ノードずつ。
- **辺は隣接層のみ接続**（前進のみ＝一方向を構造で担保）。
- **合流点（Junction）を必ず1つ以上挿入**し、全ルートが通る括れを作る（quest_flow の「3分岐→隊商宿で合流→再分岐」）。
- **ノード型と辺型を遠征（背骨）と地域で重み付け**（交易都市遠征＝Market 多め、深層ヴォールト＝Ruin 深め）。
- **逆向き利得を強制**: 近道（面少）は補給なし・危険濃、迂回（面多）は資源・安全（§11 の 3択表）。
- 生成物は §1 の `Graph`。テストは「母港から目的地へ必ず到達可能」「Junction を全経路が通る」「最短一択にならない（各層で有効選択肢≥2）」を assert。

### 7. 寒波前線と頂点

- `FrontLayer` を毎レグ前進。UI は数値でなく身体語（「寒波前線 5T 後方まで接近」＝モック上部）。
- 追いつかれ＝ラン失敗。これが「もう一つ遺跡を襲うか、荷を確保し急ぐか」（モック STEP④）の**引き際＝頂点**を毎回突きつける。

### 8. 段階実装マッピング（54.md §11 の段階移行）

- **Phase 1（本設計の主眼）**: `internal/route` 純パッケージ＋`CaravanRun` シングルトン＋`MacroMapState`（グラフ明示表示）＋抽象トラベル＋遺跡のみ既存タクティカル。**ループが一本通る**ことを最優先。
- **Phase 2（後段）**: `MacroMapState` の描画を連続マップへ差し替え、辺（面数）を歩けるタイル距離に肉付け（§1 の骨格データはそのまま）。travel 面を (b) の軽量タクティカルへ昇格。

## 採用しなかった方法

- **ノード/辺を ECS エンティティ化する** ── グラフは「全体を一括で読む凝集した1データ構造」で、ノード単位の per-entity システムを必要としない。エンティティ化は Ark strict semantics のクエリ・liveness 管理コストを増やすだけで得がない。**純パッケージ＋シングルトン参照**にした（テスト容易性も純パッケージで最大化）。
- **いきなり純シームレス（骨格グラフ無し）** ── ルート選択が暗黙化して溶け、生成も難しい（54.md §11 で却下済み）。骨格グラフを下敷きにする。
- **木構造（分岐したまま合流しない）** ── 「経路を選ぶ」でなく「次を引く」だけになる（54.md §11）。合流点を必ず挿入する層状 DAG にした。
- **面ごとに必ずフルタクティカルマップを開く** ── 長ルートが作業になる（§11「面の粒度」）。Phase1 は travel を抽象 tick、遺跡のみフルマップ。
- **マクロを既存 `DungeonState` に最初から統合（案 B 即採用）** ── Phase1 から連続マップに載せると選択の輪郭が溶け、実装も重い。骨格を共有する段階移行（案 C）にし、B は Phase2 の到達点として温存。
- **マクロ状態を全部 `Resources` の生 struct に置く** ── テストは最も楽だが、既存の run 状態 idiom（シングルトンコンポーネント）と二重管理になり save 系とも噛み合わない。グラフ型のみ純パッケージに切り出し、run 保持はコンポーネントに統一。

## コメント

- 数値（1面あたり供給消費式、積載と消費の係数、寒波前進速度、各遠征のノード型重み）は未定。`route.LegCost`/`Generate` に集約し、`internal/formula` や data.toml と連動させてバランス調整する。
- モック `b1_caravan.html` の4ステップ（交易→ルート→潜行→引き際）は、そのまま §3 の遷移ループ＝ゲーム本体。UI もモックの「身体語を主・数値を副」を踏襲する。
- 既存「ダンジョン選択→単発潜行」フローとの接続（置換か、当面は別導線で併存か）は Phase1 着手時の最初の判断。まず MacroMap を独立導線で立ち上げ、動いてから既存フローを畳むのが安全。

## 実装計画

実コードの統合点（`engine/states` ステートマシン・`factories.go`・`add-component`・`save/serde.go`・既存 `ShopMenuState`/`DungeonState`）に接地した段階計画。**最優先は Phase 1→3 を一直線に通す「歩けるスケルトン」**（§8「ループが一本通る」）。

### Phase 0 — 着手前に確定する判断（コードをブロックする4件）

型が決まらないと `route`/`CaravanRun` が書けない。「未解決の質問（TODO）」の芯4件を下記の初期値で unblock する（厳密なバランス値は後段）:

| 判断 | unblock 用の初期値 | 影響する型 |
|---|---|---|
| 食料モデル | 重量を持つ隊在庫（`route.Weight`/`Supply`）。`hunger` は個体演出で別立て | `LegCost`/`CaravanSupply` |
| 寒波前線の単位 | 累積面数。`CaravanProgress`/`FrontProgress int`、差＝リード | `CaravanRun` |
| Graph serde | `Seed int64` を持ち、`Graph` は `json:"-"`＋再構築 | `CaravanRun` |
| off-route 寄り道 | Phase 1 は未実装（層状 DAG・前進のみ） | `route.Generate` |

### Phase 1 — `internal/route` 純パッケージ（テスト先行・心臓部）

ECS/Ebiten 非依存。TDD で最初に固める。

- 型: `NodeID/NodeType/Node`・`EdgeType/Edge`・`Graph`・`Weight/Supply`・`LegResult{供給Δ, 体温Δ, 遭遇, 前線Δ}`
- `Generate(expedition, seed) *Graph`（層状 DAG・Junction 必須・分岐区間の各層で有効選択肢≥2）
- `Outgoing(from) []Edge`
- `ResolveLeg(state, edge) LegResult`（TODO章 G：`LegCost` 単体でなく1レグ結果を返す純関数）
- テスト: 到達可能／Junction 全通過／分岐層≥2／最短一択回避／頂点発生（積載増→供給消費増）

### Phase 2 — `CaravanRun` シングルトンコンポーネント

`add-component` skill（`genspec/registry.go` にエントリ→`make generate`）。

- struct: `Seed int64`・`Graph *route.Graph json:"-"`・`Current`・`Visited`・`CaravanProgress/FrontProgress int`・`ExpeditionID`（型付き定数）・供給（`CaravanSupply` 内包 or 別）
- `NewCaravanRun(seed, expedition)` が `route.Generate` を呼ぶ
- `reestablishSingleton()`（`serde.go:62-98`）に `Seed`→`Generate` の Graph 再構築を追加（`Dungeon.ExploredTiles` と同要領）
- 取得は `query.GetSingleton[CaravanRun]`／`GetCaravanRun(world)` パターン

### Phase 3 — `MacroMapState`（独立導線で立ち上げ）★歩けるスケルトン

State I/F（`OnStart/OnStop/OnPause/OnResume/Update/Draw`）実装。

- `Draw`: まず簡易ノード表示（SVG 忠実描画は後回し）。現在ノード・前線リード・辺の面数/型
- `Update`: `Outgoing(Current)` → 辺選択（`tabmenu` 流用）→ `ResolveLeg` 適用 → ディスパッチ
- `factories.go`: `NewMacroMapState(opts...) es.StateFactory[w.World]`（`NewDungeonState` パターン）
- 仮入口: `MainMenu` デバッグ項目 or `DemoStart` 分岐で既存フローを壊さず起動
- **マイルストーン**: 辺選択→供給/前線が動く→Goal 到達が抽象処理で一周する

### Phase 4 — ノード到達ディスパッチ（§5）

| Node | 遷移 | 備考 |
|---|---|---|
| Ruin | `NewDungeonState(1, ...)` を Push | 脱出先を MacroMap に（★） |
| Market/Junction/Outpost | `NewMarketState(WithEscapeTarget)` を Push（集落マップ） | 商人 interact で ShopMenu／帰還ゲートで MacroMap へ |
| Camp | 休息の軽ステート新設 | |
| Goal | 背骨ターンイン・達成の軽ステート新設 | AutoSell を通さない |

**★主リスク**: `DungeonState.WarpEscape` が `AutoSellState→Town` に固定（`dungeon.go:531-534`）。マクロループでは荷を持ったまま `MacroMapState` へ戻す必要。→ `DungeonStateOption` に `WithEscapeTarget`（AutoSell を通さない＝自動精算を切る）を追加。ここが唯一の非自明な既存改修。

### Phase 5 — 寒波前線・遠征選択入口

- `FrontProgress` 前進・呑まれ判定（`CaravanProgress` に追いつけば失敗）。純計算は `route`、適用は system/state
- 遠征選択（§13）ステート → `MacroMapState`（seed 生成）。仮入口を正式入口へ

### Phase 6 — save 統合の確認

- ラン途中で save/load → `Seed` から `Graph` 再構築・`Current/Visited/供給` 復元をテスト

### Phase 7（後段）— §8 Phase 2

`Draw` を連続マップへ、面をタイル距離に肉付け。骨格（`route` 型）は流用。今は着手しない。

### クリティカルパス

```
Phase0(判断) → Phase1(route+test) → Phase2(CaravanRun) → Phase3(歩けるskeleton)★
                                                            ↓
                                    Phase4(ディスパッチ・★脱出先差し替え)
                                                            ↓
                                Phase5(前線/遠征入口) → Phase6(save) → [Phase7]
```

最優先は Phase1→3 の一直線。Phase4 以降は既存資産（`ShopMenu` 売却・`DungeonState`）に乗り、脱出先差し替え1点が解ければ速い。

### 芯に関わる（モデル判断）

- [x] **食料モデル**: Phase 1 は `hunger`＝個体の空腹演出、隊の糧食を `CaravanSupply.Food`（在庫int）＋`Cargo`（積載重量）で別立て。`ResolveLeg` の `load`＝`Cargo` が1面消費を押し上げ頂点を作る。「糧食在庫＝重量を持つ隊アイテム（エンティティ化）」への昇格は後段（現状は int 在庫で簡素化）。
- [x] **寒波前線の単位**: `CaravanProgress`/`FrontProgress`（累積面数 int）で位置・前線を同一単位で持ち、`FrontLead()=CaravanProgress-FrontProgress` の0以下で呑まれ＝ラン失敗を定義。
- [x] **off-route→再合流の寄り道**: **Phase 1 は寄り道未実装**（層状 DAG・隣接層前進のみ）と確定。支線ノードは Phase 2 以降で別表現する。
- [x] **Graph の serde 二層分離**: `CaravanRun` に `Seed uint64`＋`Expedition` を持ち、Graph は `json:"-"` で除外→`reestablishSingleton` で `route.Generate` 再構築（`ExploredTiles` の前例に倣う）。動的 state（`Current`/`Visited`/`Supply`/progress）は直列化。

### 詰め（実装構造）

- [x] **供給系（燃料・積載）の保持先**: `CaravanRun` に `CaravanSupply`（Food・Fuel・Cargo）を**内包**と決定（§2）。archetype クエリで排他する必要がなく、マクロ run 状態を1シングルトンに凝集する意図に沿う。
- [x] **グラフの持ち方**: シングルトンコンポーネント**内包＋seed**（§2）。既存 serde 前例（`ExploredTiles`）に倣い素直。
- [x] **1レグ解決の純関数境界**: `route.ResolveLeg(edge, load) LegResult{Cost, TempDelta, EncounterChance, FrontAdvance}` に確定。state/system は適用のみ。呑まれ判定は `CaravanRun.FrontLead()`、`Visited` 更新は Phase 3 の踏破処理で行う。
- [x] **生成不変条件の言葉**: 「**分岐区間の各層で≥2**」に限定して assert 済み（`hasBranchingNext` で次層が2ノード以上のノードのみ検査）。合流層とは衝突しない。
- [ ] **既存「ダンジョン選択→単発潜行」との接続**: 置換か当面併存か（§コメント）。まず独立導線で立ち上げ、動いてから既存を畳むのが安全。
