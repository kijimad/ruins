# ECS コンポーネント取得の型安全化（ジェネリックラッパー）

## 背景

ECS のコンポーネント取得は現在、呼び出し側で手書きの型アサーションを行っている。

```go
name := world.Components.Name.Get(entity).(*gc.Name)
```

問題点:

- `SliceComponent.Get(entity)` は `interface{}` を返すため、`.(*gc.Name)` を毎回手書きする。
- エンティティが当該コンポーネントを保持しない場合、`Get` が `nil` を返し、アサーションで **panic** する。
- フィールド（`Name`）とデータ型（`*gc.Name`）の対応が **コンパイラに保証されず規約頼み**。取り違えても型検査で防げない。
- テスト含め約574箇所存在する。

## 要件

- コンポーネント取得を **コンパイラが型検査できる** 形にし、手書きアサーションを排除する。
- **save のシリアライズ挙動を変更しない**（golden/roundtrip テストがあり、壊すリスクを避ける）。
- 段階的に移行し、各段階で `make check`（全テスト + lint）が通ること。
- 移行完了後、`forcetypeassert` を有効化して生の型アサーションを恒久的に禁止する（設計 + linter の合わせ技）。

## 設計

### 方式: ジェネリックラッパー型 `Component[T]`（案A）

`Components` の各 `*ecs.SliceComponent` フィールドを、型パラメータ付きのラッパーに変える。

```go
// Component は *ecs.SliceComponent をラップし、型付きの取得を提供する。
type Component[T any] struct {
	*ecs.SliceComponent // 埋め込み。Join/AddComponent 用の DataComponent/joinable を昇格で満たす
}

// MustGet は型付きで取得する。保持していない場合はpanicする
func (c Component[T]) MustGet(entity ecs.Entity) *T {
	return c.Get(entity).(*T)
}

// TryGet は型付きで取得する。保持していない場合は (nil, false)
func (c Component[T]) TryGet(entity ecs.Entity) (*T, bool) { ... }
```

`Components.Name` は `*ecs.SliceComponent` から `Component[Name]` になる。呼び出し側:

```go
name := world.Components.Name.Get(entity).(*gc.Name) // 旧
name := world.Components.Name.MustGet(entity)        // 新（型はコンパイラ保証）

// comma-ok は TryGet（1メソッドで全対応）
tt, ok := world.Components.TileTemperature.TryGet(entity) // 旧: .Get(e).(*gc.TileTemperature)
```

### 重要な設計上の制約

- **型付きメソッドは `Get` と命名できない**。`ecs.DataComponent` は `Get(Entity) interface{}` を要求し、Join/AddComponent はこれを昇格メソッドで満たす。もし `Get(Entity) *T` を定義すると埋め込みの `Get(Entity) interface{}` をシャドウして DataComponent を満たさなくなり、Join/AddComponent が壊れる（`go test` で確認済み。`go build` は `_test.go` を無視するため検出できない点に注意）。よって型付きは `MustGet`/`TryGet` という別名にし、埋め込みの `Get` はそのまま残す。
- **コード生成は不要**。ラッパーは単一のジェネリック型で全コンポーネントに適用される。新規コンポーネントは `Foo Component[Foo]` と宣言するだけで `MustGet` が `*Foo` を返す。型束縛は構造体宣言に自己記述される。

### reflect 消費側の対応

`*ecs.SliceComponent`（ポインタ）を前提にしていた reflect を、`Component[T]`（struct値）に対応させる。

- `InitializeComponents`: 各インスタンスが別型のため型switchできない。マーカーインターフェース `sliceComponentIniter{ initSlice(*ecs.Manager) }` を `Component[T]` に実装し、`field.Addr().Interface()` の型switchで `sliceComponentIniter` / `**ecs.NullComponent` に振り分ける。
- `component_debug_state.go`: `field.IsNil()` は struct に panic するため除去。`DataComponent` へのキャストのみで判定する（Component[T]・NullComponent 両方が満たす）。
- `GetSingleton[T](world, comp *ecs.SliceComponent)` は `comp.Get()` しか使わないため、引数を `ecs.DataComponent` に変更する（Component[T] と *ecs.SliceComponent の両方を受けられる）。

### save は無変更

調査の結果、save のシリアライズは `converter.go` の値ベース関数（`hpToSaveData(gc.HP)` 等）で、`*ecs.SliceComponent` を直接触っていない。`save:"true"` タグは reflect で読まれておらず未使用。よって save 本体はラッパー化の影響を受けない（golden/roundtrip テストで確認済み）。

### 移行の非破壊性

埋め込みの `Get(Entity) interface{}` は残るため、フィールドをラッパー化しても既存の `.Get(e).(*gc.X)` は動き続ける。よって core（ラッパー + フィールド + init）投入後も全体ビルドは緑のまま、呼び出し側を段階的に `.MustGet` へ移行できる。

## 採用しなかった方法

- **案C: 型付きアクセサメソッドをコード生成（`GetName` 等）**。フィールドは `*ecs.SliceComponent` のまま、`Components` に生成メソッドを追加する案。reflect/save を触らない点は魅力だが、**生成器と生成物（約300行）を恒久的に保守**する必要があり、コンポーネント追加のたびに再生成が要る。案A は生成が不要で、型束縛が構造体宣言に自己記述され、新規コンポーネントが自動で型安全になるため、保守性で優る。プロトタイプ（Phase 1 コミット）で案C を実装したが、案A の実現可能性（ラッパーが DataComponent を満たし Join/AddComponent が無変更で動くこと、init reflect を書き換えられること、save が無変更で済むこと）を実地検証で確認し、案A に切り替えた。
- **案B: ジェネリック自由関数 `Get[T](c, e)`**。構造体・reflect・save は無変更だが、`Get[HP](Components.Name, e)` のように**コンポーネントと型の対応を縛れず**、安全性がむしろ現状より弱いため見送る。
- **`forcetypeassert` の先行有効化**。727件を一斉に `, ok` 化 or nolint するのは方針に反する。**アクセサ移行の後**に有効化し、生アサーションを置換済みにしてから禁止する。

## コメント

- 当初は「save が具体型前提」として案A を退けていたが、調査で save シリアライズは値ベース（converter.go）で reflect 非依存と判明し、具体型に依存する reflect は init とデバッグ表示のみだった。コストは想定より小さく、案A の保守性メリットが上回ると判断して切り替えた。
- `Get` 命名が使えない制約は Go のインターフェース満足（埋め込みメソッドのシャドウ）に起因する。`MustGet`/`TryGet` の別名で回避する。
- `TryGet` は単一のジェネリックメソッドで全コンポーネントに適用されるため、案C の「51個の未使用 TryX」のような YAGNI 問題は生じない。

generics 主体で、正当な型アサーションイディオム（ebiten の `SubImage(rect).(*ebiten.Image)`、

変換して恒久的に解消する。linter を抑制するのではなく linter の意図（未チェックアサーション排除）に

| 箇所 | 変換 |

検討したうえで `path-except` を緩める。

### Phase D: 書き込み経路の型安全化

Phase A〜C は取得（読み取り）のみを型安全化していた。付与（書き込み）側は
`entity.AddComponent(world.Components.X, data)` が `data interface{}` を受けるため、
フィールドとデータ型の取り違えが依然コンパイラで防げなかった（`Components.HP` に
`&gc.Melee{}` を渡してもコンパイルが通る）。これを塞ぐため型付きの自由関数
`AddComponent[T](entity, comp Component[T], data *T)` を追加し、直接呼び出しを移行する。
メソッド `comp.Add(entity, data)` は「comp に entity を Add する」と主客が逆に読めるため、
`entity.AddComponent(comp, data)` の並びを保つ自由関数を採用した。

- [x] 自由関数 `AddComponent[T any](entity ecs.Entity, comp Component[T], data *T)` を追加
- [x] Component[T] 対象の直接 `AddComponent` 呼び出し（save/manager.go の復元処理29件を含む計52件）を `gc.AddComponent(entity, comp, data)` に移行
- [x] 移行対象外を明確化: NullComponent マーカー（型パラメータを持たない）、`AddEntities` の reflect 経路（`entity.go`）、`StateChangeRequest`（インターフェース値を格納する特殊コンポーネント）はそのまま
- [x] `make check` 通過（save の golden/roundtrip を含む全テストPASS）

### Phase E: NullComponent を廃止し Component[T] に統一

Phase D 時点では `*ecs.NullComponent`（データを持たないマーカー、14個）だけが
`Component[T]` の外にあり、`AddComponent` の型安全化が及ばない例外として残っていた。
これを解消するため、**全コンポーネントを `Component[T]` に統一**する。マーカーは
空構造体を型引数にした `Component[T]`（例: `Component[Player]`, `Component[Boss]`）で表す。
これにより `gc.AddComponent` が例外なく全コンポーネントに適用でき、書き込みの型安全が100%になる。

- [x] 14個の `*ecs.NullComponent` フィールドを `Component[T]` 化（Faction系は `Component[FactionXxxData]`、Boss は新規 `type Boss struct{}`）
- [x] `initComponentField` の `**ecs.NullComponent` 分岐を削除し、`sliceComponentIniter` 単一ケースに簡素化
- [x] マーカーの直接付与（Boss の `&ecs.NullComponent{}` 等、計23件＋Boss1件）を `gc.AddComponent` に移行
- [x] reflect 経路（`AddEntities`）は無変更で動作（マーカーは型名一致、Faction は `String()` 一致で既に型付きデータを渡していたため）
- [x] 構造テスト・デバッグ表示・doc.go の NullComponent 参照を更新
- [x] `make check` 通過（save の golden/roundtrip を含む全テストPASS）

#### 採用理由と代償

- マーカーが空データスライスを持つ（空構造体ポインタは Go の `zerobase` に集約されるため実質ポインタ1個分）。ゲーム規模では無視できる。
- `NullComponent.Get` は常に nil を返すが、`Component[T]` 化後は保持時に `&gc.X{}` を返すようになる。マーカーの存在判定は `HasComponent`（タグベース）で行うため実害はない。
- NullComponent の「データを持てない」強制（`Set` が no-op）は失われるが、型が空構造体のため実質的に無害。
