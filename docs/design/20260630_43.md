# AI行動システム統一

## 背景

現在、AIエンティティの行動処理が2つの独立したシステムに分かれている:

| システム | 対象 | 行動決定方式 | ファイル |
|---------|------|------------|---------|
| Processor | 敵・中立NPC | 状態機械 + ActionPlanner | enemy_processor.go, action_planner.go, state_machine.go |
| SquadProcessor | 隊員 | 優先度チェーン + SquadPolicy | squad_processor.go |

### 問題

1. **プロセッサが2つ**: 同じAPループ、死亡チェック、移動ロジックが両方に存在する
2. **コード重複**: ランダム移動、経路探索移動、逃走、移動候補計算が散らばっている
3. **コンポーネント構造の非対称性**: NPC用AIという同じ目的なのに、コンポーネント構成が異なる

| エンティティ | 戦闘方針 | 移動方針 | アイテム方針 |
|---|---|---|---|
| 敵 | Disposition | MovementPattern | なし |
| 中立NPC | Disposition | MovementPattern | なし |
| 隊員 | SquadPolicy.Combat | SquadPolicy.Position | SquadPolicy.ItemPickup/ItemHandling |

どのコンポーネントにどの処理が適用されるかが一覧できず、新しいNPC種別を追加するときにどちらの構造を使うべきか迷う。

4. **AIChasing は死んだコンポーネント**: 書き込みが存在せず、hud_extractor.go の読み取り1箇所のみ。削除すべき

## 要件

1. ポリシーコンポーネントを1つに統一する（Disposition + MovementPattern + SquadPolicy → AIPolicy）
2. プロセッサを1つにまとめる
3. 共通ロジック（APループ、移動ヘルパー）を一箇所にまとめる
4. 行動モデルをポリシーチェーンに統一する（状態機械を廃止する）
5. 既存の動作を変えない

## 設計

### AIPolicy コンポーネント

Disposition、MovementPattern、SquadPolicy を1つの `AIPolicy` に統合する。全AIエンティティが同じコンポーネント構造を持つ。

既存の `CombatPolicy`（squad.go）は `ai_policy.go` に移動し、`CombatIgnore` を追加する。既存の `PositionPolicy` は `MovementPolicy` に統合して削除する。`ItemPickupPolicy` と `ItemHandlingPolicy` は型ごと `ai_policy.go` に移動する。結果、`squad.go` には `SquadMember` マーカーのみ残る。

```go
// internal/components/ai_policy.go

// PlannerType は適用する行動計画の種別を表す
type PlannerType int

const (
    // PlannerRoaming はAIStateのタイミングサイクルで行動する。敵・中立NPC用
    PlannerRoaming PlannerType = iota
    // PlannerSquad はリーダー追従とアイテム処理を含む。隊員用
    PlannerSquad
)

// CombatPolicy は戦闘時の行動方針を表す。
// 既存の squad.go の CombatPolicy を拡張して移動したもの
type CombatPolicy int

const (
    // CombatAttack は敵対行動。視界内のプレイヤーまたは敵を攻撃する
    CombatAttack CombatPolicy = iota
    // CombatEvade は回避行動。敵から距離を取って逃げる
    CombatEvade
    // CombatIgnore は無関心。戦闘に反応しない。被ダメージで CombatAttack に変化する
    CombatIgnore
)

// MovementPolicy は非戦闘時の移動方針を表す
type MovementPolicy string

const (
    // 敵・中立NPC 向け
    MovementRandom      MovementPolicy = "random"
    MovementPatrol      MovementPolicy = "patrol"
    MovementWallHug     MovementPolicy = "wallHug"
    MovementStationary  MovementPolicy = "stationary"
    MovementWander      MovementPolicy = "wander"
    MovementTerritorial MovementPolicy = "territorial"
    MovementSwarm       MovementPolicy = "swarm"

    // 隊員向け
    MovementEscort   MovementPolicy = "escort"
    MovementVanguard MovementPolicy = "vanguard"
    MovementRetreat  MovementPolicy = "retreat"
)

// AIPolicy は全AIエンティティ共通の行動ポリシーを保持する。
// 敵、中立NPC、隊員が同じ構造で方針を表現する
type AIPolicy struct {
    // Planner は適用する行動計画の種別。resolvePlanner が参照する
    Planner PlannerType
    // CombatDefault は初期方針。逃亡後に復帰する基準値
    CombatDefault CombatPolicy
    // CombatCurrent は現在の方針。被ダメージなどで変化する
    CombatCurrent CombatPolicy
    // Movement は非戦闘時の移動方針
    Movement MovementPolicy
    // ItemPickup はアイテム拾得方針。隊員のみが使用する
    ItemPickup ItemPickupPolicy
    // ItemHandling はアイテム処理方針。隊員のみが使用する
    ItemHandling ItemHandlingPolicy
}

// ReactToHostile は被ダメージ時に戦闘方針を変化させる。
// CombatIgnore は反撃のため CombatAttack に遷移する
func (p *AIPolicy) ReactToHostile() {
    switch p.CombatDefault {
    case CombatIgnore:
        p.CombatCurrent = CombatAttack
    case CombatAttack, CombatEvade:
        // 既に戦闘的または回避中なので変化なし
    }
}

// ResetCombat は戦闘方針をデフォルトに復帰させる
func (p *AIPolicy) ResetCombat() {
    p.CombatCurrent = p.CombatDefault
}

// AllSquadMovementPolicies は隊員UIで巡回可能な移動ポリシーを返す
func AllSquadMovementPolicies() []MovementPolicy {
    return []MovementPolicy{
        MovementEscort, MovementVanguard, MovementPatrol,
        MovementStationary, MovementRetreat,
    }
}

// AllSquadCombatPolicies は隊員UIで巡回可能な戦闘ポリシーを返す。
// CombatIgnore は中立NPC用であり、隊員には選択させない
func AllSquadCombatPolicies() []CombatPolicy {
    return []CombatPolicy{CombatAttack, CombatEvade}
}
```

### エンティティ別コンポーネント構成

統一後、全AIエンティティが同じコンポーネントセットを持つ:

| コンポーネント | 敵 | 中立NPC | 隊員 | 役割 |
|---|---|---|---|---|
| AIMoveFSM | ○ | ○ | ○ | AI制御マーカー。Join条件 |
| AIVision | ○ | ○ | ○ | 視界距離と追跡対象 |
| **AIPolicy** | ○ | ○ | ○ | **統一ポリシー（設定）** |
| **AIState** | ○ | ○ | ○ | **ランタイム状態** |
| SquadMember | - | - | ○ | 隊員マーカー |

AIState は旧 AIRoaming のリネーム。AIPolicy が設定（Planner種別を含む）、AIState がランタイム状態を担う。全AIエンティティが同じコンポーネントセットを持ち、差異は SquadMember マーカーの有無と AIPolicy の値だけになる。適用される Planner は `AIPolicy.Planner` フィールドで決定され、SquadMember の有無では判断しない。

### 行動モデルの統一: ポリシーチェーン

現在の敵AIの「状態機械」は、実は2つの関心事が混在している:

1. **行動優先度の判定**: 「追跡 > 逃走 > パターン移動 > 待機」— ポリシーチェーンそのもの
2. **タイミング管理**: 「Nターン待機してからMターン移動する」— AIState のデータ

状態機械（`StateMachine` インターフェース、`state_machine.go`）を廃止し、両方をポリシーチェーンで表現する。AIState.SubState の遷移は Plan() 内でインラインに管理する。

```
敵・中立NPCのPlan():
  1. 視界内 + CombatAttack → 追跡（SubState=Chasing, タイマーリセット）
  2. 視界内 + CombatEvade → 逃走（SubState=Fleeing, タイマーリセット）
  3. 追跡中 + 見失って3ターン以内 → 捜索継続
  4. 逃走中 + 見失って時間経過 → ResetCombat(), Driving に遷移
  5. SubState=Driving → MovementPolicy に従って移動。時間経過で Waiting に遷移
  6. SubState=Waiting → 待機。時間経過で Driving に遷移

隊員のPlan():
  1. HP低下 → 後退
  2. エリア外 → 帰還
  3. 視界内の敵 + CombatAttack → 攻撃/接近
  4. 視界内の敵 + CombatEvade → 回避
  5. バックパック内アイテム + Distribute → 転送
  6. 足元/視界内アイテム + Pickup → 拾得
  7. MovementPolicy に従って移動
```

どちらも「上から評価して最初にマッチした行動を返す」ポリシーチェーンになる。

### 統一プロセッサ

1つの `Processor` が全AIエンティティを単一パスで処理する。行動計画のロジックは `Planner` インターフェースで抽象化し、エンティティのコンポーネント構成からランタイムに解決する。

```
Processor.ProcessAll()
  └── 全AIエンティティ → processEntity(entity)
                            │
                            ├── resolvePlanner(entity) → Planner
                            └── runAPLoop(entity, planner)
```

### Planner インターフェース

```go
// internal/aiinput/planner.go

// Planner はエンティティの行動計画を担うインターフェースを表す。
// 各ターンのAPループ内で呼ばれ、次のアクションを返す
type Planner interface {
    // Plan は次のアクションを計画する。アクションがない場合は nil を返す
    Plan(world w.World, entity ecs.Entity) (activity.Behavior, activity.ActionParams)
}
```

### ProcessAll と resolvePlanner

```go
// internal/aiinput/processor.go

type Processor struct {
    logger       *logger.Logger
    visionSystem VisionSystem
}

// ProcessAll は全AIエンティティを処理する
func (p *Processor) ProcessAll(world w.World) error {
    world.Manager.Join(
        world.Components.AIMoveFSM,
        world.Components.GridElement,
    ).Visit(ecs.Visit(func(entity ecs.Entity) {
        p.processEntity(world, entity)
    }))

    return nil
}

func (p *Processor) processEntity(world w.World, entity ecs.Entity) {
    if entity.HasComponent(world.Components.Dead) {
        return
    }
    planner := p.resolvePlanner(world, entity)
    if planner == nil {
        return
    }
    runAPLoop(world, entity, planner, p.logger)
}
```

### コンポーネントと Planner の対応表

`resolvePlanner` は `AIPolicy.Planner` フィールドを参照して適用する Planner を決定する。「どのエンティティにどの処理が適用されるか」はコンポーネントの有無ではなく、AIPolicy の値で一意に決まる:

| PlannerType | Planner | 優先度チェーンの特徴 | 対象エンティティ例 |
|---|---|---|---|
| `PlannerSquad` | `squadPlanner` | リーダー追従、アイテム処理を含む優先度チェーン | 隊員 |
| `PlannerRoaming` | `roamingPlanner` | AIState のタイミングサイクルで非戦闘移動を制御する | 敵、中立NPC |

```go
func (p *Processor) resolvePlanner(world w.World, entity ecs.Entity) Planner {
    policy := world.Components.AIPolicy.Get(entity).(*gc.AIPolicy)

    switch policy.Planner {
    case gc.PlannerSquad:
        ctx, ok := gatherSquadContext(world, entity)
        if !ok {
            return nil
        }
        return &squadPlanner{ctx: ctx, logger: p.logger, vision: p.visionSystem}
    default:
        ctx, err := gatherEntityContext(world, entity)
        if err != nil {
            return nil
        }
        return &roamingPlanner{ctx: ctx, logger: p.logger, vision: p.visionSystem}
    }
}
```

### Planner 実装

```go
// internal/aiinput/planner_roaming.go

// roamingPlanner はAIStateのタイミング状態を使うポリシーチェーンで行動計画を行う。敵・中立NPC用
type roamingPlanner struct {
    ctx    *entityContext
    logger *logger.Logger
    vision VisionSystem
}

func (rp *roamingPlanner) Plan(world w.World, entity ecs.Entity) (activity.Behavior, activity.ActionParams) {
    policy := rp.ctx.Policy
    aiState := rp.ctx.State
    canSeePlayer := rp.vision.CanSeeTarget(...)

    // 優先度1: 戦闘反応
    if canSeePlayer && policy.CombatCurrent == gc.CombatAttack {
        rp.transitionToChasing(aiState, turnNumber)
        return chaseAction(...)
    }
    if canSeePlayer && policy.CombatCurrent == gc.CombatEvade {
        rp.transitionToFleeing(aiState, turnNumber)
        return fleeAction(...)
    }

    // 優先度2: 戦闘継続
    if aiState.SubState == gc.AIStateChasing {
        return rp.planChasingContinuation(...)
    }
    if aiState.SubState == gc.AIStateFleeing {
        return rp.planFleeingContinuation(...)
    }

    // 優先度3: 非戦闘移動（AIState タイミングで Waiting/Driving を切り替える）
    rp.updateIdleTiming(aiState, turnNumber)
    if aiState.SubState == gc.AIStateDriving {
        return rp.planMovementAction(...)  // MovementPolicy に従う
    }
    return waitAction(entity, "AI待機")
}
```

### スポーン時のバリデーション

MovementPolicy を1つの enum に統合した結果、型レベルでは全値が全エンティティに設定可能になる。不正な組み合わせを防ぐため、スポーン関数でバリデーションする:

```go
// spawn.go

// validateAIPolicy は AIPolicy の整合性を検証する。
// PlannerType と SquadMember の一致、MovementPolicy とエンティティ種別の組み合わせを検証する
func validateAIPolicy(policy *gc.AIPolicy, hasSquadMember bool) error {
    if policy.Planner == gc.PlannerSquad && !hasSquadMember {
        return fmt.Errorf("PlannerSquad には SquadMember コンポーネントが必要です")
    }
    if policy.Planner != gc.PlannerSquad && hasSquadMember {
        return fmt.Errorf("SquadMember には PlannerSquad が必要です")
    }

    switch policy.Planner {
    case gc.PlannerRoaming:
        switch policy.Movement {
        case gc.MovementEscort, gc.MovementVanguard, gc.MovementRetreat:
            return fmt.Errorf("非隊員に隊員用の移動ポリシーが指定されています: %s", policy.Movement)
        }
    case gc.PlannerSquad:
        switch policy.Movement {
        case gc.MovementWallHug, gc.MovementSwarm, gc.MovementTerritorial,
             gc.MovementRandom, gc.MovementWander:
            return fmt.Errorf("隊員に非隊員用の移動ポリシーが指定されています: %s", policy.Movement)
        }
    }
    return nil
}
```

### CombatCurrent のセーブ方針

`CombatCurrent` はランタイムで変化するフィールドだが、セーブ対象に含める。理由:

- セーブ対象の隊員の場合、`CombatCurrent` はプレイヤーがUIで設定した値（CombatAttack or CombatEvade）であり、保存すべきプレイヤー意図を反映する
- 隊員の `CombatDefault` は常に `CombatAttack` であり、`ReactToHostile()` による変化はない（CombatAttack の場合は変化なし）
- 敵・中立NPC はセーブ対象外（save タグなし）のため、被ダメージによる一時的な `CombatCurrent` 変化がセーブに残る問題は発生しない

## 採用しなかった方法

| 方法 | 不採用理由 |
|-----|----------|
| AIBehaviorChain コンポーネント | インターフェーススライスをECSコンポーネントに持たせるとシリアライズ不可。隊員のポリシーはランタイムに変更されるのにチェーンをスポーン時固定にする矛盾がある。敵AIは状態機械で1要素チェーンになりチェーン化のメリットがない |
| 2プロセッサ + 共通ヘルパーのみ | APループ・死亡チェック・エンティティ走査が依然として2箇所に残る。コンポーネント構造の非対称性も解消しない |
| Movement と Position を別フィールドにする | 「隊員にだけ意味のあるフィールド」が増え、コンポーネント構造の統一感が薄れる。同じ MovementPolicy enum に統合した方が、Planner が必要な値だけ解釈するシンプルな設計になる。型レベルで不正な組み合わせを防げなくなるトレードオフは、スポーン時バリデーションで対処する |
| Behavior Tree | 現在のAIの複雑さに対してオーバーエンジニアリング |
| Utility AI（スコアリング方式） | チューニングが難しく、ローグライクでは決定論的な行動の方がプレイヤーに理解しやすい |
| 敵AIの状態機械を維持する | 状態機械の実態は「優先度判定 + タイミング管理」の2つの関心事の混在であり、ポリシーチェーン + AIState データで同等の動作を表現できる。モデルを統一した方が全NPCの行動ロジックが同じ構造で理解できる |

## コメント

### AIMoveFSM の将来

`AIMoveFSM` は空のマーカー構造体で、AIPolicy がその役割を果たせる。ただし `dungeon.go` の OnStop クリーンアップや `hud_extractor.go` など複数箇所で Join 条件に使われているため、今回は維持する。プロセッサ統一が安定した後、別の設計ドキュメントで `AIMoveFSM` 廃止を検討する。

### Planner の拡張

新しいNPC種別は `resolvePlanner` に分岐を追加するか、既存の roamingPlanner の AIPolicy 値で対応する。Planner 実装を増やすのは、根本的に異なる行動決定モデルが必要な場合のみ。

### AIState（旧 AIRoaming）の統一

全AIエンティティが AIState を持つ。AIPolicy が設定（方針）、AIState がランタイム状態（現在のサブステート、タイミング、スポーン地点）を担う。隊員もタイミングサイクルを利用できるようになり、例えば MovementPatrol の隊員が待機→巡回を繰り返せる。

### state_machine.go の廃止

状態機械の実態は「行動優先度の判定」と「タイミング管理」の2つの関心事の混在だった。ポリシーチェーン統一により、優先度判定は Plan() の上位優先度として表現し、タイミング管理は Plan() 内のインライン処理（`updateIdleTiming`）として AIState のデータを直接操作する。`StateMachine` インターフェースと `state_machine.go` は不要になる。
