# CI ベストプラクティス施策の提案（安全性・開発効率・カバレッジ精度）

## 背景

ECSエンジンのArk移行とセーブ層刷新（PR #475）を機に、CIをゲーム開発向けに
さらに強化する余地を洗い出す。本ドキュメントは実装確定ではなく、費用対効果を
比較したうえで採否を判断するための提案カタログである。

### 現状のCI（導入済み・重複提案しない）

- `check.yml`: docker build / cross-compile / build-steam / test(+octocov) /
  flaky-check(`COUNT=20`) / check-ui / lint(golangci-lint) / image(ghcr push)
- `claude-code-review.yml` / `claude.yml`: 自動レビューと対話
- `wasm_build.yml`: WASMビルド + カバレッジ + APIドキュメント + Pages公開
- `semantics_release.yml`: semantic-release
- `dependabot.yml`: github-actions / gomod / docker / npm を weekly 更新
- テストは `xvfb-run go test -v -cover -shuffle=on -timeout=60m`

### 現状の弱点（本提案の起点）

- テストに `-race` がない。`gamelog.SafeSlice`(sync.Mutex)・Ebitenの
  goroutine・ECS共有状態があるためデータ競合が潜在しうる。
- 到達可能性ベースの脆弱性検知（govulncheck）がない。push時に「main依存に
  27件の脆弱性」警告が出ているが、実際に危険なものを絞り込めていない。
- 画像goldenテスト（states/widgets等）はあるが、CI失敗時に差分画像を確認できない。
- カバレッジが単一の総計値のみで、パッケージ別・高精度の内訳や閾値ゲートがない。
- `concurrency` グループがなく、連続pushで旧runが走り続ける。
- `image` ジョブが廃止済みの `::set-output` を使用。一部アクションが
  `@master`/`@main`/`@beta` 参照でpinされていない。

### CI衛生

- `image` ジョブの `::set-output`（廃止済み）→ `$GITHUB_OUTPUT` へ移行。
- `@master`/`@main`/`@beta` のアクションをタグ/SHAにpin（サプライチェーン対策。
  dependabot-actionsが更新PRを出す）。

### カバレッジ戦略の高精度化（複数値・高精度）

現状は octocov が「総計カバレッジ％ + code-to-test比 + 実行時間」の単一粒度のみ。
以下で「複数の値を、より高い精度・粒度で」出す。

#### 1. covermode を上げて情報量を増やす

`set`（各文の到達 bool）→ `count`（実行回数）に変更すると、到達だけでなく
「どれだけ実行されたか」まで得られる。`-race` と併用する場合は `atomic` が必須。

```makefile
# 例: report ターゲット
go test -covermode=count -coverprofile=coverage.out $(PKGS)
# -race ジョブと統合するなら atomic
go test -race -covermode=atomic -coverprofile=coverage.out $(PKGS)
```

#### 2. パッケージ別カバレッジを出す（単一総計→複数値）

総計1値ではなく、パッケージ単位の内訳を常時可視化する。`go tool cover -func`
は関数単位＋ファイル末尾に総計を小数1桁で出す。

```makefile
coverage-detail: ## パッケージ・関数別カバレッジを出力する
	go test -covermode=count -coverprofile=coverage.out $(PKGS)
	go tool cover -func=coverage.out            # 関数別 + 総計
	go tool cover -html=coverage.out -o coverage.html  # ドリルダウン用
```

- `coverage.html` を `upload-artifact` すればブラウザで行単位まで確認可能。
- 関数別出力で「重要関数が0%」を見逃さない（総計だと埋もれる）。

#### 3. 重要パッケージへ個別閾値を設ける（値を複数持つ）

グローバル1閾値ではなく、リスクの高いパッケージへ個別の下限を課す。これが
「複数の値を高精度で管理する」の実体。

| パッケージ | 下限（案） | 根拠 |
|-----------|-----------|------|
| `internal/save` | 85% | セーブ破損はプレイヤー影響が最大 |
| `internal/world/query` | 80% | ゲームロジックの中核参照層 |
| `internal/activity` | 75% | 行動・戦闘の分岐が多い |
| `internal/world/gameaction` | 75% | ダメージ計算等の数式 |
| その他 | 60% | 全体下限 |

octocov は総計に対する `coverage.acceptable` を持つ。パッケージ別ゲートは
octocov 単体では不足するため、`go tool cover -func` の出力をパッケージ集計して
閾値判定する小さなスクリプト（またはmatrixジョブでパッケージ別に実行）で補う。

```yaml
# .octocov.yml への追加案（総計ゲート）
coverage:
  acceptable: 70%          # 下回るとPRを赤に
codeToTestRatio:
  acceptable: 1:1.2        # テスト量の下限も数値で固定
```

#### 4. 表示精度と欠測の扱い

- `go tool cover -func` は `xx.x%` の小数1桁。より細かい判定が要るなら、
  coverage.out を集計して小数2桁で算出する（statement数が分母なので、
  精度を上げても本質は「文カバレッジ」である点は変わらない）。
- Goの標準ツールは**文（statement）カバレッジのみ**で分岐カバレッジは出せない。
  分岐まで見たい場合は別ツール検討（ただし導入コスト大）。本提案では
  「パッケージ別＋関数別＋重要パッケージ閾値」で実用的な精度を確保する方針。

#### 5. カバレッジ計測からの除外の明示

生成物・UI・oapi等（既に `grep -v -e /editor-ui/ -e /oapi/` で除外）に加え、
描画専用コード等の測りにくい領域は octocov 側で扱いを明示し、分母のノイズを減らす。

### 開発ナレッジの skills 化（反復作業の自動化）

このリポジトリは CLAUDE.md に暗黙のルールが多く（コンポーネント追加時の更新箇所、
serde互換の平坦化、Arkの死亡Get panic回避など）、手順漏れが回帰の温床になる。
反復する開発手順を Claude Code の skill（`.claude/skills/` にコミット）として
共有し、貢献者・CI・自動レビューが同じ手順を参照できるようにする。

#### skill 候補（本リポジトリ固有の根拠つき）

| skill | 目的・自動化する手順 | 根拠 |
|-------|---------------------|------|
| `add-component` | コンポーネント追加時に `InitializeComponents` / `Components` struct / `skipComponents`(serde除外要否) / `doc.go` を漏れなく更新し、必要なら往復テストも追加 | CLAUDE.md/components.go の「この関数とComponents構造体の両方を更新」。手順漏れが serde 破損に直結 |
| `save-compat-check` | `saveDataVersion` 変更時に後方互換fixtureの追加とロード検証を促す（A3と連動） | 丸ごと保存へ移行済み。フォーマット変更でプレイヤーのセーブを壊すリスク |
| `ruins-review` | プロジェクト固有レビュー観点をチェックリスト化（Arkの死亡Get/Has panic、`Upsert` 統一、平坦化のserde互換、クエリはロック回避のため最後まで反復、エンティティ参照はID空間保存で整合 等） | 本PR一連のレビューで繰り返し登場した論点 |
| `coverage-detail` | パッケージ別・関数別カバレッジの確認と重要パッケージ閾値の判定を起動 | 上記カバレッジ戦略の実行支援 |
| `new-design-doc` | `docs/design/YYYYMMDD_NN.md` の連番採番＋テンプレ展開＋必須セクション雛形 | 規約の定型作業。採番ミス防止 |

- skill と slash command の使い分け: 手順が複数ステップで判断を伴うものは skill、
  単純起動は command。いずれも `.claude/` 配下にコミットし全貢献者で共有する。
- 単一の情報源: `claude-code-review.yml` のレビュー観点は `ruins-review` skill と
  CLAUDE.md を参照させ、観点の二重管理を避ける。

### レビューフィードバックの skills 反映ループ

レビューやペア作業で得た学びを「その場限り」で終わらせず、再発防止の仕組みへ
昇格させる。判定基準は「同種の指摘が2回以上出たか」。

```
レビュー/作業で指摘・学び
        │
        ▼
  feedback として記録（agent-memory の feedback 種別 / なぜ・どう適用を明記）
        │
        ├─ 恒久ルール化 → CLAUDE.md に追記（設計・コード規約）
        ├─ 手順の自動化 → 該当 skill のチェック項目に追加（例: ruins-review）
        └─ 機械検知可能 → golangci-lint のルール有効化 or カスタム linter 化
                             （機械化できたものは skill から卒業させ二重化を避ける）
```

- 例: 「Arkの死亡エンティティに `Has`/`Get` すると panic」→ まず `ruins-review` の
  チェック項目、機械検知可能なら将来 linter へ昇格。
- 例: 「セーブ対象コンポーネント追加時の更新漏れ」→ `add-component` skill の必須手順。
- skill 自体も育てる: 誤検知・冗長な項目はレビューで削り、`make check` で担保できる
  ようになった項目は skill から除去して肥大化を防ぐ。
- 既存資産との接続: `claude-code-review.yml`（sticky comment）と agent-memory の
  feedback を、この昇格ループの入口として活用する。

## 採用しなかった方法

- **分岐カバレッジツールの導入**: Go標準は文カバレッジのみ。専用ツールは導入・
  維持コストが高く、当面はパッケージ別＋関数別＋重要パッケージ閾値で実用精度を
  確保できるため見送り（将来の再検討余地は残す）。
- **全パッケージ一律の高い閾値**: 描画・UI等は測りにくく、一律高閾値は
  無意味なテストを誘発する。リスクベースで重要パッケージのみ高く設定する。
- **flaky-check への `-race` 統合**: raceは低速で `COUNT=20` と併用すると
  CI時間が膨張する。race専用ジョブとして分離する。

## コメント

- 優先順（費用対効果）: ①A3 セーブ後方互換 → ②B1 golden差分upload →
  ③A1 race + B2 concurrency + B4/カバレッジ閾値（いずれも極小労力の即効施策）。
- カバレッジは「単一総計を眺める」から「重要パッケージ別の複数値を閾値で守る」へ
  発想を変えるのが本質。表示桁数より粒度と閾値ゲートが効く。
