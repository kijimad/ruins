# ダンジョン間設計

## 背景

3つのダンジョン（亡者の森、灰の洞窟、忘却の廃都）、ダンジョン選択メニュー、帰還精算、GameProgressによるクリアフラグ管理が実装済み。ダンジョン間の進行構造とクリア条件を定義する。

## 設計

### ダンジョン構成

3つのダンジョンはすべて20Fで、どれからでも挑戦できる。解禁制ではない。今後、階層数の異なるダンジョンを追加する可能性がある。

| ダンジョン | 階層数 | 温度 | 特徴 |
|-----------|--------|------|------|
| 亡者の森 | 20F | 極寒(0℃) | 凍晶に覆われた森。弱い敵が多いが、極寒に注意 |
| 灰の洞窟 | 20F | 寒い(5℃) | 地下に広がる鍾乳洞。複雑な地形と中程度の敵 |
| 忘却の廃都 | 20F | 穏やか(15℃) | 古代文明の都市跡。気温は穏やかだが強敵が潜む |

### ダンジョン選択メニュー

街のポータルから開くダンジョン選択メニューに以下の情報を表示する。

- ダンジョン名
- クリア済みマーク（GameProgressを参照）
- 説明文

`Definition`に`Description`フィールドを追加する。

```go
type Definition struct {
    ...
    Description string // ダンジョン説明文
}
```

現在のMessageStateベースの選択メニューをそのまま拡張する。選択肢テキストにクリアマークと説明文を埋め込む。専用ステートは作らない。

階層数はUIに表示しない。未踏の遺跡の全容がわかるのは不自然であり、「どこまで続くかわからない」緊張感を保つ。階層数のヒントは街のNPCの会話で出す（例:「20階より深いことはないんじゃないか?」）。

表示例:
```
★ 亡者の森
  凍りついた森に、かつて猟師たちが分け入った。
  戻った者は少ない。冷気が骨まで届く。

  灰の洞窟
  灰色の岩壁に凍晶が脈のように走っている。
  奥に進むほど、静かになる。

  忘却の廃都
  古代の都市が、そのまま凍りついている。
  誰が何を忘れたのか、もう誰も知らない。
```

### 全クリアイベント

3つのダンジョンすべてのボスを撃破した状態で帰還すると、イベントが挿入される。

#### 判定タイミング

街（TownState）のOnStartで判定する。毎回の帰還時に判定が走るが、`AllClearedSeen`フラグで1回限りの発火を保証する。判定コストはほぼゼロなので問題ない。

```
街のOnStart → 全クリア判定
  → 未達成: 通常の街（現行動作）
  → 達成+未視聴: イベントStateをPush → Seen = true
  → 達成+視聴済み: 通常の街（現行動作）
```

#### イベントシステム

イベントごとに個別のフラグを定義するのではなく、共通の`EventState`構造体で管理する。イベントIDは文字列定数で識別し、`GameProgress`に個別フィールドを追加する必要がない。

```go
// EventState はイベントの状態を表す
type EventState struct {
    Active bool // 発火条件を満たしている
    Seen   bool // 視聴済み
}

type GameProgress struct {
    ClearedDungeons map[string]bool
    Events          map[string]*EventState
}
```

`Active`と`Seen`を分ける理由:
- `Active`は条件成立を記録する。全クリアのように他の状態から導出できるイベントもあれば、「NPCに話しかけた」「特定アイテムを拾った」など導出できないイベントもある。統一的に`Active`で管理することで判定ロジックを共通化できる
- `Seen`は表示済みかを記録する。`Active && !Seen`が「未視聴イベントあり」の判定条件

イベントID定数の例:
```go
const (
    EventAllCleared = "all_cleared" // 全ダンジョンクリア
)
```

#### 全クリア判定

`GameProgress`にヘルパーを追加する。判定対象のダンジョン一覧は`dungeon.GetAllDungeons()`から取得する。

```go
func (gp *GameProgress) IsAllCleared() bool {
    for _, d := range dungeon.GetAllDungeons() {
        if !gp.ClearedDungeons[d.Name] {
            return false
        }
    }
    return true
}
```

全クリア条件が成立したら`Active`を立てる。街のOnStartで`Active && !Seen`を判定してイベントを挿入する。

```go
// ボス撃破時（dead_cleanup.goなど）
if gp.IsAllCleared() {
    gp.SetEventActive(EventAllCleared)
}

// 街のOnStart
if ev := gp.GetEvent(EventAllCleared); ev != nil && ev.Active && !ev.Seen {
    ev.Seen = true
    // イベントStateをPush
}
```

#### イベント内容

MessageStateでテキスト+背景画像のページ送りで実現する。既存のオープニングと同じ仕組みを使う。内容は別途定義する。

## コメント

- 解禁制にしない理由: プレイヤーが好きな順番で挑戦できることで、自分なりの攻略順を見つける楽しさが生まれる。温度が厳しいが敵が弱い森から始めるか、温度が穏やかだが強敵がいる廃都から始めるかは戦略
- MessageState拡張を選んだ理由: 専用のtabmenuベースUIは情報量を増やせるが、現時点では選択肢数が3+1（やめる）で、MessageStateで十分。ダンジョン数が増えてスクロールが必要になった時点で専用UIを検討する
- 全クリアイベントは1回限り。再度全ダンジョンを回っても発火しない
