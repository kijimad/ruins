# 有益な追加 linter の段階導入（Tier1: 正確性系、Tier2: testifylint）

## 背景

gocritic を導入（`c8dbb9b0c`）した後、さらに有益な linter を調査した。golangci-lint の未導入 linter をこのリポジトリで試走し、検出件数と価値を実測した結果、以下を段階的に導入する方針とした。

実測サマリ（`--max-issues-per-linter=0 --max-same-issues=0`）:

| linter | 件数 | 種別 |
|---|---|---|
| errorlint | 7 | 正確性（エラーラップ） |
| thelper | 5 | テスト品質 |
| nilnil | 2 | 正確性（(nil, nil) 返し） |
| wastedassign | 1 | 正確性（無駄な代入） |
| mirror | 1 | 性能（string/[]byte メソッド） |
| errname | 1 | 命名規約 |
| testifylint | 417（自動修正290） | 正確性/可読性（testify 誤用） |

見送り: perfsprint（163件、churn過大・利益薄）、recvcheck（39件、Stringer の意図的パターンが大半）、intrange/usetesting（近代化だが今回スコープ外）。

## 要件

- CLAUDE.md「linterルールは無視設定しない」に沿い、採用する linter は原則すべての指摘を解消する。誤検知や方針と衝突するチェックのみ、理由を明記して除外設定する。
- 各 linter 導入後に `make check`（全テスト + lint）が通ること。
- Tier1 と Tier2 は別コミットに分け、段階的に導入する。
- testifylint は件数が多いため `--fix` で機械修正した後、差分をレビューし、テスト挙動（golden 含む）が変わっていないことを確認する。

## 設計

### Phase 1: Tier1（正確性系、低ノイズ）

`.golangci.yml` の `enable` に以下を追加する。

- `errorlint`: `fmt.Errorf` の非ラップ動詞を検出。`errors.Is/As` を効かせる。
- `thelper`: テストヘルパの `t.Helper()` 欠落を検出。失敗時の行番号を正確にする。
- `nilnil`: `(nil, nil)` 返しを検出。センチネルエラーの導入を促す。
- `wastedassign`: 使われない代入を検出。
- `mirror`: `string(r)` 経由の非効率な変換を検出。
- `errname`: センチネルエラーの命名規約（`errXxx`）を強制。

各指摘の対応方針:

- errorlint（7件）: `%w: %v` 形式で第2エラーが `%v` の箇所。Go 1.20+ の複数 `%w` に対応させ、文脈エラーも `%w` でラップする。ただし「センチネルのみ chain に載せる意図」の箇所は個別判断し、必要なら維持理由をコメントする。
- errname（1件）: `uiResourcesErr` → `errUIResources` へリネーム。
- mirror（1件）: `trimmed.WriteString(string(r))` → `trimmed.WriteRune(r)`。
- nilnil（2件）: `resources/fonts.go` の `(nil, nil)` 返しを、呼び出し側の分岐意図を確認のうえセンチネルエラー返却へ変更する。挙動が変わらないことをテストで確認する。
- thelper（5件）: `maingame/integration_test.go` の5ヘルパ先頭に `t.Helper()` を追加。
- wastedassign（1件）: `big_room_test.go` の `variantType := ""` を宣言のみにするか、switch で直接代入して無駄な初期化を除去。

### Phase 2: Tier2（testifylint）

`.golangci.yml` の `enable` に `testifylint` を追加する。

- CLAUDE.md がテストで testify（`assert.Error`/`require.Error` 等）を必須化しており、本命の linter。
- 主な内訳: require-error 約199件（エラー検証を `require` にして失敗時に停止）、float-compare 約88件（float の `Equal` 比較を `InEpsilon/InDelta` に）、`assert.Len/Empty/Positive` 等の可読性系。
- 手順:
  1. `golangci-lint run --enable-only testifylint --fix` で機械修正（約290件）。
  2. 残りを手動修正。float-compare は許容誤差の妥当性を確認する。
  3. `git diff` をレビューし、`assert`→`require` によりテストの停止位置が変わっても検証内容が同等であることを確認する。
  4. `make check` で全テスト・golden の通過を確認する。
- 方針と衝突するチェックが判明した場合は、`disable` に理由付きで追加する（例: 過度に細かい可読性チェック）。

## 採用しなかった方法

- **perfsprint（163件）**: `fmt.Sprintf("%d", x)`→`strconv.Itoa` 等の微性能最適化。ホットパス以外では利益が薄く、変更量（churn）が過大なため見送る。
- **recvcheck（39件）**: value/pointer receiver の混在検出。大半は `String()`（値レシーバの Stringer）と `Update()` 等（ポインタレシーバ）を併用する State 型の意図的パターンであり、修正の価値が薄いため見送る。
- **intrange / usetesting（27 / 4件）**: 近代化系で低リスクだが、今回の正確性重視スコープから外す。将来別途検討する。
- **hugeParam（gocritic、128件）**: 既に不採用（値渡しの広範なポインタ化はエイリアシングのリスクと大差分を招く）。

## コメント

- errorlint と nilnil は「機械的に潰す」のではなく、エラーの意味（センチネルのみ chain に載せるか、文脈も載せるか）や呼び出し側の分岐意図を確認して対応する。
- testifylint は件数が多いが大半が自動修正可能で、CLAUDE.md のテスト方針に最も合致する本命。正確性（require-error, float-compare）を優先し、可読性系は差分を見て取捨する。
