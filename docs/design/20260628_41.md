# 隊員（メンバー）システム実装計画

## 背景

`20260628_40.md` で隊マネジメントシステムへの転換を決定した。本ドキュメントはその中核となる隊員システムの実装計画を定める。

現在のコードベースには以下の基盤がある:
- ECSフレームワーク（goecs v2）による柔軟なエンティティ構成
- `FactionAlly/Enemy/Neutral` のファクション管理
- `SpawnPlayer/SpawnEnemy/SpawnNeutralNPC` の3種のスポーン関数
- `Abilities`（6属性）、`Skills`（23スキル）、`CharModifiers` の能力システム
- AI行動（`AIMoveFSM/AIRoaming/AIVision/AIChasing/Disposition/MovementPattern`）
- 装備・インベントリ管理（Location系コンポーネント）

隊員は「プレイヤー側のAI制御エンティティ」であり、既存の敵AIとプレイヤーエンティティの中間に位置する。隊員は探索済みエリア内でポリシーに基づいて自律行動し、リーダーは弱い戦闘能力で前線を切り開く。

## 設計

### エンティティ構成

隊員エンティティは既存のプレイヤーと敵の要素を組み合わせる。

```
プレイヤーから引き継ぐ:
  - Name, Description
  - Abilities（6属性）
  - Skills
  - CharModifiers
  - HP, HealthStatus
  - TurnBased（AP）
  - WeightCapacity
  - FactionAlly
  - GridElement, Position, SpriteRender, BlockPass

敵から引き継ぐ:
  - AIMoveFSM
  - AIVision
  - Disposition（Ally固定）
  - MovementPattern
  - CommandTable（戦闘コマンドテーブル）

新規追加:
  - SquadMember（リーダーへの参照）
  - SquadPolicy（位置・戦闘・アイテム回収・アイテム処理のポリシー）
  - MemberAppearance（外見情報）

不要:
  - Player（プレイヤーマーカー）
  - Camera（リーダーのみ）
  - Wallet（リーダーのみ）
  - AIRoaming/AIChasing（隊員専用AIに置き換え）
  - Dialog（会話不要）
  - DropTable（隊員はドロップしない）
```

#### SquadPolicy

```go
// SquadPolicy は隊員の自律行動を制御するポリシーを保持する
type SquadPolicy struct {
    Position     PositionPolicy
    Combat       CombatPolicy
    ItemPickup   ItemPickupPolicy
    ItemHandling ItemHandlingPolicy
}

type PositionPolicy int

const (
    PolicyEscort   PositionPolicy = iota // 護衛: リーダーの近くにとどまる
    PolicyVanguard                       // 前衛: リーダーの前方に展開する
    PolicyPatrol                         // 巡回: 探索済みエリア内を自律巡回する
    PolicyHold                           // 待機: その場にとどまる
    PolicyRetreat                        // 撤退: 出口に向かって移動する
)

type CombatPolicy int

const (
    PolicyAttack CombatPolicy = iota // 攻撃: 敵を攻撃する
    PolicyEvade                      // 回避: 戦闘を避けて逃げる
)

type ItemPickupPolicy int

const (
    PolicyPickup ItemPickupPolicy = iota // 回収: 探索済みエリアのアイテムを拾う
    PolicyIgnore                         // 無視: アイテムを拾わない
)

type ItemHandlingPolicy int

const (
    PolicyKeep      ItemHandlingPolicy = iota // 保持: アイテムを持ち続ける
    PolicyDistribute                          // 分配: 運搬役に渡す
    PolicyDismantle                           // 分解: その場で分解して素材にする
)
```

#### MemberAppearance

```go
// MemberAppearance は隊員の外見情報を保持する
type MemberAppearance struct {
    SpriteKey string // スプライトシートのキー
}
```

### 隊員AI行動システム

隊員の行動はポリシーの組み合わせで決まる。既存の敵AI（`AIRoaming/AIChasing`）は使わず、専用のシステムを実装する。隊員は探索済みエリア（リーダーが視界に入れたマス）内でのみ行動できる。

```
毎ターンの隊員行動判定:

0. エリア制限チェック
   - 現在位置が探索済みエリア外なら、最寄りの探索済みマスに移動する
   - 移動先候補が探索済みエリア外なら、その方向には移動しない

1. 位置ポリシーを評価する
   - 護衛: リーダーから2マス以内にとどまる。離れていたらリーダーに近づく
   - 前衛: リーダーの向いている方向の前方3マス以内に展開する
   - 巡回: 探索済みエリア内を自律的に巡回する
   - 待機: 移動しない
   - 撤退: 出口への経路を探索して移動する

2. 戦闘ポリシーを評価する
   - 攻撃: 視界内の敵を攻撃する。直接命令で指定されたターゲットがあれば優先する
   - 回避: 敵から距離を取る。攻撃しない

3. アイテムポリシーを評価する
   - 回収: 視界内のアイテムを拾いに行く。フィルタ条件があれば適用する
   - 無視: アイテムを無視する

4. アイテム処理ポリシーを評価する
   - 保持: 拾ったアイテムを持ち続ける
   - 分配: 運搬役の隊員にアイテムを渡す
   - 分解: その場で分解して素材にする

5. HPが低い場合の自律判断
   - HP 25%以下: ポリシーに関わらず後退する
```

## コメント

### 採用しなかった案

| 案 | 不採用理由 |
|----|----------|
| 敵AIをそのまま流用する | 敵AIは `AIRoaming/AIChasing` の2状態マシンで、命令体系と合わない。専用AIを作るほうがシンプル |
 | 隊員をPlayer派生にする | Playerコンポーネントに紐づく処理が多すぎてAI制御と衝突する |
| 隊をエンティティにする | 隊自体にコンポーネントを持たせるより、各隊員がリーダーを参照するほうがECSの設計に沿う |
| ポリシーをコンポーネントではなくリソースで管理する | 隊員ごとに異なるポリシーを持つため、コンポーネントが自然 |

### 既存コードへの影響

- `GetPlayerEntity()` は `Player + FactionAlly` でフィルタしているため、SquadMember（FactionAlly だが Player なし）は返さない。変更不要
- `GetVisibleEnemies()` は `FactionEnemy` でフィルタしているため、隊員（FactionAlly）は返さない。変更不要
- 敵AIの攻撃対象選択で `FactionAlly` を参照している場合、隊員も攻撃対象になる。これは意図通り
- ターン処理で隊員のターンを追加する必要がある（リーダー→隊員→敵の順）

### リスク

- 隊員AIの「ちょうどいい賢さ」の調整が最大のリスク。フェーズ3でプロトタイプを作り、プレイテストで調整する
- アイテムポリシー（回収・分配・分解）の自動化が煩雑にならないか。ポリシー設定UIの使いやすさが鍵
- 探索済みエリアの管理コスト。マップが広くなると探索済みマスの追跡が重くなる可能性がある
