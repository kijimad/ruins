# シームレスワールド Phase 8: 状態統合・ポータルの行き先保持・オーバーマップ中心化

## 背景

[20260719_63.md](./20260719_63.md) で共存方式のステージ機構を実装し、ダンジョンの階層往復が動く。全ステージのエンティティを同一 world に共存させ、`StageBound` で所属を、`Suspended` で稼働可否を表し、往復を `stage.SwapTo` に集約した。63.md は「遺跡は『隣のステージがオーバーワールド帯』という特殊ケースにすぎない」「往復はマーカー付け替えに還元」と述べた。

本 Phase はその論理を最後まで進め、**世界をオーバーマップ中心に一本化**する。到達したい姿:

- 状態は1つ。オーバーワールドも遺跡も同じ探索状態で扱う。
- ポータル、すなわち階段・入口が行き先を保持し、往復は「触れて行き先へ swapTo」に還元される。
- 街はオーバーマップ上に生成され、新規ゲームの開始点になる。
- 「戻れない通常ダンジョン」と、その起動口であるダンジョン選択・街からの起動を廃止する。`StageKindDungeon` を撤廃する。

これは [[project-stagekind-dungeon-transitional]] に記録した「StageKindDungeon は過渡的、将来オーバーワールド+遺跡へ寄せて撤廃」の実行である。

## 前提の再確認（設計が依拠する事実）

- **`OverworldState` は既に `DungeonState` を埋め込む薄いラッパー**。`OverworldState.Update` は `DungeonState.Update` へ委譲し、末尾で `updateFront` と `maybeShift`（帯シフト）を足すだけ。描画も DungeonState へ委譲。よって統合は「この2フックを DungeonState 側へ畳む」だけで済む。
- **共存方式の忠実復元はエンティティが serde されるから**であり、**State マシンのスタックは serde 対象でない**。ゆえにステージの継続情報はスタックに預けてはならない。ECS 側、コンポーネントとシングルトンに置く。前設計案の TransPush/Pop 退避はここで reload に破綻したため棄却する。
- **`findPortalPosition` は ActiveFilter で Suspended を除外**する。退避中ステージのポータルは検索に出ない。帰還は「入口を探す」でなく「行き先をポータルが保持」で解く。

## 要件

- 状態を統合する。オーバーワールドと遺跡フロアを単一の探索状態が扱う。挙動は現在ステージの種別で分岐する。
- ポータルは行き先 `StageKey` と行き先座標を保持する。触れると `stage.SwapTo(行き先)` し、行き先座標へプレイヤーを配置する。入口・上り階段・下り階段のすべてがこの1機構に還元される。
- 遺跡はオーバーワールドの入口から入り、上り階段でオーバーワールドへ戻る。戻り先座標は入口の座標で、入場時に遺跡の上り階段へ焼き込む。
- 街はオーバーマップの開始チャンクに生成し、新規ゲームはそこで始まる。
- `StageKindDungeon`・ダンジョン選択・街からのダンジョン起動を廃止する。ステージ種別は Overworld と Ruin の2つにする。
- すべてのステージは離れても保持され、徒歩で到達し直せる。セーブ・ロードで共存状態が復元される。

## 設計

### 1. 状態統合 — DungeonState に一本化

`OverworldState` を廃し、`DungeonState` に統合する。`updateFront`・`maybeShift`・帯生成・`restoreFromSave` を DungeonState へ移し、`CurrentStage.Kind == Overworld` のときだけ実行する。

```go
func (st *DungeonState) Update(world w.World) (...) {
    // 既存の探索更新（入力・ターン・視界・カメラ・HUD・死亡判定）
    ...
    if query.GetDungeon(world).CurrentStage.Kind == gc.StageKindOverworld {
        st.updateFront(world)
        st.maybeShift(world)
    }
}
```

帯ドライバ、ChunkGen と Band は DungeonState のフィールドとして持つ。enter/exit は同一 State 内の `swapTo`（TransNone）なので State は破棄されず、遺跡滞在中もドライバはメモリに残る。退避中の帯はシフトしないので、戻ったとき ECS とドライバが一致する。reload 時は `restoreFromSave` が SeamlessBand から再構築する。State 名は責務が広がるため将来 `FieldState` 等へ改名を検討するが、本 Phase では `DungeonState` のままにして差分を絞る。

### 2. ポータルの行き先保持 — 往復の一般化

ポータルを「行き先を持つステージ・グラフの辺」に一般化する。ポータルのエンティティに行き先を持たせる。

| ポータル | 行き先(StageKey) | 行き先座標 |
|---|---|---|
| オーバーワールドの遺跡入口 | 遺跡1階 | 遺跡の入口 |
| 遺跡の上り階段(最上階) | オーバーワールド | 入った入口の座標 |
| 下り階段 | 1つ深い階 | その階の入口 |
| 上り階段(中間階) | 1つ浅い階 | その階の下り階段 |

- ポータルに触れる → そのポータルの行き先 `StageKey` へ `stage.SwapTo` → 行き先座標へ `MovePlayerToPosition`。種別分岐なしの単一機構。
- 未訪問の行き先は SwapTo の generate で生成し、生成時に「戻りのポータル」へ戻り先座標を焼き込む。
- 行き先座標はポータルのコンポーネントに持つので serde 対象。reload を跨いで戻り先が保たれる。「どの入口から入ったか」を階段自身が覚える。
- これで `findPortalPosition` の先着ヒューリスティックを廃し、複数入口があっても曖昧さが無い。

### 3. 街をオーバーマップに生成

- 街はステージでなくオーバーワールドの地物。開始チャンク、`eastIndex==0` 付近に街の構造物、店・収納・隊員雇用などの prop を決定的に配置する。
- 新規ゲームは、キャラ作成後に街を含むオーバーワールドを生成し、街の位置でプレイヤーを開始する。`TownState` と、そこへ至る導線を廃止する。街の機能は overworld 上の prop への相互作用で提供する。

### 4. 破棄スコープ

一方向ダンジョンと街への完全離脱が無くなるため、「潜行を丸ごと捨てる」完全離脱が消える。全ステージは共存・徒歩往復可能で保持される。`DungeonState.OnStop` の全フィールド一括 purge を廃止する。world を捨てるのはタイトルへ戻る・ロードのときで、そこは world リセットが担う。ステージ単位の破棄が要る場合は `stage.Purge` をハンドラで明示的に呼ぶ。

### 5. StageBound タグ付け（前提）

- チャンク生成直後に `stage.Bind(world, gc.NewOverworldStage())` で帯エンティティを束縛する。
- `CurrentStage` は開始時に `NewOverworldStage()`。以降は swapTo が更新する。

## 採用しなかった方法

- **OverworldState を残し TransPush/Pop で遺跡へ出入り**: 前案。State スタックが serde 外なので、遺跡中セーブ→ロードで戻り先 State が消え TransPop が破綻する。共存方式の「継続は ECS に置く」原則に反するため棄却。
- **専用 RuinState を新設**: 遺跡専用状態。DungeonState と描画・往復・入力をまるごと重複させる。遺跡はダンジョンと構造が同じで、統合 State + ポータル行き先で足りるため不要。
- **帰還を入口 prop 探索で解く**: `findPortalPosition` で入口を探して戻る。入口が複数あると先着で誤る。ポータルが行き先座標を持つほうが曖昧さゼロで serde 安全。
- **街を専用ステージにする**: 街を Ruin と同様の別ステージにして入口から入る。だが「開始点」なので入場でなく初期在場が要り、オーバーマップ地物とするほうが素直。

## コメント

- 統合は `OverworldState` が既に `DungeonState` ラッパーなので小さい。まず挙動保存の統合から入り、その後にポータル一般化・遺跡・街生成・旧ダンジョン撤廃を重ねる。
- 撤廃(旧ダンジョン・選択・街起動)は代替(遺跡・街オーバーマップ化)が入ってから最後に行い、常にゲームが動く状態を保つ。
- 新たな不変条件: Update は `CurrentStage.Kind` で分岐する。swapTo が CurrentStage を更新するので、遷移中に古い種別で誤動作しないようテストで固める。

## 追記(2026-07-21): シングルトン Dungeon の分解

### 背景

`components.Dungeon` シングルトンは「冒険1セットの情報」という当初のドメイン概念だが、共存方式で
オーバーワールドと遺跡が同一 world に共存し「1回の潜行」という単位が消えた結果、責務が異なる状態を
詰め込んだ grab bag、実質的な神クラスになっている。現在の11フィールドは次の5系統に分かれる。

- **現ステージの identity**: `CurrentStage`。これが「現在ダンジョンのキー」。
- **identity と重複**: `Depth`(=`CurrentStage.Depth`。dungeon.go:429 で既に `CurrentStage.Depth` を使用)、`DefinitionName`(ほぼ `CurrentStage.Name`。ただし地上帰還で `""` にクリアするズレあり)。
- **現ステージのビュー状態**: `Level`(現ステージ寸法)、`ExploredTiles`(現ステージの探索履歴。進入時リセット)。
- **無関係な状態の誤帰属**: `GameTime`(グローバル時間。turn/temperature/status が参照)、`MinimapSettings`(UI 設定)、`SelectedWeaponSlot`(プレイヤー/戦闘 UI)、`VisibleTiles`/`LightSourceCache`/`NeedsForceUpdate`(VisionSystem の一時状態)。
- **オーバーワールド固有の永続状態**: `SeamlessBand`(帯の serde 対象)。

加えて「オーバーワールドか」の判定方法が乱立している。`CurrentStage == NewOverworldStage()`(名前キー比較)、`CurrentStage.Depth == 0`、`SeamlessBand.Active`、`Front.Active`、`session != nil`、`isSeamless()` が場所判定として互換的に使われるが、これらは実は3つの別概念を混同している。

1. **現ステージがオーバーワールドか**(場所): frost 描画・帯シフトゲート・ascend の戻り先。
2. **この State が帯モードか**(`isSeamless`/`session != nil`): 遺跡滞在中も真。State は破棄されず帯を保持するため。
3. **前線が有効か**(`Front.Active`): 帯の性質。共存方式では遺跡滞在中も真のまま残る。

`Front.Active` を「オーバーワールドにいる」の代理に使う誤りが実害を出している。霜(修正済み)に加え、`temperature.go` の `frostZoneModifier` と `move.go` の `frontAllowsMoveTo` が `Front.Active` だけで判定し、遺跡内で寒さ効果や移動制限が漏れる同型のバグを抱える。名前文字列での直接判定もマジック定数依存で脆い。

### 要件

- 現ダンジョンの identity は `CurrentStage` に集約する。そこから導出できる冗長フィールドは持たない。
- 現ステージの状態はステージ側に置き、`CurrentStage` で引く。共存方式の理念「継続は ECS に置く」に沿わせる。
- ダンジョンと無関係な状態は本来の持ち主へ移す。
- 分解の各段はゲームが常に動く状態を保ち、`make check` で担保する。

### 設計

4つの軸で分解する。

0. **「オーバーワールドか」の判定を一本化する**
   - 単一の述語 `StageKey.IsOverworld()` を正典にし、乱立した判定を置き換える。名前文字列(`Name == OverworldStageName`)でなくフィールドで判定する。オーバーワールドは唯一の深度0ステージなので `Depth == 0` を実装とし、不変条件をメソッドに閉じ込める。将来この不変条件が崩れるならメソッドだけ直せばよい。
   - 3概念を明確に分ける。「現ステージがオーバーワールドか」は `CurrentStage.IsOverworld()`。「State が帯モードか」は `isSeamless()`/`session` を維持し名前で意図を示す。「前線効果」は前線の有無でなく **オーバーワールドかつ前線有効** で gate する。
   - 前線効果(frost・temperature・move)を `CurrentStage.IsOverworld()` で gate し、遺跡内への漏れを塞ぐ。ロード復元判定(`factories.go` の `SeamlessBand.Active`)も、現ステージ由来の判定へ寄せられるか検討する。

1. **identity を `CurrentStage` に集約する**
   - `Depth` を削除し、全参照を `CurrentStage.Depth` へ統一する。冗長で二重管理のため。
   - `DefinitionName` を `CurrentStage.Name` から導出する。地上帰還で `DefinitionName` を `""` にクリアしていた挙動は、オーバーワールドの `CurrentStage.Name`(=`OverworldStageName`)で自然に表せるので不要になる。

2. **`Level`・`ExploredTiles` をステージ側へ移す(採用: 案2)**
   - 各ステージに「ステージメタ」エンティティを1つ持たせ、`Level` と探索履歴を保持する。`StageBound` で自ステージに束縛し、共存の他エンティティと同様に退避・復元される。
   - 参照は `CurrentStage` のメタエンティティを引く。swap でシングルトンを差し替えるのでなく、現ステージのメタを見る。これで「現ステージ状態」がシングルトンから消える。
   - **帯シフトの考慮**: オーバーワールドの探索履歴は帯シフトでキーが平行移動する(`worldstream/band.go` の `translateTileKeyMap`)。この帯ローカル性はメタエンティティへ移しても保つ。ステージメタ単位でシフト対象を特定できるので、むしろ帯とそれ以外の切り分けが明確になる。
   - **serde**: 探索履歴は `GridElement` struct キーで serde 不可のため、現状どおりロード時に再構築するか、serde 可能な表現(スライス等)へ変える。ステージメタが `StageBound` を持つことで、どのステージの履歴かは復元後も自明になる。

3. **無関係な状態を本来の持ち主へ追い出す**
   - `GameTime` → 独立したグローバル時間シングルトン。潜行と無関係で全編にわたる。
   - `MinimapSettings` → UI 設定(config/resources)。実行時に変わらない静的値。
   - `SelectedWeaponSlot` → プレイヤー/装備状態。
   - `VisibleTiles`/`LightSourceCache`/`NeedsForceUpdate` → VisionSystem 内部、または Vision 専用シングルトン。視界計算の一時状態で、描画が読むだけ。

これらを終えると `Dungeon` は「identity(`CurrentStage`)＋オーバーワールド帯(`SeamlessBand`)」まで痩せる。名前も責務に合わせて改名を検討する(例: `WorldLocation` 等)。

### 採用しなかった方法

- **案1: シングルトンを現ステージのビューのまま残す**。`Level`・`ExploredTiles` を swap で差し替える今の方式。変更は小さいが、「現ステージ状態」と「グローバル状態」の同居が残り、共存方式の理念から外れる。将来ステージが増えると差し替えコストと取り違えバグの温床になる。今回の一連の帰還バグ(Level 未復元・現ステージ誤判定)も、状態がシングルトンに集約され swap で暗黙に差し替わる構造が一因だった。
- **`GameTime` を `Dungeon` に残す**。時間は潜行と無関係で全編グローバル。誤帰属を続けると「Dungeon が無いと時間が読めない」不自然な依存が残る。
- **場所判定に `StageKind` enum を再導入する**。かつて撤廃した種別フラグを StageKey に戻す案。明示的だが、全 StageKey 生成箇所・serde・比較に種別を通す必要があり侵襲的で、撤廃で得た単純さを損なう。当面は唯一の深度0という不変条件を `IsOverworld()` に閉じ込めれば足りる。不変条件が崩れる要件が出たときに enum 化を再検討する。
- **場所判定を名前文字列のまま各所で行う**。`Name == OverworldStageName` を散らす現状。マジック定数依存で、意味の違う3概念が混ざり、境界でバグる。実際に霜・気温・移動の漏れを生んだ。
- **一括で全フィールドを移す**。段階(H1〜H4)に分けず一度に行うと `make check` が長く赤になり、共存の帰還経路の回帰を切り分けにくい。設計方針どおり常に動く状態を保って進める。

### 段階(Phase H)

リスクの低い順に H0→H4 で進める。各段で `make check` を通し、共存の往復・serde を回帰確認する。

- H0: `IsOverworld()` 一本化と前線効果の gate。temperature/move の潜在バグを塞ぐので先に片付ける価値が高い。
- H1: `Depth` 削除。最小・確実。
- H2: 無関係状態の追い出し。各々独立に移せる。
- H3: `Level`・`ExploredTiles` のステージメタ移設。serde・帯シフト・クエリを含む本丸。
- H4: `DefinitionName` の導出化。
