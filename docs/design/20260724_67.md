# 難所調査から抽出した自走可能な改善のバックログ

## 背景

リポジトリの難所を7領域で徹底調査した。ECS ライフサイクルの共存方式、状態機械、戦闘と AI、セーブ、マップ生成、描画、OpenAPI 連携である。その過程で「人間の判断を必要とせず、エージェントが自走で実施できる改善」が複数見つかった。本ドキュメントはそれを一箇所に集め、着手可否を判別できる形で残す。

判別の軸は自走可能度とする。基準は「挙動不変か、機械的に検証可能で、失敗しても `make check` が止めること」である。この基準で3階層に分ける。

- 🟢 即着手可: 挙動不変または純追加。参照ゼロを実測済み、あるいは純粋関数テスト。`make check` だけで担保できる。
- 🟡 初期仕分けのみ人手: 最初に allowlist 整備やゴールデン基準づくりで一度だけ人手が要り、以後は自走する。仕組み化そのものが再発防止になる。
- 🔴 判断要につき除外: 将来の意図、設計方針、大量の文章判断を含み、機械的に決められない。参考として理由付きで残す。

調査環境は X11 ヘッダ不在のため `go tool deadcode` と `go test` が完走しない。よって静的読解と grep による分析であり、削除候補は本体参照ゼロを実測して裏付けた。

## 要件

- 各提案は独立した小さな変更に分割でき、単体でレビューできる。
- 🟢 は着手時に挙動を変えない。担保は `make check` で完結する。
- 🟡 は初期の人手を一度だけ挟み、以後は `make lint` または `make check` が自動で守る。機械化できた項目は該当 skill から外し、二重管理を避ける。この方針は CLAUDE.md「フィードバックの昇格」に従う。
- 🔴 は着手前に人へ確認する。無人では触らない。

## 設計

### 🟢 即着手可・低リスク

#### G1 未使用の公開型・メソッド削除

定義のみで本番・テストいずれからも参照されない exported シンボル。YAGNI 方針の残滓で、ゲッター/セッターを予防的に生やした跡である。本体参照ゼロを実測済み。

- 型: `components.SheetImage`, `components.Renderable`, `components.FloatPool`。いずれも定義のみ、ECS 登録表にも無い。
- メソッド: `widgets/hud.SetEnabled`, `widgets/hud.GetConfig`, `widgets/tabmenu.UpdateTabs`。interface 要求でないことを確認済み。
- 孤児 interface: `engine/resources.ResourceProvider`。コンパイル時アサーションも消費コードも無く、言及はコメント1箇所のみ。

誤検出の除外を明記する。`widgets/styled.GradientLine` の `PreferredSize`/`IsValidated` は grep 上ゼロだが、ebitenui の `PreferredSizeLocateableWidget` 実装であり interface 経由で呼ばれる。削除しない。

#### G2 陳腐化コメントの修正

参照先の実装が存在しない、または実態と食い違うコメント。

- `editor-ui/server/api.ts:119`「Go の itemSortKey と同じロジック」。Go 側に `itemSortKey` は存在しない。実測ゼロ。Go の項目ソートは `world/query/sort.go` の名前順のみでロジックも異なる。コメントを「Go 側に対応実装は無い。これは TypeScript 独自の分類ソート」と書き換えるか、誤解を招く参照コメント自体を削除する。
- `resources/resources.go:11` の `ResourceProvider` 孤児コメント。G1 の型削除と同時に整理する。

#### G3 純粋関数のテスト追加

world・実乱数・Ebiten に依存しない純粋関数で、直接テストが無いか薄いもの。X11 非依存で設計できる。

- `mapplanner/line_corridor.go:114` `createLShapedCorridor`。水平のみ、垂直のみ、L字4象限、始点と終点が同一、逆方向、連続性で斜め飛びが無いこと。
- `mapplanner/wall_type.go:30` `WallType.String`。全9値と不正値 panic。対の `AutoTileIndex.String` は網羅済みだが `WallType` 側が未テスト。
- `balance/stats.go` の `Percentile` は p=1.0、p>1、p<0 の境界。`Median` は偶数長で下中央値を返す仕様。`RunStats` は P5 < Median < P95 の順序。

#### G4 エラー内容検証の追加

`require.Error`/`assert.Error` だけで、どのエラーに落ちたかを見ていない既存テスト。分岐が複数あるほど価値が高い。

- `consts/weight_test.go:104` の7ケースを、形式不正・数値不正・負値・単位不正の各メッセージへ `assert.ErrorContains` で紐付ける。
- `balance/balance_test.go:160` の裸の `require.Error`、`mapplanner/warp_portal_planner_test.go` ほか。sentinel エラーは `assert.ErrorIs` へ、メッセージ検証は `assert.ErrorContains` へ置換する。同ファイル 170-172 は既に `assert.Contains` 済みなので対象外。行番号は着手時に確認する。

#### G5 タイル名の定数化

定数 `consts.TileNameWall`/`Floor`/`Void`/`Dirt` が存在するのに、同一ファイル内で生文字列 `"wall"`/`"floor"` が混在する。文字列リテラル完全一致なので挙動不変で置換できる。

- 対象は `cave_planner.go`、`forest_planner.go`、`big_room.go`、`room_draw.go`。wall/floor/void/dirt のみに限定する。
- 除外: `forest`/`tree`/`grass`/`water` などは対応定数が無い。定数追加は設計判断なので今回の対象外とする。

#### G6 重複リテラルのヘルパ化

`styled/table.go:150` と `styled/eui.go:313` の `NewGradientLine(res.GradientLine, color.RGBA{255,255,255,80}, 1)` が同一引数で重複する。区切り線の生成を `newSeparatorLine(res)` へ集約する。

#### G7 テスト専用の公開シンボルを unexport

本番未参照でテストのみが使う exported シンボルのうち、同一パッケージの white-box テストが使うものは unexport が安全。`gamelog` の構造化 API 群、`dungeon.GetAllDungeons`、`logger.ResetConfig`、`activity.GetLastResult`/`ResumeActivity` など。外部 `_test` パッケージ併存の `world/query` 系は壊れるため個別確認とし、🟡 扱いにする。

### 🟡 初期仕分けのみ人手・以後は自走

#### Y1 命中/ダメージ式の単一情報源化

命中率とダメージが本番 `activity/attack.go:353-388` と balance `combat.go:37-57` で二重実装されている。同期担保はコメント宣言のみで、スキル倍率や元素耐性の有無で既に非対称化している。計算コアを `formula` パッケージへ純粋関数として抽出し、両者が同一関数を呼ぶ。乱数ロールは引数注入して純粋に保つ。

```go
// internal/formula に新設する純粋関数の骨子。
// 乱数の結果を引数で受け取り、ECS にも rand.Rand にも依存しない。
func RollHit(attacker, defender CombatStats, weaponAcc int, roll int) (hit bool, critical bool)
func RollDamage(attacker, defender CombatStats, weaponDmg int, roll int, critical bool) int
```

`depguard` に `formula_guard` があり gostd のみ許可なので、抽出先として層も正しい。抽出が重ければ暫定策として、固定 seed で両実装を N ケース回し出力一致を assert するゴールデン一致テストを先に敷く。乖離した瞬間に赤くなる。機械化後は `ruins-review` skill の「同じ計算順序」項目を削除する。

#### Y2 ActiveFilter 付け忘れの機械検知

ステージ跨ぎクエリで生の `ecs.NewFilterN` を使うと退避中エンティティが silent leak する。現状は `ruins-review` skill の grep 手運用のみで、本体に生 `ecs.NewFilter` が約50箇所あり玉石混交である。`go/packages` で型引数を検査し、ステージ束縛コンポーネントを含む生 NewFilter を列挙して allowlist と差分でエラーにする。

「ステージ束縛集合」は登録表を単一情報源にできるよう、メタ情報を1列足す。

```go
// internal/components/genspec のエントリに列を追加する案。
type Entry struct {
    Field       string
    Comment     string
    StageScoped bool // 追加。true の型を含む生 NewFilter を検知対象にする
}
```

初期の allowlist 仕分けだけ人手を挟む。singleton 引きなど正当な生 NewFilter を除外する。安定後に `go/analysis` linter へ昇格し `make lint` へ組み込む。

#### Y3 未使用公開シンボルの継続検知

`unused` は `default: standard` に含まれ、`unparam` も11行目で有効化済みで、両者とも既に効いている。ただし `unused` すなわち staticcheck U1000 は exported シンボルを対象外とする仕様のため、G1 の未使用公開シンボルは検出されない。`make lint` は `go tool deadcode -test` も回すが、これは到達不能な関数のみを対象とし、未使用の型やエイリアスは拾わない。よって Y3 は「G1 のような未使用公開シンボルを X11 なし環境で継続検知する手段の調査と対策」とする。候補は deadcode の対象や運用の見直し、exported の未使用を検出する analyzer の追加、CI での参照ゼロ grep 化など。導入判断が入るため 🟡 とする。

#### Y4 internal/world の doc.go 整備

`internal/world` は1行のパッケージコメントのみで doc.go が無い。規約は説明・使い分け・責務・仕様の記載を求める。文面に軽い判断が要るため 🟡 とする。`internal/oapi` は生成物なので対象外。

### 🔴 判断要につき除外

着手前に人へ確認する。透明性のため理由付きで残す。

- R1 `server.gen.go` と `SaveData*` 型の生成デッドコード。両者とも参照完全ゼロを実測済み。だが削除には生成設定 `oas/config/server.yml` や `save.tsp` の変更が要り、編集 UI のバックエンドを将来 Go で実装する意図や、セーブ外部仕様として保持する意図の確認が前提になる。
- R2 `cave_planner` の `Iterations` が生成箇所で 3、デフォルトで 5。唯一の生成箇所が常に 3 を渡すためデフォルト 5 は到達不能で無害である。統一するかは設計意図の判断で、かつ golden スナップショットの再生成を伴う。型に「全然わかってない」TODO もあり仕様把握が先。
- R3 コメント規約違反の一括修正。丸括弧書きが479件、体言止めが多数。1件ずつ文の分割や言い換えという文章判断が要り、機械一括置換はできない。個別修正より `ruins-review` への観点追加か、丸括弧のみ限定的な lint 化が本筋。

## 採用しなかった方法

- ActiveFilter 検知を forbidigo の全面禁止で行う案。生 NewFilter は singleton や非ステージ束縛エンティティで正当に使うため誤検知が出る。forbidigo は型引数まで見られない。型引数を検査する AST テストか analyzer が要る。
- exhaustive の `default-signifies-exhaustive` を false へ締める案。`inputmapper.ActionID` のように一部だけ処理し残りを default に流すのが正当な enum が大量に赤くなる。不採用。
- 生成物の手編集検知を goheader などで追加する案。CI の generate-check が再生成で差分ゼロを強制しており、手編集は結果的に無効化される。屋上屋なので優先度は低い。
- G5 で `forest`/`water` など未定義のタイル名にも定数を新設する案。定数の設計判断を含むため今回の自走範囲から外す。

## コメント

本バックログの価値の中心は 🟡 の仕組み化にある。Y1 と Y2 はいずれも難所調査で「未機械化の穴」として挙がった箇所で、機械化できれば再発防止として効き、`ruins-review` skill の該当項目を削除できる。着手は 🟢 の掃除から始め、テストで安全網を張ってから 🟡 の抽出と検知へ進むのが安全である。
