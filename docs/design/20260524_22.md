# ゲームバランス調整の方法論

## 背景

一人開発ではプレイテストのマンパワーが不足する。実際のプレイによる感覚的な調整には限界があるため、数理的な手法とシミュレーションを主軸としたバランス調整を行う。

## ゲーム進行の構造

バランス調整の前提となるゲームの構造を整理する。

- プレイヤーは街から遺跡に潜り、任意のタイミングで帰還できる
- HP・所持品はフロアをまたいで持ち越される（リセットされない）
- 1フロアの敵数は可変（最低数〜最高数の範囲）
- 深度が進むと敵テーブル・アイテムテーブルが切り替わる

この構造から、バランス分析の基本単位は **ラン（1回の潜り）** になる。フロアを独立に分析しても実態を反映しない。深度3の難易度は深度1-2でどれだけ消耗したかに依存するためである。

## バランスの核心: 帰還判断

帰還可能なローグライクでは「どの深度で引き返すか」がプレイヤーの中心的な判断になる。バランスが意味するのは以下の問いに対する回答である。

- **引き返すべきタイミングが存在するか?** 常に進み続けられるなら緊張感がない。どこかで「これ以上は危ない」と感じるポイントが必要
- **引き返すべきタイミングが読めるか?** 突然死するのでなく、消耗の蓄積から「そろそろ限界」と判断できる猶予が必要
- **もう1フロア進むリスクとリターンは釣り合っているか?** 深層ほど良いアイテムが出るなら、リスクを取る動機がある

これらを数値で制御するために、以下のフレームワークを使う。

## 方針: シミュレーション駆動のバランス調整

### Step 1: 基本指標を定義する

3つのスケールで指標を定義する。

**戦闘テンポ（1戦闘単位）:**

- **撃破ターン数（TTK）**: 通常の敵を倒すのに必要なターン数
- **被ダメージ期待値**: 1戦闘で受ける平均ダメージ
- 設計原則: 1戦闘で死ぬことはまずないが、HPは確実に削られる

**リソース減衰（深度の関数）:**

- **累積消耗量**: 深度1からその深度までに受けた累積ダメージの期待値
- **累積補給量**: 深度1からその深度までに拾える回復量の期待値
- **残HP推移**: 累積消耗量 - 累積補給量 の推移。これが0に近づくほど帰還すべき

```
残HP(深度d) = 初期HP + Σ[i=1..d](補給量_i) - Σ[i=1..d](消耗量_i)
```

**ラン全体:**

- **帰還適正深度**: 残HPが危険域に入る深度。これが「引き返すべきポイント」
- **到達深度分布**: N回のランで到達する深度の分布
- **死亡深度分布**: 帰還せず死亡した場合の深度分布

### Step 2: 期待値で大枠を検算する

現在のダメージ計算式（`internal/activity/attack.go`）から期待値を導出する:

```
基礎ダメージ = 攻撃力 + rand(1..6) + 武器ダメージ
最終ダメージ = max(基礎ダメージ - 防御力, 1)
命中率 = clamp(80 + (器用度-敏捷度)×2 + 武器命中補正, 5, 95)

EV(1ターン与ダメージ) = 命中率/100 × max(攻撃力 + 3.5 + 武器ダメージ - 防御力, 1)
TTK = 敵HP / EV(与ダメージ)
EV(1戦闘の被ダメージ) = TTK × EV(1ターン被ダメージ)
```

深度dまでの累積消耗を期待値で見積もる:

```
EV(累積消耗, 深度d) = Σ[i=1..d] EV(敵数_i) × EV(1戦闘の被ダメージ_i)
EV(累積補給, 深度d) = Σ[i=1..d] EV(回復アイテム数_i) × EV(回復量_i)
EV(残HP, 深度d) = 初期HP + EV(累積補給, 深度d) - EV(累積消耗, 深度d)
```

EV(残HP)が0を下回る深度が「期待値的な限界深度」になる。実際にはこれより手前で帰還すべきで、その余裕が分散（運の悪さ）の吸収幅になる。

### Step 3: モンテカルロシミュレーションで検証する

期待値で見えないことをシミュレーションで計測する:

- **残HPの分散**: 同じ深度でも運によってHPがどれだけばらつくか
- **突然死率**: 前の深度では余裕があったのに次の深度で死ぬ確率。これが高いと「読めない死」になる
- **敵テーブル境界**: 深度帯が切り替わるポイントでの消耗量ジャンプ
- **装備ランダム性**: 武器を拾えたランと拾えなかったランの到達深度差

計測指標:

| 指標 | 意味 |
|:-----|:-----|
| 残HP推移（深度ごとの平均・P5・P95） | 「いつ引き返すべきか」が読めるか |
| 突然死率（深度ごと） | 前深度で生存→当深度で死亡する確率 |
| 到達深度分布 | ランの到達範囲が設計意図に合うか |
| 敵テーブル境界での消耗量ジャンプ | 難易度が急変していないか |

## 実装案

### 方針

ゲーム本体の戦闘ロジック（`internal/activity/attack.go` の `calculateDamage`, `calculateHitRate`）はECSのエンティティに依存しているため、シミュレーション用に軽量な計算関数を切り出す。ゲーム本体とシミュレーションで計算式を二重管理しないよう、純粋な計算部分を共有する。

### 計算ロジックの切り出し

現在の `calculateDamage` と `calculateHitRate`（`internal/activity/attack.go`）はECSの `ecs.Entity` と `w.World` に直接依存している。シミュレーション用に純粋な値ベースの関数を切り出す:

```go
// internal/balance/combat.go

// CombatantStats は戦闘シミュレーション用のステータス
type CombatantStats struct {
    HP        int
    Strength  int
    Sensation int
    Dexterity int
    Agility   int
    Defense   int
}

// WeaponStats は武器のシミュレーション用パラメータ
type WeaponStats struct {
    Damage   int
    Accuracy int
    IsRanged bool
}

// CalcHitRate は命中率を算出する。activity.calculateHitRate と同じ計算式
func CalcHitRate(attacker, target CombatantStats, weapon WeaponStats) int {
    hitRate := 80 + (attacker.Dexterity-target.Agility)*2
    hitRate += weapon.Accuracy - 80
    if hitRate > 95 { hitRate = 95 }
    if hitRate < 5 { hitRate = 5 }
    return hitRate
}

// CalcDamage はダメージを算出する。activity.calculateDamage と同じ計算式
func CalcDamage(attacker, target CombatantStats, weapon WeaponStats, rng *rand.Rand) int {
    baseAbil := attacker.Strength
    if weapon.IsRanged { baseAbil = attacker.Sensation }
    baseDamage := baseAbil + rng.IntN(6) + 1 + weapon.Damage
    finalDamage := baseDamage - target.Defense
    if finalDamage < 1 { finalDamage = 1 }
    return finalDamage
}
```

`activity.calculateDamage` 側もこの関数を呼ぶようにリファクタすれば、計算式の二重管理を防げる。ただし初期はシミュレーション側だけで独立させ、計算式が安定したら統合する段階的アプローチでもよい。

### 1戦闘シミュレーション

```go
// BattleResult は1戦闘の結果
type BattleResult struct {
    Turns      int  // 決着までのターン数
    PlayerWon  bool // プレイヤーが勝ったか
    DamageTaken int // プレイヤーが受けた総ダメージ
}

// SimulateBattle は1戦闘を模擬する
func SimulateBattle(player, enemy CombatantStats, weapon WeaponStats, rng *rand.Rand) BattleResult {
    playerHP := player.HP
    enemyHP := enemy.HP
    turns := 0
    damageTaken := 0

    // 敵の攻撃パラメータは raw.Master.GetCommandTable() から取得して変換する
    enemyWeapon := WeaponStats{...}

    for playerHP > 0 && enemyHP > 0 {
        turns++
        // プレイヤーの攻撃
        hitRate := CalcHitRate(player, enemy, weapon)
        if rng.IntN(100)+1 <= hitRate {
            dmg := CalcDamage(player, enemy, weapon, rng)
            enemyHP -= dmg
        }
        if enemyHP <= 0 { break }
        // 敵の攻撃
        enemyHitRate := CalcHitRate(enemy, player, enemyWeapon)
        if rng.IntN(100)+1 <= enemyHitRate {
            dmg := CalcDamage(enemy, player, enemyWeapon, rng)
            playerHP -= dmg
            damageTaken += dmg
        }
    }

    return BattleResult{Turns: turns, PlayerWon: playerHP > 0, DamageTaken: damageTaken}
}
```

### ランシミュレーション

```go
// RunResult は1ランの結果
type RunResult struct {
    ReachedDepth int          // 到達深度（死亡 or 帰還）
    Died         bool         // 死亡したか
    HPByDepth    map[int]int  // 各深度終了時の残HP
}

// SimulateRun はラン全体を模擬する
func SimulateRun(master *raw.Master, tableName string, player CombatantStats, weapon WeaponStats, rng *rand.Rand) RunResult {
    result := RunResult{HPByDepth: make(map[int]int)}
    currentHP := player.HP

    for depth := 1; depth <= maxDepth; depth++ {
        // 1. 敵テーブルから敵を決定する
        //    calculateEnemyProbabilities() と同じロジックで確率を算出し、
        //    rng で敵を選択する
        // 2. フロアの敵数を決定する（最低数〜最高数の範囲で乱数）
        // 3. 各敵と戦闘し、消耗を累積する
        // 4. アイテムテーブルからドロップを決定し、回復を適用する

        enemies := rollEnemies(master, tableName, depth, rng)
        for _, enemyName := range enemies {
            enemyStats := loadEnemyStats(master, enemyName)
            br := SimulateBattle(
                CombatantStats{HP: currentHP, ...player以外はplayerから},
                enemyStats, weapon, rng,
            )
            currentHP -= br.DamageTaken
            if currentHP <= 0 {
                result.ReachedDepth = depth
                result.Died = true
                return result
            }
        }

        // アイテムによる回復
        healing := rollHealing(master, tableName, depth, rng)
        currentHP += healing
        if currentHP > player.HP { currentHP = player.HP }

        result.HPByDepth[depth] = currentHP
    }

    result.ReachedDepth = maxDepth
    return result
}
```

### 実装順序

1. **`balance/stats.go`**: パーセンタイル計算などの統計ユーティリティ。依存なし
2. **`balance/combat.go`**: `CalcHitRate`/`CalcDamage` と `SimulateBattle`。`raw.Master` からステータスを読む変換関数も含む
3. **`balance/balance_test.go`**: 1戦闘のテスト。ここで基本的な数値感覚を確認する
4. **`balance/run.go`**: `SimulateRun`。敵テーブル・アイテムテーブルを使ってラン全体を模擬する
5. **`balance/balance_test.go` に追加**: ランシミュレーションのテスト

Step 2 の時点で、計算式を `activity.calculateDamage` と二重管理するか、共通関数に切り出すかを判断する。

## リソースの汎化

ここまでHPを例にしたが、同じフレームワークは他の消耗リソースにも適用される。

| リソース | 消耗要因 | 補給手段 | 帰還圧力 |
|:---------|:---------|:---------|:---------|
| HP | 戦闘での被ダメージ | 回復アイテム、休息 | 死亡リスク |
| 弾薬 | 射撃 | 弾薬ドロップ、店 | 武器が使えなくなる |
| 燃料 | 光源の維持、移動 | 燃料ドロップ | 暗闇で行動不能になる |
| 食料 | 時間経過、行動 | 食料ドロップ | 飢餓ダメージ |

どのリソースも構造は同じで、深度の関数としての残量推移カーブを描ける:

```
残量(深度d) = 初期量 + Σ補給(1..d) - Σ消耗(1..d)
```

複数リソースがある場合、帰還判断は**最も枯渇に近いリソース**に支配される。HPに余裕があっても弾薬が尽きれば帰還せざるを得ない。シミュレーションでは各リソースの残量推移を並行して追跡し、いずれかが危険域に入る深度を「帰還適正深度」とする。

まずはHPで仕組みを作り、リソースが増えたときに同じ枠組みへ追加する。

## シミュレーションの限界と方法論の再考

参考文献の調査と実装経験から、当初の方法論にいくつかの修正が必要であることが判明した。

### シミュレーションが扱えること/扱えないこと

モンテカルロシミュレーションは「単一のシステムや局面」の分析には有効だが、ゲーム全体のバランス評価には限界がある（Boards and Barley, Toth Games）。具体的には以下の制約がある。

**シミュレーションで検証できること:**

- 特定の装備・ステータスでの1フロアあたりの消耗量
- 敵テーブル境界での難易度ジャンプの有無
- 突然死率（前深度で生存→当深度で死亡する確率）
- 回復量と消耗量のバランス

**シミュレーションでは検証できないこと:**

- プレイヤーの戦略的判断（逃走、回避、アイテム温存）
- 装備拾得のタイミングと装備成長の曲線
- 「楽しいか」「理不尽に感じるか」の主観的体験
- 複数リソースの同時管理によるトレードオフ

Toth Gamesの手法が示すように、プレイヤー判断のような不確定要素は「確率分布として近似（ファジング）」することで部分的に組み込める。ただし、これは簡略化であることを認識する必要がある。

### 装備成長のシミュレーション問題

実装で明らかになった最大の課題は、装備の変動をどうモデル化するかである。

- **素手（初期装備）で全深度をシミュレーション** → 死亡率100%。現実のゲームでは1階で武器を拾える
- **最強装備で全深度をシミュレーション** → 回復で常に全快。現実のゲームでは段階的に強くなる

現実のランでは「深度1-2で鉄の剣を拾い、深度5で鋼の剣に持ち替える」といった装備成長が起きる。これをシミュレーションで再現するには以下のアプローチが考えられる。

**方法A: 固定装備での条件別シミュレーション**

CLIフラグで装備を指定し、複数条件を比較する。最もシンプルで、「この装備があれば深度Xまで行ける」という問いに答えられる。ただしラン全体の体験は再現しない。

**方法B: 装備成長モデルの組み込み**

アイテムテーブルから武器・防具のドロップをシミュレーションし、深度ごとに最良装備に切り替えるモデルを組み込む。より現実に近いが、「プレイヤーが最適な装備を選択する」という仮定が入る。

**方法C: 複数スナップショットの組み合わせ**

深度帯ごとに「この深度で期待される装備」を手動で設定し、それぞれの深度帯をシミュレーションする。装備成長は設計者の意図として明示的に制御できる。

現状では方法Aを実装済み。方法Bは装備判断のモデル化が複雑すぎるため見送り、方法Cが次のステップとして現実的である。

### ダメージ計算式の設計原則

Stat Balancing in Roguelikes（Temple of The Roguelike）の手法は、本プロジェクトのダメージ計算式の検証に直接適用できる。

1. **属性スケールの定義**: 攻撃力1（最弱の敵）〜攻撃力20（最強の敵）を明確に配置する
2. **結果スケールの定義**: ダメージを「無害」「軽傷」「中傷」「重傷」「致命的」「即死」に分類する
3. **公理の設定**: 「プレイヤー（攻撃力5）がスライム（防御力3）を殴ると軽傷」のような具体例から計算式を逆算する
4. **一貫性の検証**: すべての属性組み合わせで表を作り、ジグザグ（非単調な結果）が出ていないか確認する

現在の計算式 `damage = max(攻撃力 + rand(1..6) + 武器ダメージ - 防御力, 1)` は線形である。Temple of The Roguelikeの記事では二次関数 `damage = a×x² + b×x + c` による非線形カーブを推奨している。線形だとステータス差が大きい場合にダメージが極端になりやすい。ただし、現状ではまだ線形で十分な段階であり、問題が具体化したら検討する。

### パワーカーブの選択

RogueBasinのPower Curveの議論に照らすと、本プロジェクトは**緩勾配のパワーカーブ**が適切である。

- 帰還可能なローグライクでは、1回のランで到達する深度範囲が限られる
- 急勾配だと「深度10の敵を深度5のプレイヤーが絶対に倒せない」状況が生まれ、帰還判断ではなく壁に当たるだけになる
- 緩勾配なら「深度10の敵は厳しいが、装備とアイテムが良ければ行ける」という判断の余地が残る
- プレイヤーと敵のステータスを同率でスケールすると実質的に変化なしになるため、深度が進むにつれて敵の成長率をわずかに上回らせることで消耗圧力を生む

### シミュレーションの適切な使い方

以上を踏まえ、シミュレーションの位置付けを修正する。

1. **シミュレーションは万能ではない**: 局所的なバランス検証ツールであり、ゲーム全体のバランスを保証するものではない
2. **条件を固定して使う**: 「装備X、ステータスYで敵テーブルZに挑んだとき」という限定的な問いに答えるツールとして使う
3. **異常値の検出に使う**: 突然死率が5%を超える深度、消耗量が1.5倍以上ジャンプする境界など、「明らかな問題」の発見に使う
4. **プレイテストの代替にはならない**: 最終的な体験の検証は実際のプレイが必要

## コメント

- 分析の基本単位はラン全体であり、フロアを独立に分析しても消耗の文脈を無視することになる
- 帰還可能なゲームでは「帰還すべき深度」がバランスの中心になる。残量推移カーブを設計ツールとして使う
- パーマデスがあるため「突然死率」を重視する。前深度で余裕があったのに突然死ぬのは理不尽に感じる
- 敵テーブル切り替え境界は消耗量のジャンプが起きやすいので、個別にストレステストする
- リソースが増えても同じフレームワーク（残量推移 + パーセンタイル計測）で扱える
- シミュレーションは「明らかな問題の発見」と「条件別の比較」に使い、ゲーム全体のバランスはプレイテストで補完する

## ヘッドレスシミュレーションの振り返りと方針転換

### 経緯

本番ECSを使ったヘッドレスシミュレーション（`internal/headsim`）を実装し、全職業×全ダンジョンの組み合わせで自動シミュレーションを実行した。結果、全職業が100%の死亡率を示した。

### 問題点

ヘッドレスシミュレーションには以下の構造的な問題がある:

1. **AIの行動が本番プレイと乖離する**: PlayerAIは近接攻撃のみで、射撃・アイテム使用・帰還判断を行わない。猟師・狙撃手のような射撃職が不当に不利になる
2. **AIの精度を上げるコストが高い**: 射撃、アイテム使用、帰還判断、装備切替など、プレイヤーの行動をすべてモデル化する必要がある。モデルが複雑になるほどバグの余地も増える
3. **本番ECSへの依存が重い**: スプライトシート読み込み、ダンジョン生成、ターンシステム登録など、バランス検証に本質的でない初期化コストが大きい
4. **結果の解釈が困難**: 死亡率100%という結果が「バランスが悪い」のか「AIが弱い」のか判別できない

### 方針転換

ヘッドレスシミュレーション（`headsim`パッケージ）は凍結し、`balance`パッケージの軽量モデルに集中する。

**凍結するもの:**
- `internal/headsim/` — 本番ECSベースのシミュレーション
- `internal/cmd/simulate_balance.go` の `--headless` モード

**継続するもの:**
- `internal/balance/` — 数式ベースの戦闘シミュレーション（`CalcHitRate`, `CalcDamage`, `SimulateBattle`）
- `internal/cmd/simulate_balance.go` の simpleモード
- `internal/balance/balance_test.go` — バランス回帰テスト

**理由:**

軽量モデルは以下の利点がある:

- AIの行動モデルが不要で、数式だけで完結する
- 入力パラメータ（ステータス・武器・敵）を変えるだけで職業間比較ができる
- CDDAのDPS回帰テストと同じ手法が直接適用できる
- テストが高速に実行でき、`make check` に組み込める

ヘッドレスシミュレーションは「本番に近い」ことが目的だったが、AIの行動モデルが不完全な限り「本番に近い」という前提自体が成り立たない。シンプルなモデルで明確な問いに答えるほうが実用的である。

## CDDA（Cataclysm: Dark Days Ahead）のバランス手法からの学び

CDDAのソースコード調査から、ruinsに応用可能な知見を整理する。

### CDDAの戦闘数式

CDDAの命中/回避は正規分布ロールで設計されている。

```
命中ロール = normal_roll(accuracy × 5, SD=25)
回避ロール = dodge × 5
命中判定 = 命中ロール - 回避ロール - サイズ補正 > 0
```

各ステータスの寄与率が明確に定義されている:

| ステータス | 命中への寄与 | 回避への寄与 | クリティカルへの寄与 |
|:-----------|:-------------|:-------------|:---------------------|
| DEX | +1/4 per point | +1/2 per point | +1% per point |
| PER | — | — | +2% per point |
| 武器スキル | +1/3 per level | — | +2.5% per level |
| 格闘スキル | +1/2 per level | — | +1% per level |
| 回避スキル | — | +1 per level | — |

ダメージ計算は防御を貫通力で相殺する仕組みで、じゃんけん的な要素を生む:

```
有効防御 = max(基礎防御 - 貫通力, 0) × 倍率
```

**ruinsへの示唆:** 現在のruinsの命中率計算 `80 + (器用度-敏捷度)×2` は線形で単純である。CDDAのようにステータスごとの寄与率を明示的に設計テーブルとして定義すると、調整の見通しが良くなる。

### CDDAの敵難易度算出

CDDAはモンスターの `difficulty` を自動算出する複合指標を持つ:

```
difficulty = 攻撃力項 + 防御力項 + 特殊能力項

攻撃力項 = (melee_skill + 1) × dice × (damage + sides) × 0.04
防御力項 = (dodge + 1) × armor × 0.04
特殊能力項 = difficulty_base + special_attacks + 8 × emissions

最終difficulty = difficulty × 生存力補正
生存力補正 = (HP + speed - attack_cost + (morale + agro) × 0.1) × 0.01
           + (vision_day + 2 × vision_night) × 0.01
```

この指標はスポーン制御や進行管理に使われる。

**ruinsへの示唆:** 現在ruinsの敵テーブルは手動で深度に配置している。CDDAのように敵ごとの複合difficulty値を自動算出すれば、テーブル配置の妥当性を機械的に検証できる。たとえば「深度3に配置された敵のdifficulty範囲が深度2より低い」といった逆転を検出できる。

### CDDAの職業バランス: ポイントコスト制

CDDAは職業をポイントコストで定量化している:

| 職業 | コスト | 特徴 |
|:-----|:-------|:-----|
| 無職 | 0pt | 基本装備のみ。基準線 |
| 警察官 | 4pt | 射撃4, 拳銃4, 格闘3, 打撃3 |
| ライフルマン | 5pt | 射撃4, ライフル4, 格闘3, 回避3 |

スキルレベル、装備、特性がすべてポイントに換算され、職業間の公平性が検証可能になっている。

**ruinsへの示唆:** 現在の全職業100%死亡率の問題を分析するために、各職業の「総合戦力スコア」を定義して比較する手法が有効である。たとえば:

```
職業スコア = Σ(能力値 × 寄与係数) + 武器DPS + 防御力 + スキル補正
```

これにより「避難民(evacuee)と軍人(soldier)のスコア差がどれだけあるか」を定量的に把握できる。

### CDDAのバランステスト手法

CDDAのバランステストは統計的手法を使い、CI/CDで自動実行される。ruinsのシミュレーションを強化する上で最も参考になる部分である。

#### a) DPS回帰テスト

全武器の期待DPS値をJSONファイルに保持し（`swords_dps.json`, `guns_dps.json` 等）、テスト時に1000+回のシミュレーションで実測DPSを算出する。期待値から乖離するとテストが失敗し、意図しないバランス変更を検出する。

**ruinsへの応用案:**

```go
// balance/expected_dps_test.go

// 期待DPS値。武器やステータスを変更したときに意図しない変化を検出する
var expectedDPS = map[string]float64{
    "素手_vs_スライム":   2.5,
    "鉄の剣_vs_スライム": 5.0,
    "素手_vs_ゴブリン":   1.8,
}

func TestExpectedDPS(t *testing.T) {
    for scenario, expected := range expectedDPS {
        actual := measureDPS(scenario, 1000)
        assert.InDelta(t, expected, actual, expected*0.1,
            "%s: DPSが期待値から10%%以上乖離", scenario)
    }
}
```

#### b) 命中率テスト

10,000回の試行で特定条件の命中率を検証する:

| 対象 | 素手 | スキル+武器 | 熟練+最良武器 |
|:-----|:-----|:-----------|:-------------|
| ゾンビ(回避2) | 60% | 80% | 97.5% |
| マンハック(回避5) | 20% | 40% | 70% |

ruinsでも同様に「深度1の敵に対する命中率は80%以上」「深度10の敵に対する命中率は50%以上」のような条件をテストに組み込める。

#### c) 統計フレームワーク

CDDAは `test_statistics.h` にAdjusted Wald信頼区間を実装し、Z99.9レベルの信頼度で検証する。`uncertain_about()` メソッドで確信が得られるまで反復を続ける仕組みもある。

**ruinsへの応用案:**

```go
// balance/stats.go に追加

// WithinConfidenceInterval は観測値が期待値の信頼区間内にあるかを検証する
func WithinConfidenceInterval(observed, expected float64, trials int, confidence float64) bool {
    z := zScore(confidence)
    se := math.Sqrt(expected * (1 - expected) / float64(trials))
    lower := expected - z*se
    upper := expected + z*se
    return observed >= lower && observed <= upper
}
```

#### d) CI/CDでの自動実行

CDDAではPRごとにバランステストが自動実行される。ruinsでも `make check` にバランス回帰テストを含めれば、パラメータ変更時の意図しない影響を早期に検出できる。

### CDDAの時間経過による敵進化

CDDAは `half_life` パラメータで時間経過による敵の進化を制御する:

```json
{
  "id": "mon_zombie",
  "upgrades": {
    "half_life": 120,
    "into_group": "GROUP_ZOMBIE_UPGRADE"
  }
}
```

`half_life: 120` は「120ターン経過で50%の確率でより強い個体に進化する」ことを意味する。これにより時間とともに世界が難しくなる。

**ruinsへの示唆:** ruinsでは深度で難易度をスケールするため直接の対応はないが、「同じダンジョンでもクリア回数に応じて敵テーブルが強化される」といったメタ進行に応用できる概念である。

### 次のアクション候補

CDDA調査から得た知見をruinsに適用する優先度順:

1. **PlayerAIの射撃対応**: 現在のAIは近接攻撃のみで、猟師・狙撃手の職業特性が活かせていない。CDDAのようにAIが武器種別に応じた行動を選択する仕組みが必要
2. **DPS回帰テストの導入**: 武器×敵の組み合わせごとに期待DPS値を保持し、パラメータ変更時の回帰テストとする
3. **敵difficulty自動算出**: 敵テーブルの深度配置の妥当性を機械的に検証する指標を導入する
4. **職業ポイントコスト制の設計**: 各職業の総合戦力をスコア化し、バランスの偏りを定量的に把握する
5. **統計的信頼区間の導入**: 現在のシミュレーション結果に信頼区間を付加し、有意な差かノイズかを判定可能にする
