# OSS 調査 2026-07

## 背景

月次の OSS 調査ルーチンとして、GitHub 検索と GitHub Trends から ruins に役立ちそうな OSS を探した。`go.mod` を確認し、`ebiten`・`ark`/`ark-serde`・`ebitenui`・`testify`・`goldie`・`urfave/cli`・`caarlos0/env`・`oapi-codegen` 等は既存依存として除外した。`internal/mapplanner/pathfinder.go`・`internal/aiinput/planner_squad.go` の BFS 経路探索、`internal/systems/vision.go`・`internal/aiinput/vision_system.go` の視界計算、`internal/components/ai.go` のステートマシン的な AI サブ状態など、既に自前実装がある領域も把握した上で候補を絞った。

GitHub Trends の月間トレンドは `fzf`・`kubernetes`・`prometheus`・`tailscale` 等インフラ/CLI 系が中心で、ゲーム・ECS・ローグライク関連は無く、そのまま採用できる候補は無かった。そのため本調査は `gh search repos` 相当のドメイン検索で得た候補のみを扱う。

このドキュメントは提案であり、採否の判断は人がレビューで行う。コード変更・依存追加は行っていない。

## 候補

### 1. ebitengine/debugui

- リポジトリ: https://github.com/ebitengine/debugui
- 概要: Ebitengine 公式チームが開発する、Ebitengine アプリ向けのデバッグ用 UI ツールキット。ボタン・スライダー・テキスト入力・ウィンドウなどのウィジェットを持ち、[microui](https://github.com/rxi/microui) の設計を踏襲する。
- ruins での用途: ゲーム内デバッグパネル、パラメータ調整用のライブチューニング UI、ECS 状態のインスペクタなど。`internal/widgets/hud/debug_overlay.go` や `internal/states/mapgen_visualizer.go` 等、既存のデバッグ表示を汎用ウィジェットで置き換えられる可能性がある。
- 成熟度: 101 stars、直近更新 2026-07-22 と活発。ただし README に「API は頻繁に変わる」旨の注記があり、開発中段階。
- ライセンス: Apache License 2.0、microui 由来コンポーネントは MIT。
- 採用時の注意: API 破壊的変更が起きやすいため、バージョン固定と追従コストを見込む必要がある。デバッグ用途なら本編のリリースサイクルとは独立して追従できるため、活発さと API 不安定さは両立する。バージョンを固定して使う分には即戦力候補と判断した。

### 2. joeycumines/go-behaviortree

- リポジトリ: https://github.com/joeycumines/go-behaviortree
- 概要: Go 向けの軽量な振る舞い木（Behavior Tree）実装。シーケンス・セレクタといった基本ノードに加え、非同期実行やコンテキスト対応のユーティリティを持つ。
- ruins での用途: `internal/components/ai.go` の `SoloAI`/`SquadAI` は `AIStateSubState`（WAIT/DRIVING/CHASING/FLEEING）による手製の状態遷移で AI 行動を表現している。敵の行動パターンが増えて分岐が複雑化した場合、振る舞い木へ寄せることで行動ロジックの見通しを保ちやすくなる。
- 成熟度: 78 stars、直近更新 2026-07-04。
- ライセンス: Apache License 2.0。
- 採用時の注意: 現状の AI ロジックはステートマシンで足りており、導入は複雑さに見合うかを要検討。YAGNI の観点から、実際に分岐が肥大化してから検討するのが妥当。

### 3. quasilyte/pathing

- リポジトリ: https://github.com/quasilyte/pathing
- 概要: グリッドベースのゼロアロケーション経路探索ライブラリ。Greedy BFS と A* を実装し、`container/heap` を避けた sparse-dense データ構造でベンチマーク上他の Go 実装より高速。
- ruins での用途: 現状 `internal/mapplanner/pathfinder.go`（マップ生成時の到達性検証）と `internal/aiinput/planner_squad.go`（敵の追跡移動）に、それぞれ独立した BFS 実装がある。実行時パフォーマンスがボトルネックになった場合や、2実装の重複を1ライブラリへ集約したい場合の選択肢になる。
- 成熟度: 135 stars、直近更新 2026-06-20。
- ライセンス: MIT。
- 採用時の注意: 1グリッドあたり8タイル種別まで、パス長は最大56ステップという制約がある。ruins のマップ規模で問題ないか要確認。既存実装を置き換えるほどの性能上の必要性が今あるかは未検証で、まずは自前実装で困っていないなら見送ってよい。

### 4. quasilyte/ebitengine-input

- リポジトリ: https://github.com/quasilyte/ebitengine-input
- 概要: Godot に着想を得た、Ebitengine 向けのアクション入力抽象化システム。キーボード・複数ゲームパッドの入力を「アクション」単位で束ね、キーリマップやマルチデバイス対応をサポートする。
- ruins での用途: `internal/input/keyboard.go` は自前のキーボード入力抽象化を持つが、ゲームパッド対応やキーコンフィグ機能は無い。`go.mod` に `hajimehoshi/go-steamworks` があり Steam 展開を見据えているため、Steam Deck 等でのゲームパッド対応・キーリマップが将来要件になるなら候補になる。
- 成熟度: 98 stars、直近更新 2026-06-16。
- ライセンス: MIT。
- 採用時の注意: 現状ゲームパッド対応の要件は確認できていない。既存の `KeyboardInput` インターフェースとの責務分担、リプレイスか併存かの設計判断が要る。

### 5. mlange-42/ark-tools

- リポジトリ: https://github.com/mlange-42/ark-tools
- 概要: ruins が使う ECS `mlange-42/ark` の作者自身による補助ライブラリ。独立した更新レートでロジックを走らせるスケジューラ、CSV 書き出し等の既製システム、共有乱数ジェネレータや更新ティックカウンタといった ECS 共有リソースを提供する。
- ruins での用途: ターン制の更新ループや、`internal/systems/` 群のスケジューリングを整理する際の参考・部品になりうる。同じ作者が保守しているため `ark` 本体の API 変化に追従しやすい可能性がある。
- 成熟度: 13 stars、直近更新 2026-04-09とやや低調。利用者は少ない。
- ライセンス: MIT。
- 採用時の注意: 小規模で採用実績が少ないライブラリ。ruins の更新ループは既に `internal/systems/turn_system.go` 等で組まれているため、部分的な参考実装として読む程度に留め、依存追加は慎重に判断する。

### 6. flyingmutant/rapid

- リポジトリ: https://github.com/flyingmutant/rapid
- 概要: Go 向けのプロパティベーステスト（Property-Based Testing）ライブラリ。ジェネリクスで型安全なデータ生成を行い、失敗したテストケースを自動的に最小化する。`go test` にそのまま統合される。
- ruins での用途: `internal/mapplanner`（マップ生成の到達性・接続性）や `internal/save`（serde の往復性）のような「あらゆる入力に対して不変条件が成り立つべき」領域のテスト強化に向く。現状 `testify` によるテーブルテストと `goldie` によるゴールデンテストが中心で、ランダム入力に対する不変条件検証は手薄。
- 成熟度: 852 stars、直近更新 2026-07-20 と活発。
- ライセンス: Mozilla Public License 2.0。ファイルレベルのコピーレフトで、`rapid` 自体のソースを改変せず外部ライブラリとして `go get` するだけなら通常は問題にならない。組織の OSS ライセンス方針と照らして最終確認する。
- 採用時の注意: 既存の `go-mutesting`（ミューテーションテスト）と役割が異なる。導入するならまず serde 往復性や mapplanner の接続性など不変条件が明確な箇所に絞ると効果が出しやすい。

## コメント

- 本ドキュメントは実装を伴わない OSS 調査であり、通常の設計ドキュメント雛形（要件/設計/変更箇所/採用しなかった方法）の代わりに `## 候補` セクションで各 OSS を並べる構成にしている。次回以降の調査でも同じ構成を踏襲する想定。
- GitHub Trends は無関係なジャンルのみだったため、今回は候補に採らなかった。ドメイン検索で足りることが確認できた。
- 6件のうち即戦力なのは `ebitengine/debugui`（デバッグ UI）と `flyingmutant/rapid`（テスト強化）で、他は「将来要件が顕在化したら検討する」寄りの参考候補。
- `quasilyte/pathing` と `joeycumines/go-behaviortree` は、いずれも既存の自前実装を置き換える提案であり、YAGNI の観点から今すぐの導入根拠は薄い。置き換えの実益が出た時点で再検討する形が妥当。
