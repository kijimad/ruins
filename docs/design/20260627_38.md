# マップ生成過程の可視化

## 背景

mapplannerはPlannerChainで複数フェーズを経てマップを生成するが、各フェーズでタイルがどう変化するかを確認する手段がない。生成結果の良し悪しはゲームを実際にプレイしないと分からず、アルゴリズムの調整が非効率である。

digger_rsでは `take_snapshot()` でフェーズごとのマップ状態を保存し、`MapGeneration` ステートで300msごとにアニメーション再生する仕組みがある。同等の仕組みをruinsに導入する。

### 要件

- 各PlannerChainのフェーズ完了時にタイルとエンティティ配置のスナップショットを保存する
- ゲーム内の専用ステートでスナップショットをフェーズ順にアニメーション表示する
- ランダムシードを変えてパターンを確認できる
- ゴールデンテストで各プランナータイプの生成結果をスナップショットとして検証できる

### 要件外

- 独立したコマンド (`cmd/mapvis` 等) は作らない。ゲーム内ステートとVRTで完結させる

## 設計

### 方針: ゲーム内ステート + VRTゴールデンテスト

digger_rsと同様に `MapGenVisualizerState` をゲーム内ステートとして追加する。スナップショット記録機能をPlannerChainに追加し、ステート内でスプライト描画する。さらにVRTフレームワークを使ってゴールデンテストで各フェーズの描画結果を検証する。

### スナップショットの仕組み

PlannerChain に履歴記録機能を追加する。各 MetaMapPlanner の `PlanMeta()` 実行後にタイル配列とエンティティ配置のコピーを保存する。

```go
type Snapshot struct {
    Label         string                  // フェーズ名。reflect.TypeOfで自動取得する
    Tiles         []oapi.Tile             // タイル配列のコピー
    Rooms         []gc.Rect               // その時点での部屋情報
    Corridors     [][]gc.TileIdx          // 廊下情報
    NPCs          []NPCSpec               // 配置予定のNPC
    Items         []ItemSpec              // 配置予定のアイテム
    Props         []PropsSpec             // 配置予定のProps
    Doors         []DoorSpec              // 配置予定のドア
    NextPortals   []consts.Coord[int]     // 次階ポータル
    EscapePortals []consts.Coord[int]     // 脱出ポータル
    SpawnPoints   []maptemplate.SpawnPoint // スポーン地点
}

type PlannerChain struct {
    // 既存フィールド
    Starter    *InitialMapPlanner
    Planners   []MetaMapPlanner
    PlanData   MetaPlan
    // 追加
    Snapshots  []Snapshot // フェーズごとのスナップショット
    Recording  bool       // スナップショット記録を有効にするか
}
```

`PlannerChain.Plan()` 内で各プランナー実行後に自動記録する:

```go
func (c *PlannerChain) Plan() error {
    if err := c.Starter.PlanInitial(&c.PlanData); err != nil {
        return err
    }
    c.takeSnapshot("Initial")

    for _, p := range c.Planners {
        if err := p.PlanMeta(&c.PlanData); err != nil {
            return err
        }
        // ラベルはreflect.TypeOf(p).Elem().Name()で自動取得する
        c.takeSnapshot(reflect.TypeOf(p).Elem().Name())
    }
    return nil
}

func (c *PlannerChain) takeSnapshot(label string) {
    if !c.Recording {
        return
    }
    d := &c.PlanData
    c.Snapshots = append(c.Snapshots, Snapshot{
        Label:         label,
        Tiles:         slices.Clone(d.Tiles),
        Rooms:         slices.Clone(d.Rooms),
        Corridors:     slices.Clone(d.Corridors),
        NPCs:          slices.Clone(d.NPCs),
        Items:         slices.Clone(d.Items),
        Props:         slices.Clone(d.Props),
        Doors:         slices.Clone(d.Doors),
        NextPortals:   slices.Clone(d.NextPortals),
        EscapePortals: slices.Clone(d.EscapePortals),
        SpawnPoints:   slices.Clone(d.SpawnPoints),
    })
}
```

`Recording` がfalse（デフォルト）の場合、スナップショットは記録されず既存の動作に影響しない。

### MapGenVisualizerState

ゲーム内のデバッグ用ステートとして `MapGenVisualizerState` を追加する。デバッグモード時の新フロア進入時に、このステートでマップ生成アニメーションを表示してからダンジョンモードへ遷移する。

```go
type MapGenVisualizerState struct {
    es.BaseState[w.World]
    snapshots   []mapplanner.Snapshot
    currentIdx  int
    playing     bool
    timer       float64
    PlannerType mapplanner.PlannerType // ゴールデンテストで外部から指定する
    Seed        uint64                 // ゴールデンテストで外部から指定する
    PhaseIndex  int                    // ゴールデンテストで特定フェーズを表示する。ゲーム内では0から開始
    mapWidth    consts.Tile
    mapHeight   consts.Tile
}
```

#### 描画

- 既存の RenderSystem を使ってスプライト描画する。ゲーム本体と同じ描画パイプラインを再利用する
- NPC、アイテム、ドア、ポータル、スポーン地点もスプライトで重ねて表示する。フェーズが進むにつれて配置されていく様子が見える
- 現在のフェーズ名とインデックスをHUDで表示する
- カメラはマップ全体が画面に収まるようにスケールを自動調整する

#### 操作

- スペースキー: 再生/一時停止
- 左右矢印: 前後のフェーズに移動
- R: 同じシードで再生成
- N: 新しいランダムシードで再生成
- 1-5: プランナータイプ切り替え（小部屋/大部屋/洞窟/廃墟/森）
- Escape: ステートを閉じる
- 自動再生時は500msごとに次のフェーズへ遷移

### ゴールデンテスト

VRTフレームワークの `AssertStateGolden` を使い、各プランナータイプの各フェーズをゴールデン画像として検証する。固定シードで再現性を確保する。フェーズごとに個別のPNGファイルとして保存することで、どのフェーズで変化が起きたかを特定しやすくする。画像ビューワーでファイルを切り替えれば生成過程を確認できる。

テーブル駆動テストでスナップショットの長さ分ループし、各フェーズを個別のゴールデン画像として検証する。フェーズの追加・削除時にテスト関数の手動修正が不要になる。

```go
func TestGolden_MapGen(t *testing.T) {
    t.Parallel()
    types := []mapplanner.PlannerType{
        mapplanner.PlannerTypeSmallRoom,
        mapplanner.PlannerTypeLargeRoom,
        // ...
    }
    for _, pt := range types {
        chain := buildChain(pt, 12345)
        chain.Recording = true
        chain.Plan()
        for i, snap := range chain.Snapshots {
            t.Run(fmt.Sprintf("%s/Phase%d_%s", pt, i, snap.Label), func(t *testing.T) {
                t.Parallel()
                vrt.AssertStateGolden(t, vrt.States(&gs.MapGenVisualizerState{
                    PlannerType: pt,
                    Seed:        12345,
                    PhaseIndex:  i,
                }))
            })
        }
    }
}
```

1枚あたり数KBなので全フェーズ保存してもサイズは問題ない。マップ生成アルゴリズムを変更した際に、どのフェーズで意図しない変化が起きたかを即座に検知できる。

### 採用しなかった方法

| 方法 | 不採用の理由 |
|------|-------------|
| 独立した `cmd/mapvis` コマンド | ゲーム本体のスプライト描画パイプラインやリソースを再利用できない。二重実装になる |
| テキスト出力（ASCIIアート） | フェーズ間の差分が見づらい。アニメーションできない |
| Webブラウザベース（HTML Canvas） | HTTPサーバーの実装が必要。Ebitengineが既にあるのに二重投資 |
| 各プランナー内で手動snapshot呼び出し | 全プランナーに変更が必要。PlannerChainで自動記録するほうがシンプル |
| ギャラリーモード（複数シードのサムネイル一覧） | 有用だが初期スコープでは過剰。Nキーでの逐次確認で十分。必要になれば後で追加する |

## コメント

- digger_rsの `take_snapshot()` と同じアプローチだが、ruinsではPlannerChain側で自動記録する。プランナー側の変更が不要
- ゲーム内ステートにすることで、RenderSystemの描画パイプラインをそのまま使える
- ゴールデンテストではPlannerChainを直接構築してリトライを迂回し、固定シードの再現性を確保する
- ゴールデンテストによりマップ生成アルゴリズムの変更時にビジュアルリグレッションを検知できる
- plannerName はreflect.TypeOf(p).Elem().Name()で取得する。MetaMapPlannerインターフェースへのメソッド追加は不要
- セルラーオートマトン（洞窟）のように反復ごとに状態が変わるプランナーは、反復ごとにスナップショットを取りたい場合がある。その場合はプランナー内から `PlannerChain.takeSnapshot()` を呼べるように、MetaPlanにスナップショット記録用のコールバックを持たせることも将来検討する
