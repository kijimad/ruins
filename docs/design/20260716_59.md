# ゲームパフォーマンステスト戦略

## 背景

`docs/design/20260715_58.md` §7-§8 で AI 処理コストのカリング・空間インデックス増分更新を実装し、`BenchmarkProcessAll` で数値化した。その過程で以下が判明した。

- 壁時計ベンチだけでは「デグレを CI で確実に止める」用途に弱い（共有ランナーのノイズでフレーキー）。
- CPU プロファイルで初めて判明したコスト（インデックス再構築チャーン）があり、**推測でなく計測**が要る。
- 割り当て/GC がフレーム安定性の隠れた主役（§8 でメモリ 1/15 → GC ポーズ低減）。

ゲームの「滑らかさ（FPS 安定性）」を継続的に守るため、パフォーマンステストの体系を定める。

## このゲーム固有の前提

- **ターン制＋描画補間なし**：通常プレイ中の入力待ち時間は何も動かないため FPS は自明に問題ない。フレーム時間が問題化するのは **①走り（自動で毎秒多数のターンサイクル）** と **②密集シーン（ホード）** の2つに集約される。計測はこの2シーンに絞ると少ない fixture で実効カバレッジを得られる。
- **Ebiten はヘッドレスで Draw（GPU）が測りにくい**が、ボトルネックは Draw でなく `Update`（システム群のロジック）側。`Update` は xvfb で測れる。
- 既存 CI は `make` ベースで、`test`/`lint` を PR で回し、`octocov` がカバレッジを PR コメントする。ベンチの CI 実行は現状なし。

## 要件

- **デグレを確実に止める**：性能に効く不変条件が壊れたら CI で PR をブロックする。壁時計に依存しない決定的な判定であること。
- **傾向を数値で追える**：実時間・割り当てを継続的に計測し、大きな劣化を検知できる。
- **原因究明できる**：デグレ検知時に「どこが」「なぜ」を辿れる。
- **運用が破綻しない**：共有ランナーのノイズで CI が不安定に落ちないこと。

## 設計

パフォーマンステストを4層に分ける。**「ブロックは決定的な proxy、追跡は壁時計」**を鉄則とする。

### L1: 決定的ガード（`make test` 内・CI ブロック・非フレーキー）

壁時計ではなくアルゴリズム的性質を固定する。主防衛線。

- **処理数の上限**（実装済み `TestCullDistantSolo_ScalingInvariant`）：大マップで processed が総数を大きく下回る。カリング無効化を検知。
- **インデックス再構築回数の上限**（実装済み `TestSpatialIndex_移動で再構築チャーンが起きない`）：`SpatialIndex.BuildCount`（観測用累積カウンタ）を使い、複数回移動しても再構築が増えないことを固定。§8 のチャーン再発を検知。旧実装なら移動数だけ増える弁別テスト。
- **AIフェーズで再構築しない**（実装済み `TestProcessAll_AIフェーズで空間インデックスを再構築しない`）：上記はプレイヤー移動のみを見るが、§8 の**実際のホット文脈は AIフェーズ**（多数の敵が毎ターン移動）。40体×数ターンの `ProcessAll` 後も `BuildCount` 増分がターン数以下＝再構築が移動数に比例しないことを固定。旧実装なら移動数（数十〜百）に比例して増える。
- **ストレス無 panic**（実装済み `TestProcessAll_大規模でpanicしない`）：500体規模で数ターン `ProcessAll` を回して panic・エラーしないこと。今回の攻撃 panic（入れ子 ProcessTurn）のような大規模時のみ顕在化する回帰を捕捉。

**割り当て（allocs）は L1 ハードゲートにしない**：①`make test` は `-race` で走り race は allocs 数を変えるため絶対閾値が壊れる、②カリングは全 solo を targets に収集してから除外するので収集スライスが敵総数に比例して伸び、処理数0でも allocs は総数依存＝カリング効果の clean な proxy にならない。よって allocs は L2 の `-benchmem` 追跡（`BenchmarkProcessAll` が既に allocs/op を報告）に委ねる。

### L2: ベンチマーク（`make bench`・非ブロック追跡）

- **マイクロ**：`BenchmarkProcessAll`（実装済み）に加え、FOV（`VisionSystem`）・経路探索を個別に。`-benchmem` で allocs も測る。
  - **`BenchmarkRestoreAllActionPoints`（実装済み）**：毎ターン終了時に全 TurnBased エンティティへ AP 回復（カリング対象外）するコストを計測。**計測結果：100体 0.0027ms / 400体 0.0099ms / 1000体 0.024ms**。O(N) だが絶対値が極小（1000体でも予算の 0.15%）で、走行中も含め**最適化不要**と判明（「走行中は restore が支配的では」という仮説を測定で否定）。ベンチは将来 `CalculateSpeed` 等が重くなった際の回帰ガードとして残す。
- **フレーム統合ベンチ**：`DungeonState.Update` のシステム群を1フレーム分まとめて実行し ns/op を測る（Draw 抜き・xvfb 必要）。16.67ms 予算との比較が「実 FPS」に最も近い。
- **標準シナリオ（fixture）**：`{小/中/大マップ}×{疎/密}×{敵数}` を共有ヘルパ化。固定 seed・固定配置で再現性を持たせる。
- **CI 比較（実装済み）**：PR のベースブランチと現在のベンチを `benchstat` で比較し、結果を **sticky コメント**でPRに表示、**sec/op が 2倍以上悪化したら失敗**させる（`.github/workflows/check.yml` の `bench-compare` ジョブ、`scripts/bench-compare.sh`）。gh-pages 履歴は不要でベース vs PR を都度比較する。
  - 「壁時計を CI のブロックにしない」原則（下記）と矛盾しないよう、**しきい値は 2倍（+100%）と粗く**する。共有ランナーのノイズは 2倍には届かないため、ノイズで落ちず破壊的悪化のみ捕捉できる。緩やかな悪化は sticky コメントの数値で人が気づく（門番はしない）。
  - 門番は自前で old/new の平均 sec/op 比を計算する（benchstat の「vs base」列は低サンプルだと `~` になり判定に使えないため）。benchstat 出力はコメント表示専用。

### L2 フレーム統合ベンチの重要発見（カバレッジの実態）

`BenchmarkDungeonFrame`（実ダンジョン、~2500タイル、Update/Draw 分離、Ryzen 9 9900X）：

| ビルダー | Update | Draw | Draw/予算 |
|---|---|---|---|
| SmallRoom | 0.0065 ms | 1.76 ms | ~11% |
| BigRoom | 0.0055 ms | 2.93 ms | ~18% |
| Cave | 0.0062 ms | 3.08 ms | ~18% |

- **Draw が Update の 270〜560倍**。steady-state（通常プレイ・入力待ち）では TurnSystem は Player フェーズで待機し AI フェーズは走らないため Update は極小。**通常プレイの1フレームはほぼ全部 Draw**。
- 本ドキュメント §7-§9 の最適化はすべて **Update 側（AIフェーズ）**。したがって**通常プレイ FPS に対するカバー率は約0.2〜0.4%**。走行・戦闘の Update スパイクはカバーしたが、**定常 FPS の主コスト（Draw）は未計測・未ガードだった**。
- Draw のホットパス（`render_sprite.go`）は毎フレーム**全スプライト（床/壁タイル含む）を集めて `sort.Slice` して描画**する。~2500タイルで 1.8〜3.1ms＝予算の 1〜2割。シームレスワールド方針で**マップが大きくなると予算を食い潰す**恐れが高い（§9 の `FindNearestEntity` と同型の「全タイル走査」問題が描画側にある）。
- **次の最適化ターゲットは Draw の画面外カリング**（可視範囲のスプライトだけを sort/描画する）。B2 と同型の構造なので効果が見込める。golden テストは見た目の正しさは守るが Draw 時間は守らないため、Draw ベンチ（`BenchmarkDungeonFrame`）が回帰ガードになる。

**対応（実装済み・床レイヤーの viewport カリング）**：Draw の CPU プロファイルで `renderFloorLayer` が Draw の 62%（GPU 描画コール `DrawImage`）と判明。視界半径24タイル > 画面（~30×22）のため可視/記憶タイルの多くが**画面外なのに描画**されていた。`viewportTileBounds`＋`inViewport` で床タイルを画面矩形（+2タイル余白）にカリングし、ソート・描画対象から外す。

- 計測：SmallRoom/Draw 1.76→1.41ms、Cave/Draw 3.08→2.89ms。**小マップ（~2500タイル）では 6〜20% だが、効果はマップサイズに比例**。視界/記憶が viewport を大きく超える大マップ（シームレスワールド方針）で本領を発揮する。
- **golden 不変**（画面外カリングは絵を変えない）で正しさを担保。`BenchmarkDungeonFrame` が Draw の回帰ガード。
**対応（実装済み・オブジェクト/影レイヤーと影 tileMap）**：`renderObjectLayer`・`renderShadows`（物体影・壁影）も viewport カリング。特に `renderShadows` の下タイル参照用 `tileMap` 構築を全タイル → viewport 内のみに変更し、大マップで毎フレーム O(全タイル) の map 化を回避。

- 計測（床＋全レイヤー累積）：SmallRoom/Draw **1.76→1.25ms（-29%）**、Cave/Draw 3.08→2.84ms。golden 不変。
- これで Draw の主要3レイヤー（床・オブジェクト・影）と暗闇がすべて viewport 限定になり、大マップでも描画コストが画面サイズにほぼ比例（マップ全体サイズに非依存）になる。

### L3: 実フレーム計測ハーネス（最高忠実度・手動/夜間）

L1/L2 は `Update` のみ。**本物の frame time（Draw 込み・GC 込み）**を測るには実ゲームを走らせる。

- **シナリオ再生モード**：`make run` にフラグを足し「大マップに敵 N 体スポーン→自動で走り続ける」を無人実行。
- Ebiten の `ActualFPS()`/`ActualTPS()` と `Update`/`Draw` の手動計測で**フレーム時間の分布（p50/p95/p99）**を記録し JSON 出力。
- **平均でなく tail（p99）を見る**：体感の引っかかりは最悪フレームで決まる。
- リリース前や夜間 CI（xvfb）で実行。

### L4: プロファイル（原因究明・オンデマンド）

- `make profile`（CPU/mem pprof）を用意。デグレ時に L1/L2 が「どこ」を、L4 が「なぜ」を答える。§8 のチャーン発見と同じ流れ。

### 構築順（費用対効果）

1. **L1 拡充**（allocs 上限・再構築回数・ストレス無 panic）← 本ドキュメントで着手。最小労力で最大の回帰防御。
2. L2 フレーム統合ベンチを1本追加。
3. L4 `make profile` ターゲット。
4. L2 CI 追跡（要承認）／L3 実機ハーネス（必要時）。

## 採用しなかった方法

- **壁時計ベンチを CI の細かいブロック条件にする** ── 共有ランナーの CPU 変動でフレーキー。決定的 proxy（processed 数・再構築回数）を主ブロックに置く。ただし**「破壊的悪化（2倍）」だけは壁時計でもノイズに埋もれない**ため、bench-compare で 2倍ゲートとして採用した（緩やかな悪化はゲートせず sticky コメントで可視化）。
- **平均フレーム時間だけで評価する** ── 体感 jank は tail（p95/p99）が支配する。分布で見る。
- **全システムをヘッドレスでフル計測（Draw 含む）を CI 常時実行** ── GPU 依存で不安定・重い。Draw 込みの計測は L3 の手動/夜間に限定し、CI は Update 側の決定的ガードに集中。
- **全ベンチを毎 PR 実行** ── 実行時間が膨らむ。マイクロは `make bench` の手動、CI 追跡は対象を絞る（例 `BenchmarkProcessAll`）。

## コメント

- L1 の allocs 上限・再構築回数上限は「現状値＋ヘッドルーム」で設定し、改善したら閾値も締める運用にする（デグレ検知と過敏さのバランス）。
- 再構築回数カウンタはテスト専用の観測用。プロダクションのホットパスに分岐を増やさないよう、ビルド関数呼び出し回数を数える最小実装にとどめる。
- L3 の実機ハーネスは忠実度が高いが構築コストも高い。実機の jank を疑う具体的な状況が出てから着手する。
