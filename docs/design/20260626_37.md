# 設計ドキュメント駆動の自律実装ワークフロー

## 背景

機能の実装には設計→実装→ビルド・テスト→レビュー→修正という一連のサイクルが必要だが、これを手動で逐一指示するのは非効率である。設計ドキュメントを入力として渡せば、Claude Codeが自律的に実装・検証・修正ループを回し続けるワークフローが求められている。

仕事中など不在の間にトークン限界まで実装を進め、復帰時に最小の操作で再開できることが重要である。

### 要件

- 設計ドキュメント（`docs/design/*.md`）を入力として、自律的に実装→`make check`→修正ループを回す
- 設計ドキュメント内の進捗セクションにタスク完了を逐次書き込み、トークン切れで突然終了しても復元できる
- 次のセッションで、進捗セクションと`git diff`から状態を把握して実装を再開できる
- 各タスクの完了・未完了が明確に追跡できる
- 実装中に人間の介入を必要としない

### 要件外

- 設計ドキュメントの自動生成は含まない。設計は人間が書く

### ブランチ戦略

- 1設計ファイルにつき1ブランチ、1PRとする
- タスク完了（`make check`通過）ごとにcommitする
- pushはしない。人間が復帰時にレビューしてからpushする
- 設計ドキュメントの範囲外のファイル変更は許容する。pushしないので人間がレビューで確認する

### 自動再開

- 手動で起動するスクリプトがセッション終了→再起動ループを回す
- 全タスク完了またはブロッカー発生時はスクリプトが自動停止する
- Ctrl+Cやマシンのシャットダウンでも停止する

## 設計

### ワークフロー全体像

```
人間: 設計ドキュメントを渡して「実装して」と指示
  ↓
Phase 1: タスク分解
  - 設計ドキュメントの「変更箇所」をTodoWriteでタスクに分解する
  - 依存関係を考慮して実装順序を決定する
  ↓
Phase 2: 実装ループ（自律）
  - タスクを1つ選択
  - 実装する
  - make check を実行
  - 失敗したら修正する（成功するまでループ）
  - タスクを完了にマークする
  - 次のタスクへ（全タスク完了またはトークン切れまで）
  ↓
Phase 3: タスク完了ごとの永続化
  - 設計ドキュメントの進捗セクションを更新する
  - make check が通ったらcommitする
  - トークン切れで突然終了しても、commit済みの進捗から復元できる
  ↓
トークン切れでセッション終了
  ↓
Phase 4: 自動再開
  - スクリプトがセッション終了を検知し、状態ファイルを確認する
  - "pending"なら新セッションを起動する（進捗セクションとgit diffから状態を把握して続行）
  - "done"または"blocked"ならスクリプトも終了する
```

### 進捗管理

設計ドキュメントの「変更箇所」テーブルの各行をタスクとする。粒度が粗い場合はサブタスクに分解する。

進捗は設計ドキュメント内に進捗セクションを設けて、サブタスク含めて逐次書き込む。トークン切れで突然終了しても、最後にcommitされた進捗セクションから復元できる。

設計ドキュメント内の進捗セクションのフォーマット:

```markdown
## 進捗

- [x] タスク1
  - [x] サブタスク1-1
  - [x] サブタスク1-2
- [x] タスク2
  - [x] サブタスク2-1
  - [x] サブタスク2-2
- ブロッカー: {あれば記載}
```

次のセッション開始時は、進捗セクションの状態と`git diff`でuncommittedな変更を確認してから続行する。

### make check の扱い

- 各タスク完了時に `make check` を実行する
- 失敗した場合はエラー内容を読み取り、自律的に修正する
- 同じ系統のエラーで `make check` が5回失敗した場合は、ブロッカーとして進捗セクションに記録して停止する

### 自動再開の仕組み

セッションの起動→終了検知→再起動をループするスクリプト。手動で起動し、Ctrl+Cやシャットダウンで停止する。

状態ファイル `.claude-task-state`:

```
pending   → 未完了タスクあり。次の設計ドキュメントを探してセッションを起動する
running   → セッション実行中
done      → 現在のドキュメント完了。次のドキュメントを探す
blocked   → ブロッカー発生。スクリプトが終了する
```

スクリプト `scripts/dev-loop.sh`:

`docs/design/*.md` を自動スキャンし、進捗セクションに `- [ ]`（未完了タスク）を含むファイルを順に処理する。全ドキュメント完了で終了、ブロッカー発生時のみ停止する。

```bash
#!/bin/bash
set -eu

PROJECT_DIR="$(cd "$(dirname "$0")/.." && pwd)"
STATE_FILE="$PROJECT_DIR/.claude-task-state"
DESIGN_DIR="$PROJECT_DIR/docs/design"

# 未完了タスクがある設計ドキュメントを探す
find_next_design_doc() {
  for doc in "$DESIGN_DIR"/*.md; do
    [ -f "$doc" ] || continue
    if grep -q '^\- \[ \]' "$doc"; then
      echo "$doc"
      return
    fi
  done
  echo ""
}

# 起動時は常にpendingにリセットする。blocked/doneは前回セッションの終了状態であり、
# スクリプトを再起動する時点で人間が状況を確認済みと判断する
echo "pending" > "$STATE_FILE"

while true; do
  STATE=$(cat "$STATE_FILE")

  case "$STATE" in
    pending)
      DESIGN_DOC=$(find_next_design_doc)
      if [ -z "$DESIGN_DOC" ]; then
        echo "未完了の設計ドキュメントなし"
        exit 0
      fi

      REL_PATH="${DESIGN_DOC#"$PROJECT_DIR"/}"
      TASK=$(grep -m1 '^\- \[ \]' "$DESIGN_DOC" | sed 's/^- \[ \] //')
      echo "実装中: $REL_PATH ($TASK)"
      PROMPT="設計ドキュメント ${REL_PATH} の進捗セクションとgit diffを確認し、未完了の実装タスクを続けて。タスク完了ごとにmake checkを実行し、通ったらcommitすること。同じ系統のエラーで5回失敗したらブロッカーとして進捗セクションに記録し、状態ファイルをblockedにして停止すること。全タスク完了時は状態ファイルをdoneにすること。"

      echo "running" > "$STATE_FILE"
      cd "$PROJECT_DIR"
      claude -p "$PROMPT" --dangerously-skip-permissions || true

      # Claudeが状態を更新せず終了した場合はpendingに戻す
      [ "$(cat "$STATE_FILE")" = "running" ] && echo "pending" > "$STATE_FILE"
      ;;
    done)
      # 現在のドキュメントが完了。次のドキュメントを探す
      echo "pending" > "$STATE_FILE"
      ;;
    blocked)
      echo "ブロッカー発生。設計ドキュメントの進捗セクションを確認してください"
      exit 1
      ;;
    running)
      # 前回セッションがクラッシュした可能性がある
      echo "前回セッションが異常終了。再開する"
      echo "pending" > "$STATE_FILE"
      ;;
  esac

  sleep 60
done
```

起動:

```bash
./scripts/dev-loop.sh
```

### 変更箇所

| ファイル | 内容 |
|---------|------|
| `scripts/dev-loop.sh` | 自律実装ループを駆動するスクリプト |
| `.gitignore` | 状態ファイル `.claude-task-state` を追跡対象外にする |
| `docs/design/20260626_37.md` | 本設計ドキュメント |

### 採用しなかった方法

| 方法 | 不採用の理由 |
|------|-------------|
| `/loop`のみ | セッションがトークン切れで終了すると`/loop`も止まるため、セッションをまたぐ再開ができない |
| cronによるポーリング | セッション終了から最大1時間の空白が生まれる。制御が間接的で停止しにくい |
| `/make-plan` → `/do` スキル連携 | 汎用的すぎて、このプロジェクト固有の設計ドキュメント形式や`make check`フローに合わない |
| 設計ドキュメントなしの自律実装 | 設計なしで実装を始めると方向性がブレる。設計は人間の責務として分離する |

## コメント

- 設計ドキュメントの「変更箇所」テーブルが明確であるほど、タスク分解の精度が上がる
- 中断・再開を前提とするため、各タスクは独立性が高いほうが望ましい
- このワークフロー自体はコードの変更を伴わない。Claude Codeの運用手順の定義である
- `--dangerously-skip-permissions` を使うため、信頼できるローカル環境でのみ実行する。pushはしないので、意図しない変更は人間がレビューで防ぐ
- セッション終了後60秒の待機を入れて、レート制限に引っかかるのを防ぐ
- 設計ドキュメントの変更箇所テーブルが粗い場合、Claudeがサブタスクに分解して進捗セクションに書き出す
