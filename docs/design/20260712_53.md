# コンポーネント定義の単一登録表化とコード生成

## 背景

現状、1つのコンポーネント型を追加・変更すると、以下の **4箇所**を手で同期する必要がある。

1. `EntitySpec` 構造体のフィールド（`X *X`）— 生成用の設計図
2. `Components` 構造体のフィールド（`X *ecs.Map[X]`）— ストレージハンドル
3. `InitializeComponents` の登録行（`c.X = ecs.NewMap[X](world)`）
4. `AddEntity` の付与行（`addComp(c.X, entity, spec.X)`）

これらは同じコンポーネント型集合をミラーしており、いずれか1箇所の更新漏れが実行時バグ（コンポーネントが保存・生成されない等）を生む。`EntitySpec` と `Components` は役割が異なり（設計図 vs アクセサ）両方必要だが、**型集合のミラーは機械的な重複**である。

`TestAddEntity_AllFields`（reflection で全フィールドの網羅を検証）で `AddEntity` の漏れは検出できるようになったが、残る3箇所の同期は依然として手作業であり、コンポーネント追加のたびに定型作業とレビュー負荷が発生する。

このミラーを**単一の登録表（source of truth）から生成**することで、重複と同期漏れを構造的に排除する。

## 要件

- コンポーネント型の追加・変更を **1箇所の登録**で完結させる（型定義そのものを除く）。
- 生成対象は `EntitySpec` / `Components` / `InitializeComponents` / `AddEntity`（+ interface 振り分けの `addLocation`/`addFaction`）。
- singleton専用コンポーネント（`Components` のみ・`EntitySpec` 非対象）を表現できる。
- interface フィールド（`FactionType`/`LocationType`）を表現できる（1つの EntitySpec フィールド ↔ 複数の具体型 Map）。
- 型安全を維持する（生成コードは `any`/reflection を使わない。現状の手書きと同等）。
- 生成物が最新か CI で検証できる（ドリフト検出）。
- 生成コードはリポジトリにコミットし、可読・grep 可能に保つ。

## 設計

### 全体像

```
components_registry.go   … 単一の登録表（手書き・source of truth）
        │  go:generate
        ▼
internal/components/genspec/registry.go … 登録表(データ・package genspec)
internal/cmd/gencomponents.go … 生成器(ruins gencomponents サブコマンド)
        │
        ▼
components_gen.go        … 生成物: EntitySpec / Components / InitializeComponents / AddEntity
components.go            … 手書きのまま: 各コンポーネントの型定義（type X struct{...}）と非生成ロジック
```

コンポーネントの **型定義そのもの**（`type HP struct{ Max, Current int }` 等）は実データ・ロジックを持つため手書きのまま残す。生成するのは4種類の**定型プラミングのみ**。

### 登録表の形式

Go の宣言的スライスとして持つ（型検査が効き、外部フォーマット不要）。ジェネレータは同じ登録表を import して実行する。

```go
// components_registry.go
type componentDef struct {
    Type      string // コンポーネント型名（フィールド名の既定値）
    Field     string // フィールド名（Typeと異なる場合のみ指定）
    Singleton bool   // trueなら Components のみ生成し EntitySpec/AddEntity から除外
    Marker    bool   // 参考情報（ゼロサイズマーカー。生成には影響しない任意メタ）
}

var componentRegistry = []componentDef{
    {Type: "Name"}, {Type: "Description"}, {Type: "HP"}, /* … 直接コンポーネント … */
    {Type: "VisualEffects", Field: "VisualEffect"},          // 型名≠フィールド名
    {Type: "Dungeon", Field: "DungeonState", Singleton: true},
    {Type: "GameProgress", Singleton: true},
    {Type: "TurnState", Singleton: true},
    {Type: "SpatialIndex", Singleton: true},
    {Type: "GameLog"},                                       // singletonだがEntitySpecにも出す（現状踏襲）
    // …
}

// interfaceグループ: EntitySpecはinterfaceフィールド1つ、Componentsは具体型ごとにMap
type interfaceGroup struct {
    SpecField string      // EntitySpec 上の interface フィールド名（例: FactionType）
    Concretes []concrete  // 具体型と対応する Components フィールド
}
type concrete struct{ Type, Field string }

var interfaceGroups = []interfaceGroup{
    {SpecField: "FactionType", Concretes: []concrete{
        {"FactionAllyData", "FactionAlly"},
        {"FactionEnemyData", "FactionEnemy"},
        {"FactionNeutralData", "FactionNeutral"},
    }},
    {SpecField: "LocationType", Concretes: []concrete{
        {"LocationInBackpack", "LocationInBackpack"},
        {"LocationEquipped", "LocationEquipped"},
        {"LocationOnField", "LocationOnField"},
        {"LocationInStorage", "LocationInStorage"},
    }},
}
```

コンポーネント追加時は「型定義（components.go）」と「登録表に1行」の**2箇所**で完結する（従来の4箇所から削減）。登録表は宣言的でレビューしやすい。

### 生成物（components_gen.go）

登録表から `text/template` で以下を出力する。

- **Components 構造体**: 全 `componentDef` + 全 interfaceGroup.Concretes を `Field *ecs.Map[Type]` に。
- **InitializeComponents**: 各 Map を `c.Field = ecs.NewMap[Type](world)`。
- **EntitySpec 構造体**: `Singleton==false` の直接コンポーネントを `Field *Type`、interfaceGroup を `SpecField *SpecFieldType`。
- **AddEntity**: 直接コンポーネントを `addComp(c.Field, entity, spec.Field)`、interfaceGroup を `c.addSpecField(entity, spec.SpecField)`。
- **addLocation / addFaction**: interfaceGroup ごとに具体型 type-switch を生成。

`addComp` ヘルパ・各コンポーネント型定義・`Upsert` 等の非生成ロジックは手書き側（components.go）に残す。

### ドリフト検出

- `Makefile` に `generate` ターゲット（`go generate ./...`）を追加。
- CI で「生成 → `git diff --exit-code`」を実行し、登録表と生成物の不一致を失敗させる。
- 生成物ヘッダに `// Code generated by "ruins gencomponents". DO NOT EDIT.` を付与し、`.golangci.yml` の生成コード扱いと整合させる。

### 完全性の担保

生成に移行後も `TestAddEntity_AllFields` を維持する。生成ロジックのバグ（テンプレート誤り）に対する独立した検証層として機能する。加えて、登録表と `EntitySpec`/`Components` の実フィールドが一致することを reflection で検証するテストを追加する。

## 採用しなかった方法

- **型定義から go/ast で自動抽出（登録表なし）**。`type X struct` を走査してコンポーネントを判別する案。真の単一ソースになるが、「コンポーネント型」と「ヘルパ型」の判別にマーカーや命名規約が必要で侵襲的。singleton/interface/フィールド名差のメタも別途注釈が要り、結局明示的な情報が必要。明示的な登録表の方が単純で堅牢。
- **`[]any` のコンポーネントリストで生成を回避**。EntitySpec を廃し `[]any{&Name{}, &HP{}}` を渡す案。今回撲滅した `any`/reflection/type-switch が復活し、型安全と archetype 適性を失う。却下。
- **現状維持（手書き4箇所 + 完全性テスト）**。追加コスト（ジェネレータ・ビルド段・ドリフト検証）を負わない。コンポーネント追加頻度が低ければ ROI は限定的で、これも合理的な選択肢。本移行は「重複を構造的に排除したい」場合の投資判断。
- **struct タグによる生成指示**。`Components` のフィールドにタグを付けて他を生成する案。生成対象の `Components` 自体がタグ付けの担い手になり循環的。登録表を独立させる方が明快。

## コメント

- `EntitySpec` と `Components` の**役割の違い**（設計図 vs アクセサ）は本移行後も維持される。生成が消すのは「型集合のミラーという機械的重複」だけで、両構造体の存在意義（`raw` データ層と ECS runtime の分離境界）はそのまま。
- 削減は「4箇所 → 2箇所（型定義 + 登録表1行）」であり、**完全なゼロ手作業にはならない**。登録表は依然手書きだが、宣言的・一元的で、生成とドリフト検証が同期を保証する。
- 投資判断: コンポーネント追加が頻繁でドリフトが実害化しているなら移行の価値が高い。現状は `TestAddEntity_AllFields` 等で最悪ケース（実行時バグ）は防げているため、**緊急性は高くない**。本ドキュメントは移行の設計を確定し、着手可否を判断するための土台。
- 段階移行が可能: まず生成物が現行の手書きと**バイト等価**になるようジェネレータを作り込み、テスト green を確認してから手書きを削除する（生成物 diff = 0 を移行の受け入れ条件にする）。

### 設計時からの変更点

- **interface 対応が不要になった**。設計時は `FactionType`/`LocationType` interface の振り分け（`addLocation`/`addFaction`）が必要だったが、その後の平坦化（別マーカーコンポーネント化）で interface が撤去されたため、登録表は「1コンポーネント=1行」のフラットな表に単純化できた。
- **生成器を `cmd` サブコマンド化し、go generate をルート main.go に集約した**。既存の genreadme と同じ流儀に揃え、`//go:generate go run . gencomponents` / `go run . genreadme` をルートに置いて `go generate ./...`（＝`make generate`）で一括生成する。登録表は純データパッケージ `genspec` に残す。トレードオフとして、生成器が `package cmd`（ゲーム本体を推移的に import）に載るため、生成物が壊れていると生成器がビルドできず自己修復できない。実務上は生成物をコミット済みに保ち、CI の `generate-check` で担保する。
- **ドリフト検証を CI 専用ステップではなく通常テスト**にした。生成器自身のテスト（`generate()` の出力とコミット済みファイルの比較）が `make test` に含まれるため、追加の CI 設定なしで検出できる。
- **spec/非spec の区別（`Spec` フラグ）を廃止し、EntitySpec を Components と同一集合にした**。当初は「spawn で付与する成分だけ EntitySpec に出す」ため `Spec bool` を持たせたが、AddEntity が nil を skip するため全件を EntitySpec に含めても実害がない（singleton は固定の SingletonEntity から読むため迷子コンポーネントが読み取りを壊さず、Location も MoveToX が先に clearLocation する）。フラグと生成器のフィルタを消してシンプルにした。EntitySpec に spawn 用途でないフィールドが並ぶ点は許容する。
