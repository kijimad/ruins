# 解像度は論理固定・ウィンドウは可変とする

## 背景

設計ドキュメント `20260708_47.md` でユーザー設定を永続化する器（`config.UserConfig` の `WindowWidth`/`WindowHeight` と `settings.toml`）を用意した。当初はこれを使って「解像度プリセットを選ぶ設定UI」を追加する方針で検討・実装した（本ドキュメントの初版）。

しかし実機で確認した結果、この方針は本ゲームには合わないと判明した。

- スプライトは **32x32 固定のピクセルアート**である。論理解像度を上げても1タイルあたりの情報量は増えず、単に**見える範囲（視野）が広がるだけ**になる。視野が解像度で変わるのはゲームバランス上望ましくない。
- 一方、論理解像度を固定してウィンドウだけ拡大縮小する方式では、960x720 の描画をスケールするだけで**見た目が変わらない**ため、解像度を選ぶUIに意味がない。

結論として、**論理解像度は 960x720 に固定し、ウィンドウサイズだけを可変にする**のが本ゲームに適する。ウィンドウの可変化は Ebiten の `WindowResizingModeEnabled`（`cmd/play.go` で設定済み）で既に実現されているため、解像度を切り替えるUIは不要である。

本ドキュメントは、検討の結果として解像度UIを設けない判断と、それに伴う最小限の修正を記録する。

## 要件

- 論理解像度は 960x720 固定とする（ピクセルアートのため）
- ウィンドウは可変とする（ユーザーがドラッグでリサイズできる。Ebiten が固定解像度をスケールする）
- 解像度を切り替える設定UIは設けない（意味がないため）
- 描画の基準（カメラ中心・HUD配置）が、固定論理解像度と一致していること
- グローバル設定画面（メインメニューからの「設定」）は残す。将来の設定（音量など）を追加する土台とする

## 設計

### 論理解像度は固定、ウィンドウは Ebiten のリサイズに任せる

`MainGame.Layout()` は従来どおり `consts.MinGameWidth/Height`（960x720）を返す。ウィンドウのリサイズは `cmd/play.go` の `ebiten.SetWindowResizingMode(ebiten.WindowResizingModeEnabled)` で有効化済みで、Ebiten が 960x720 の描画をウィンドウサイズへ自動スケールする。追加の実装は不要である。

`config.UserConfig.WindowWidth/Height` は、起動時の**初期ウィンドウサイズ**として `cmd/play.go` の `ebiten.SetWindowSize` に渡す用途で残す（`settings.toml` や環境変数で初期サイズを指定できる）。論理解像度とは別概念である。

### ScreenDimensions を論理解像度に固定する（潜在バグ修正）

`ScreenDimensions` はカメラ中心（`render_sprite.go` の `ScreenDimensions/2`）や HUD 配置（`widgets/hud/*`）の基準として参照される。これは**描画対象（`Layout()` が返す論理解像度 = 960x720）と一致していなければならない**。

従来は init で `cfg.User.WindowWidth/Height`（初期ウィンドウサイズ）をセットしていた。既定の 960x720 では偶然一致しているが、`settings.toml` や環境変数で初期ウィンドウサイズを 1280 などにすると、カメラ中心・HUD 位置が実際の描画面（960x720）とズレる潜在バグがある。

論理解像度は固定なので、`SetScreenDimensions` には**論理解像度（`consts.MinGameWidth/Height`）を渡す**。これによりウィンドウサイズと描画基準を分離し、初期ウィンドウサイズを変えても描画がズレないようにする。

```go
// 変更前
world.Resources.SetScreenDimensions(cfg.User.WindowWidth, cfg.User.WindowHeight)

// 変更後: 論理解像度に固定する
world.Resources.SetScreenDimensions(consts.MinGameWidth, consts.MinGameHeight)
```

### グローバル設定画面は土台として残す

メインメニューからの「設定」（`SettingsMenuState`）は残す。ただし解像度項目は設けず、現状は「戻る」のみの最小構成とする。将来の設定（音量など）は、この画面の項目として追加していく。`main_menu.go` の既存メニューと同じ `hooks.Mount` + `UseTabMenu` パターンで構築する。

## 採用しなかった方法

- **解像度プリセットUI＋論理解像度変更**: `Layout()` が選択解像度を返し、設定メニューでプリセットを切り替える方式。実装したが、32x32 ピクセルアートでは論理解像度を上げても情報量が増えず**視野が広がるだけ**で、ゲームバランスに影響する。ユーザーの用途に合わないため撤回した
- **解像度プリセットUI＋ウィンドウスケーリング（論理固定）**: 論理解像度を固定しプリセットをウィンドウサイズとする方式。ウィンドウを拡大縮小するだけで**見た目が変わらず**、それは Ebiten のウィンドウリサイズと同じことなので、専用のプリセットUIを設ける意味がない
- **ドラッグしたウィンドウサイズの永続化**: リサイズ後のサイズを `settings.toml` に保存して次回復元する案。今回の要件（可変にするだけ）には含まれず、YAGNI のため見送る。必要になれば `config.UserConfig` に保存する形で追加できる

## コメント

- ウィンドウの可変化は Ebiten の `WindowResizingModeEnabled` で既に実現されており、追加実装は不要だった。今回残す変更は `ScreenDimensions` の固定（潜在バグ修正）のみである
- `config.UserConfig.WindowWidth/Height` は「初期ウィンドウサイズ」として残す。論理解像度（`consts` 固定）とは別概念である
- グローバル設定画面は土台として残す。現状は設定項目が無く「戻る」のみだが、音量など将来の設定をこの画面に追加していく。設定値の永続化基盤（`20260708_47.md`）はそのまま利用できる。値を左右で変更する項目が必要になった時点で、その入力処理を追加する
- テストは Makefile と同様に bwrap + xvfb-run 経由で実行する。素の `go test` は実ディスプレイでウィンドウを開き、ゴールデン比較が環境差で失敗するため使わない
