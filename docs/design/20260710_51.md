# ECSエンジンの goecs から Ark への移行

## 背景

ゲームは ECS エンジンに `github.com/x-hgg-x/goecs`（以下 goecs）を使っている。goecs のコンポーネント格納は `data []interface{}`（実測確認済み）で、**全コンポーネントを interface{} にボクシング**する。このため:

- `SliceComponent.Get(entity)` が `interface{}` を返し、呼び出し側で `.(*gc.X)` の手書き型アサーションが必要。
- 直近で導入した `Component[T]` ラッパー（`docs/design/20260709_50.md`）はこの untyped なエンジンの上に被せた**緩和策**であり、根本原因（エンジンが untyped）を解消したものではない。
- ボクシングによりヒープ確保とキャッシュ効率の劣化がある。

`Component[T]` ラッパーで型安全は達成したが、より根本的な対応として、型安全（generics ネイティブ）・archetype 格納（ボクシング撤廃）・シリアライズ簡素化を備えた後継エンジンへの移行を検討した。

候補として Arche（`mlange-42/arche`）と、その後継 Ark（`mlange-42/ark`、同作者・generics-first・より高速）を比較し、**Ark（v0.8.3）+ ark-serde（v0.3.2）** を対象に PoC（スパイク）で実現性を検証した。

### スパイクの結論（`internal/archespike/`、本体非依存）

移行の二大懸念（save 層の載せ替え、systems の書き換え）がともに解消できることを実地で確認した。

1. **save 往復**: `arkserde.Serialize(world)` でワールド丸ごとJSON化、`Deserialize` で復元。**コンポーネント内の素の `ecs.Entity` フィールドも serde が自動で再マッピング**する（現状の stableID 手動再マップが不要）。決定性（同一ワールド→同一JSON）も確認。
2. **システム移植**: `camera_system` の追従ロジックを Ark の Filter/Query に移植。`Join(...).Visit(...)` → `NewFilterN(world).Query()` + `for q.Next()` は**ほぼ機械的な1:1変換**。

## 要件

- コンポーネントアクセスを generics でネイティブに型安全化する（`interface{}` ボクシングを撤廃）。
- save/load を ark-serde ベースに載せ替え、手書きの per-component 変換を撤廃する。
- 段階移行とし、各段階で `make check`（全テスト + lint）が通ること。
- セーブ形式は変わってよい（開発中でリリース済みセーブは無い）。ただし移行完了時点で往復（roundtrip）と決定性のテストを担保する。

## 設計

### 方針: 薄い互換層を挟んで段階移行する

一括置換は 116 ファイルに及び高リスクなため、`w.World` が Ark の `*ecs.World` を保持する薄い互換層を用意し、パッケージ単位で移行する。

### API 対応表（goecs → Ark）

| goecs | Ark |
|---|---|
| `manager.NewEntity()` + `AddComponent` | `ecs.NewMapN[...](world).NewEntity(&a, ...)` |
| `world.Components.X.MustGet(e)` | `mapX.Get(e)`（`ecs.NewMap1[X](world)`） |
| `world.Components.X.TryGet(e)` | `mapX.GetUnchecked` + Has 判定、または filter |
| `entity.HasComponent(X)` | `mapX.Has(e)` |
| `Join(A, B).Visit(func(e){...})` | `ecs.NewFilterN[A,B](world).Query()` + `for q.Next(){ a,b := q.Get() }` |
| マーカー（NullComponent） | ゼロサイズ struct コンポーネント |
| save: 44関数 + 49スキーマ型 | `arkserde.Serialize`/`Deserialize` |
| エンティティ参照の stableID 手動再マップ | serde が自動再マッピング |

### コンポーネントの扱い

- コンポーネントは Ark 上ではプレーンな struct（値型）として archetype に格納される。現状の `gc.X` 型は概ねそのまま使える。
- Ark はグローバルにコンポーネント型を登録するモデルのため、現状の `Components` 構造体（コンポーネントハンドルの束）は不要になり、各所で `ecs.NewMapN[...]` / `ecs.NewFilterN[...]` を使う。Filter/Map は生成コストがあるためシステム側でキャッシュする。

### 監査が必要なコンポーネント（JSON化できない要素）

serde は JSON マーシャルのため、以下は `json:"-"` 除外か非保存で対応する（現状の save でも同様に除外/特別扱いしている）。

- map を持つもの（例: `spatial_index` の `Characters map[GridElement]ecs.Entity`）。
- interface を持つもの（例: `StateChangeRequest`、`AI.Planner`）。
- 関数フィールドを持つもの（あれば）。

## 採用しなかった方法

- **`Component[T]` ラッパーのまま（現状維持）**。型安全は達成済みで低コストだが、ボクシング撤廃・save 簡素化は得られず、untyped エンジンという根本は残る。将来機能（relations 等）も無い。ゲームの規模（ターン制・最大〜1000エンティティ）では性能は当面問題にならないため、「移行しない」も合理的な選択肢。移行の主駆動は**save 簡素化と土台の健全化**であり、性能ではない。
- **Arche（`mlange-42/arche`）**。Ark の前身。API がやや冗長で、後継の Ark があるため新規移行先としては非推奨。
- **donburi**（ebiten 界隈で実績のある別 ECS）。generics で型安全寄りだが、archetype 性能と serde の完成度で Ark を上回る決め手が無い。
- **一括ビッグバン移行**。116ファイルを一度に置換するのは高リスク。互換層を挟んだ段階移行を採る。

## コメント

- 当初 save 層を「移行の最難関」と見立てていたが誤りだった。PoC の結果、save は**むしろ大幅に簡素化される**（44関数+49スキーマ型+手動再マップが消える）側で、これが Ark 移行の最も強い動機になる。
- 型安全は `Component[T]` ラッパーで既に取得済みのため、Ark 移行で得られる型安全の増分は小さい。移行を正当化するのは主に「save 簡素化 + ボクシング撤廃 + 土台の健全化」。
- 移行するか否かは**投資判断**。しない場合もラッパーで型安全は保たれる。する場合は段階移行で物量を捌く。
- スパイク（`internal/archespike/`）は本体に一切依存しないため、判断が「見送り」でも安全に破棄できる。

## 追記: serde 監査結果と最終決定

Phase 1 の監査（JSON化できない要素の洗い出し）を先行実施し、実データで以下を確定した。

- **map キー型**: `SkillID`/`ElementType`/`ModifierKey` は string のため `map[...]V` は serde OK。**struct キー（`GridElement`/`Position`）のみ不可**。該当は視界情報（`ExploredTiles`/`VisibleTiles`/`BlockPass`/`Characters`）で、いずれも**一時状態＝save 対象外**（ロード時に再構築）。
- **interface フィールド**: `AI.Planner PlannerConfig` が save 対象で唯一の実ブロッカー。ark-serde は interface を復元できない（実測: `cannot unmarshal ... into ... interface`）。`Interactable.Interactions []InteractionData` も interface スライスだが save 対象外。

結論として、whole-world serde の実ブロッカーは **`AI.Planner`（interface）だけ**に狭まった。

### 最終決定: AI.Planner を平坦化したうえで Ark へ移行する

`AI.Planner interface` を**プレーンデータの別コンポーネント**（`SoloAI` / `SquadAI` を独立コンポーネント化）に分割する。振る舞い（`ReactToHostile`/`ResetCombat`）は aiinput の system 側へ移す。これは ECS のデータ指向原則（コンポーネントは振る舞いを持たないデータ）に沿った健全化であり、副次的に serde 互換になる。平坦化は goecs 上で先行実施し、その後 Ark へ移行する。

一時状態コンポーネント（SpatialIndex・視界 map）は save 対象外として `json:"-"` 除外する。
