# UIコンポーネント単体のゴールデンテスト

## 背景

現在のVRTはステート全体のスクリーンショットのみで、個々のUIコンポーネントの見た目を検証する手段がない。`styled.NewListItemText` や `styled.NewTableRow` などのウィジェットファクトリが返すコンポーネントを、単体で描画してゴールデン画像と比較するテスト基盤を構築する。

## 現状の課題

- ステート単位のVRTでは、レイアウト変更の影響範囲が広く、どのコンポーネントが壊れたか特定しにくい
- `styled/` のファクトリ関数はロジックテストのみで、描画結果を検証していない
- UIリソース（フォント、NineSlice画像等）を必要とするテストの初期化手段がない。`testutil.InitTestWorld` はUIリソースを読み込まない

## 設計

### 描画方式

`ebiten.RunGame` を使わず、`ebitenui.UI` に直接ウィジェットを載せて描画する。ebitenui の内部テストや `internal/maingame/integration_test.go` で実証済みのパターンである。

```go
ui := &ebitenui.UI{Container: root}
screen := ebiten.NewImage(width, height)
for i := 0; i < 3; i++ {
    ui.Update()
}
ui.Draw(screen)
```

レイアウト確定のため数フレーム `Update()` を回してから `Draw()` する。

### テストヘルパー

`internal/vrt/golden.go` にヘルパー関数を追加する。

#### RenderWidget

ウィジェットを指定サイズの画像に描画して返す。

```go
func RenderWidget(root *widget.Container, width, height int) *ebiten.Image
```

#### AssertGolden

描画結果をゴールデン画像と比較する。環境変数 `UPDATE_GOLDEN=1` で画像を更新する。

```go
func AssertGolden(t *testing.T, root *widget.Container, width, height int)
```

ゴールデン画像は呼び出し元パッケージの `testdata/TestName.png` に保存する。

### UIリソースの初期化

`maingame.InitWorld` はECS・ゲームデータ含む完全初期化で重い。UIリソースのみを読み込む軽量関数を `internal/vrt/` に用意する。

```go
// SharedUIResources はテスト用UIリソースをsync.Onceで初期化して返す
func SharedUIResources(t *testing.T) resources.UIResources
```

`sync.Once` で1回だけ初期化し、複数テストで共有する。フォント・NineSlice画像・スプライトシートの読み込みのみ行い、ECSやゲームデータは不要。

### テストコード例

`internal/widgets/styled/golden_test.go` に集約する。

```go
func TestGolden_ListItemText(t *testing.T) {
    res := vrt.SharedUIResources(t)
    root := widget.NewContainer(
        widget.ContainerOpts.Layout(widget.NewRowLayout(
            widget.RowLayoutOpts.Direction(widget.DirectionVertical),
        )),
    )
    root.AddChild(styled.NewListItemText("選択中", theme.TextPrimary, true, res))
    root.AddChild(styled.NewListItemText("非選択", theme.TextPrimary, false, res))

    vrt.AssertGolden(t, root, 300, 80)
}

func TestGolden_TableRow(t *testing.T) {
    res := vrt.SharedUIResources(t)
    // テーブルコンテナを構築して描画
    container := styled.NewTableContainer([]int{100, 200, 80}, res)
    styled.NewTableHeaderRow(container, []int{100, 200, 80}, []string{"名前", "説明", "数量"}, res)
    styled.NewTableRow(container, []int{100, 200, 80},
        []string{"回復薬", "HPを回復する", "3"},
        []styled.Align{styled.AlignLeft, styled.AlignLeft, styled.AlignRight},
        true, res)

    vrt.AssertGolden(t, container, 400, 80)
}
```

### ゴールデン画像の管理

| 項目 | 方針 |
|------|------|
| 保存場所 | `internal/widgets/styled/testdata/TestGolden_ListItemText.png` 等。Go標準の `testdata` 慣習に従う |
| 更新方法 | `UPDATE_GOLDEN=1 go test ./internal/widgets/styled/` で全画像更新 |
| 画像比較 | ピクセル完全一致。決定論的な描画なので許容差は不要 |
| CI | ゴールデン画像をgit管理し、差分があればテスト失敗 |

### 対象コンポーネント

初期のテスト対象。段階的に追加する。

| コンポーネント | 関数 | テスト観点 |
|------|------|------|
| リストアイテム | `NewListItemText` | 選択/非選択の表示、セパレータ、選択バー |
| テーブル行 | `NewTableRow` | カラム幅、選択状態、テキスト配置 |
| テーブルヘッダ | `NewTableHeaderRow` | ヘッダテキストの表示 |
| メニューテキスト | `NewMenuText` | フォント・色 |
| ページインジケータ | `NewPageIndicator` | ページ番号表示 |
| ウィンドウ | `NewSmallWindow` | タイトルバー、パネル背景 |

### 不採用とした選択肢

| 選択肢 | 不採用の理由 |
|------|------|
| `ebiten.RunGame` を使うフル描画 | コンポーネント単体テストには過剰。RunGame不要で描画できる |
| SSIM等の類似度比較 | 決定論的描画のため完全一致で十分。閾値管理の複雑さを避ける |
| テストフラグ `-update` | Go testのカスタムフラグ定義は煩雑。環境変数のほうが簡潔 |
| 各パッケージにテスト分散 | 主な対象が `styled/` のファクトリ関数に集中しているため集約が効率的 |
| `maingame.InitWorld` でリソース初期化 | ECS・ゲームデータが不要なのに初期化される。UIリソースのみの軽量関数を新設する |

## 実装で判明した課題と対策

### 1. ebitenuiのグローバル状態とテスト並列化

#### 問題

ebitenuiは「UIは1アプリに1つ」を前提に設計されており、以下のグローバル状態を持つ。

| グローバル状態 | 場所 | 影響 |
|---|---|---|
| `deferredActions []DeferredAction` | `internal/event/deferred.go` | ウィジェット生成時にキュー追加。リセットAPIなし |
| `InputHandler` 単一インスタンス | `internal/input/input.go` | 全UI操作で共有 |
| `deferredRenders []RenderFunc` | `widget/widget.go` | 描画キュー |

`t.Parallel()` で複数テストが同時にウィジェットを生成すると、`event.AddDeferred()` がグローバルスライスに並行書き込みし、データ競合で `event.ExecuteDeferred()` がパニックする。

```
panic: runtime error: invalid memory address or nil pointer dereference
goroutine 68 [running]:
github.com/ebitenui/ebitenui/internal/event.ExecuteDeferred()
```

#### 試行: renderMuによるレンダリング直列化

当初、レンダリングのみを `sync.Mutex` で直列化した。

```go
func RenderWidget(root *widget.Container, width, height int) *image.NRGBA {
    renderMu.Lock()
    defer renderMu.Unlock()
    // ...
}
```

しかしウィジェット**生成**自体がグローバルイベントキューに書き込むため、mutexの範囲が不足していた。

#### 解決: ビルダー関数方式

`AssertGolden` がビルダー関数を受け取り、ウィジェット生成からレンダリングまでをmutex内で一括実行する。

```go
func AssertGolden(t *testing.T, buildFn func() *widget.Container, width, height int) {
    t.Helper()
    rendered := renderContainer(buildFn, width, height)
    // ...
}

func renderContainer(buildFn func() *widget.Container, width, height int) *image.NRGBA {
    renderMu.Lock()
    defer renderMu.Unlock()
    root := buildFn()  // ウィジェット生成もmutex内
    ui := &ebitenui.UI{Container: root}
    // ...
}
```

`renderState` はすでにmutex内でステート初期化を行っていたため、このパターンと一致する。

### 2. トレランスの自動算出

#### 問題

ebitenuiのグローバル `InputHandler` 状態がテスト間で累積し、UIエッジのピクセルに微小な差分が出る。`-shuffle=on` でテスト順序が毎回変わるため、差分量も変動する。

当初は固定トレランス（5%）を使っていたが、大きい画像（960×720）では15,000ピクセル以上の差分を見逃し、小さい画像（300×30）ではギリギリだった。

#### 試行1: 固定ピクセル数モデル

ノイズが画像サイズに依存しない固定量と仮定し、`maxNoisyPixels / totalPixels` で比率を算出した。

```go
const maxNoisyPixels = 500
ratio := float64(maxNoisyPixels) / float64(total)
```

小さい画像では適切だったが、大きいstate画像ではUIエッジが多い分ノイズも多く、500ピクセルでは不足した。

#### 解決: √面積スケーリング

ノイズはUI要素のエッジで発生し、エッジ量は面積ではなく周長（≈ √面積）に比例する。

```go
const noiseScale = 5.0

func toleranceForSize(width, height int) float64 {
    total := width * height
    if total <= 0 {
        return 0
    }
    return noiseScale / math.Sqrt(float64(total))
}
```

`assertPNGGolden` はPNGヘッダから画像サイズを読み取り、`toleranceForSize` で自動算出する。呼び出し側はトレランスを意識しない。

| 画像サイズ | ピクセル数 | トレランス | 許容ピクセル数 |
|---|---|---|---|
| 300×30 | 9,000 | 5.3% | 474 |
| 400×120 | 48,000 | 2.3% | 1,095 |
| 960×720 | 691,200 | 0.6% | 4,147 |

### 3. gamepad初期化のEINTRエラー

#### 問題

ebitenの `gamepad_linux.go` が `/dev/input/event*` を `unix.Open` する際、`EINTR`（シグナル割り込み）を受けてエラーを返す。ebitenはこのエラーをリトライせず致命的として扱うため、`ebiten.RunGame` が失敗する。xvfb環境で散発的に発生し、テストの安定性を損なっていた。

```go
// ebiten/internal/gamepad/gamepad_linux.go:109
fd, err := unix.Open(path, unix.O_RDONLY|unix.O_NONBLOCK, 0)
if err != nil {
    // EACCES, EPERM, ENOENT は無視されるが、EINTR は致命的エラーになる
    return fmt.Errorf("gamepad: Open failed: %w", err)
}
```

#### 試行1: RunTestMain内でリトライ

`ebiten.RunGame` が失敗した場合にリトライを試みたが、一度失敗するとebitenのグローバル状態（glfw等）が壊れ、同一プロセスでの再起動は不可能だった。

```
RunGame failed (retry 1/3): gamepad: Open failed: interrupted system call
RunGame failed (retry 2/3): glfw: The GLFW library is not initialized
```

#### 試行2: サブプロセス方式

テストバイナリ自身をサブプロセスとして起動し、初期化エラー時にリトライする方式を実装した。特殊終了コード（99）でebiten初期化失敗とテスト失敗を区別した。

```go
func RunTestMain(m *testing.M) int {
    if os.Getenv("VRT_SUBPROCESS") != "1" {
        return runAsParent()  // サブプロセスを起動してリトライ制御
    }
    return runGameLoop(m)    // サブプロセス内でebiten.RunGame実行
}
```

10/10で安定したが、環境変数・終了コード・プロセス管理で約40行のコードが必要で複雑だった。

#### 解決: bwrapで/dev/inputを隠す

ebitenのgamepad初期化コードは `/dev/input` が存在しない場合にスキップする（`ENOENT` → `return nil`）。bubblewrap（`bwrap`）でマウント名前空間を作り、`/dev/input` を空のtmpfsで覆い隠すことで、gamepad初期化自体を発生させない。

```makefile
bwrap --dev-bind / / --tmpfs /dev/input -- \
xvfb-run -a go test ...
```

root権限不要。サブプロセス方式のコードは全て削除し、Makefileの1行追加で解決した。10回×全37パッケージで安定パス。

## コメント

- ebitenui内部テスト（`ebitenui/widget/widget_testing_test.go`）で `ebiten.NewImage` + `Render()` のパターンが実証済み
- `maingame/integration_test.go` でも RunGame 外での `ebiten.NewImage` + `Draw()` が動作確認済み
- フォント初期化に `opentype.NewFace` が必要で、フォントファイルの読み込みパスに注意が必要。`resources.InitUIResources` の分離を検討する
- `messagewindow.Window` のテストはWorldに依存するため、コンポーネント単体テストの範囲外とする。ステートVRTで引き続きカバーする
- ebitenuiのグローバル状態はインスタンス分離が不可能なため、mutex直列化で対処する。ebitenui側の設計制約である
- ebitenの `gamepad_linux.go` は `EINTR` をリトライすべきだが、外部ライブラリのため `bwrap` で回避する。上流へのPRが正道
