# マクロ移動の廃止 ── 出口選択型ラン構造（Hades/DD型）

## 背景

55.md でマクロ移動（`MacroMapState`）を実装し、実プレイ検証しながら形を反復した（ノード網→レーン方式→CDDA風グリッド→FTL風停留点）。しかしどの形でも「ミクロとマッチしない・選択に意味がない・ミクロ（楽しい部分）を潰している」という不満が残った。

### 診断1: design.md の原則との突き合わせ（現実装の違反）

| design.md の原則 | 現実装（FTL風停留点） | 判定 |
|---|---|---|
| 「マクロは問いを立てる、ミクロが答える」──ルート選択は"どんな戦術挑戦を、どの条件で受けるか"を問う | 停留点の選択はダンジョンの中身を何も変えない。どの遺跡も同じ定義・同じ条件 | ❌ 「意味がない」の正体 |
| フィードバックループ──マクロ状態→ミクロ難度／ミクロ結果→マクロ位置 | マクロ→ミクロの矢印が存在しない。逆向きも「戦利品価値÷8=糧食」の抽象換算のみ | ❌ 往復の半分が欠落 |
| 漏れバケツ禁止──「消費は選択の結果（面数×積載）」「ルートと無関係の税は最小に」 | ジャンプごとに flat 税（糧食-4/燃料-2）。どこへ跳んでも同額 | ❌ 55.md §4 の原設計から退行 |
| 橋は頂点＝引き際──「もう一つ遺跡を襲うか、荷を確保し急ぐか」 | 潜行の前線コストは入場時に flat 先払い。中で何をしようが同じ＝ミクロの巧拙がマクロに返らない | ❌ 「潰している」の正体 |
| ミクロの2つの場所──「強化・生産はミクロ（タイル上）で起きる。マクロ移動に畳まれない」 | 野営=メニューで糧食回復、平原=自動加算。マクロ移動に畳んでしまった | ❌ 明文違反 |

### 診断2: そもそも「移動」が仕事をしていない

マクロ層が果たすべき仕事を分解すると、どれも「移動メカニクス」を要求していない:

| 仕事 | 「移動」が必要か |
|---|---|
| ランの構造（ダンジョンの連なり・ペーシング） | 不要。「次はどれ」の選択列で足りる |
| 圧（寒波前線・一方向・グラインド禁止） | 不要。場面遷移ごとに前線が進めばよい |
| 選択（どの挑戦をどの条件で受けるか） | 不要。分岐の提示はメニューで足りる（Slay the Spire が証明） |
| 経済（供給消費・交易の地理） | 不要。「この道は長い＝糧食◯」はコスト表示で足りる |
| 旅のフィクション | ここだけ空間が欲しいが、読み物（地図・演出）でも立つ |

移動は空間を意味し、空間は踏破を意味し、踏破は「作業（グリッド）」か「無意味（ジャンプ）」に落ちる。反復の振動（選択肢が多すぎ↔無さすぎ、細かすぎ↔抽象的すぎ）は、この層に安定した役割が無いことの症状だった。

さらに design.md 自身が、実装してきたものより軽いものしか要求していない：「ルート選択は長いミクロの合間の**短い幕間**」「ルートは**少数(2〜4)の横並びトレードオフ**」。独立した画面と操作ループを持つ"マクロ移動ゲーム"は過剰だった。

### 参考（移動を持たない成功例）

- **Hades**: マクロ画面ゼロ。部屋のクリア後、出口の扉に次の報酬がプレビューされ、扉を選ぶ＝ルート選択。マクロの全機能がミクロの出口に畳まれている。
- **Darkest Dungeon**: オーバーワールド移動なし。拠点→依頼選択→ダンジョン。旅はダンジョンの廊下の中にある。
- **Slay the Spire**: 地図は全貌が見える読み物＋メニュー。移動感ゼロでも先読みの戦略は成立。
- 反例 **RimWorld キャラバン**: 時間は食うが決定が少ない移動＝作業として悪名高い（前回調査）。

## 要件

- マクロの**仕事**は残す: ランの構造／寒波前線の圧・一方向／横並びの選択／経済（供給・交易）。
- マクロの**画面と移動ループ**は消す: 専用 UI ループ・移動入力を持たない。
- ミクロとの結合を回復する: 選択がミクロの条件を変え、ミクロの結果（時間・戦利品）がランに返る。
- 漏れバケツ禁止を守る: 消費は選択の結果。引き際の判断はミクロ内で起きる。
- 既存資産（`CaravanRun`・遠征選択・`DungeonState`・供給・前線・`WithEscapePop` 機構）を再利用する。実装は主に削除。

## 設計

### 1. ループ（出口選択型）

```
母港(遠征選択) → 場面1 → 出口で次を選ぶ(2〜3ゲート) → 場面2 → … → 目標地点
```

- **場面** = ミクロ（遺跡潜行／集落マップ／野営／イベント）。プレイ時間のほぼ全てはここ。
- **出口ゲート** = ダンジョンの脱出ポータル相当が「次の行き先の選択」になる。各ゲートに行き先プレビューを出す（遺跡=戦利品/危険、集落=交易、野営=回復、目標まで残り N）。
- **遷移時に供給消費・前線前進**。フェード中に表示する:「凍原を3面進んだ。糧食 -6 ／ ❄寒波が3面詰めた」。数値は選んだ辺の距離（面数）×積載で決まる（漏れバケツ回避＝消費は選択の結果）。

### 2. ルート骨格は保持、地図は読み物へ

- ルート網の**骨格グラフ**（route パッケージの生成）は残す。先読み・交易計画（「2つ先に街がある」）のため。
- 全体図は **M キー等で開く read-only の地図**（現在地・先の分岐・街や遺跡の位置）。StS 的な事前計画はここで行う。操作は「閉じる」のみ。
- `MacroMapState`（操作ループ持ちのマクロ画面）は廃止する。

### 3. 選択の中身（マクロが問いを立てる）

ゲートのプレビューが「どんな戦術挑戦を、どの条件で受けるか」を示す。横並び 2〜3 択・支配解なし:

- **遺跡**: 前線との距離で条件が変わる（前線ギリギリ＝凍結して敵が強いが戦利品は手つかず／余裕あり＝暖かいが漁られている）。
- **集落**: 交易・補給。需給プレビュー（「凍晶が高く売れる」）。
- **野営/迂回**: 長く安全（糧食コスト大・前線余裕）／短く危険（節約・圧）。
- **距離（面数）**を各ゲートに明示し、供給コスト差＝選択の重みにする。

### 4. 結合（フィードバックループの回復）

- **前線＝ミクロ内時間**: 潜行の前線コストを「入場時 flat」から「ダンジョン内で経過したターン数に比例」へ変える。ミクロの巧拙＝マクロコストになり、「もう一部屋行くか、荷を確保して引くか」の頂点がダンジョンの中で起きる。HUD に寒波接近を表示する。
- **マクロ状態→ミクロ条件**: 飢餓で潜れば空腹デバフ、過積載なら移動 AP 増（design.md の例そのまま）。前線が近い場面は低温（体温系と接続）。
- **供給の実体化**: 「戦利品価値÷8=糧食」の換算式を廃止し、凍晶（燃料）・糧食袋（食料）を**ダンジョン内の実アイテム**として配置する。拾った物がそのまま供給＝「凍晶を掘りに潜る」が成立する。
- **UI は換算を見せる**: 「この迂回＝遺跡1個ぶんの糧食」「糧食＝あと約 N 面」。

### 5. 実装マッピング

既存の `WithEscapePop`（脱出で呼び出し元へ戻る）機構と `CaravanRun` を土台に、遷移の**向き先**を変えるだけの構成にする:

| 部品 | 現状 | 変更 |
|---|---|---|
| `MacroMapState` | 停留点マップの操作ループ | **廃止**（Phase A4 で read-only 地図として最小復活） |
| `DungeonState` 脱出 | `EscapePop`→MacroMap へ Pop | 脱出＝**次の行き先選択**（MessageState でプレビュー付き 2〜3 択）→ 選択先の場面へ Switch |
| `CaravanRun` | Current/前進/前線/供給 | ほぼ温存。JumpTo→遷移 API に改名。前線のターン比例加算を追加 |
| `route` 生成 | BeaconMap | 骨格として温存（選択肢とプレビューの供給源） |
| 遠征選択 | MacroMap へ Switch | 最初の場面へ直接 Switch |
| 野営 | マクロのメニュー処理 | 当面は選択イベントとして遷移中に処理（後段でミクロ場面化＝逗留ゾーンへの道） |

### 段階

- **Phase A1（畳む）**: 出口選択（MessageState・プレビュー文）＋遷移コスト表示。`MacroMapState` の操作ループ廃止。遠征選択→最初の場面へ。
- **Phase A2（ゲート実体化）**: 出口を Hades 式に複数ポータルとしてマップ上に立てる（ゲートまで歩いて選ぶ）。
- **Phase A3（結合強化）**: 前線＝ミクロ内ターン比例／凍結遺跡（前線距離→温度・敵）／供給の実アイテム化。
- **Phase A4（読み物地図）**: M キーの全体図（read-only）。

## 採用しなかった方法

- **B: 旅をミクロに埋め込む（トラベルレグ＝歩けるタクティカルマップ、54.md Phase 2 の前倒し）** ── 旅の身体感は最も立つが実装が大きく、「歩きの作業感」リスクを 54.md §11 自身が警告している。将来足すなら「マクロ層の復活」ではなく**ミクロ側の拡張**として足す（本設計と両立する）。
- **C: 現行マクロ画面を維持し結合だけ修正（前線＝ミクロ内時間、停留点＝潜行の条件）** ── 結合修正は本設計の Phase A3 に吸収した。マクロ画面という独立ループが残る限り「幕間が別ゲームに感じる」構造問題が解けないため、画面ごと畳む。
- **マクロ移動の歴代形式（ノード網／レーン／グリッド／FTL停留点）** ── それぞれ「経路が絡む」「選択が無い」「移動ごとの操作が多い」「イベントが空」で失敗。共通原因は移動メカニクス自体に安定した役割が無いこと（診断2）。

## コメント

- 本設計は「マクロの**仕事**は正しいが、マクロ**移動**は正しくなかった」という結論。仕事（問い・圧・経済）は場面遷移とプレビューに畳み、移動を消す。
- 一方向・寒波前線・供給・遠征選択・交易など 54.md/55.md の確定方針はすべて維持される。変わるのは「それをどの画面で表現するか」だけ。
- 未解決の質問:
  - 野営・集落を「場面」としてどこまでミクロ化するか（design.md の逗留ゾーン構想への道筋。Phase A2 以降で判断）。
  - 出口選択で「選ばず留まる」を許すか（前線が詰むだけの停滞。原則許可でよいはず）。
  - 目標到達後の継続（次の遠征選択）は現仕様を維持。
