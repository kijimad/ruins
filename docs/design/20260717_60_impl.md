## 1. 全体像 — 何を作ったか

CDDA の reality bubble 方式を 1D（東西のみ）に絞り、**アクティブ帯 ＝ 常に一定幅の単一マップ**に畳むことで、既存の単一マップ機構（`Dungeon.Level`・`SpatialIndex`・`CanMoveTo`・描画・AI）を**破壊的変更なし**に無限東進へ拡張した。

```
 西 ◄──────────────────  東進の絶対軸 eastIndex  ──────────────────► 東
                                                              (未生成)
         ┌──── アクティブ帯 = 単一マップ (幅 K*chunkW, 高さ H) ────┐
 寒波前線 │  chunk[west]   │ chunk[center] │  chunk[east]   │
 (未実装) │  破棄候補       │  プレイヤー P   │  先読みバッファ  │   ◄ これ以東は
         │                │               │                │     まだ無い
         └────────────────┴───────────────┴────────────────┘
          ↑localX=0                                  localX=K*chunkW↑
          bandOriginX = eastIndex*chunkW ;  absX = localX + bandOriginX
          （描画・AI・ターン・SpatialIndex は帯内ローカルだけで動く＝現状のまま）
```

## 2. アーキテクチャ — 層の分離

下位のみに依存する4層スタック。循環なし。各層とも単体テスト済み。

```
┌─────────────────────────────────────────────────────────────┐
│ internal/states                                              │
│   OverworldState : ゲームループ接続                          │
│     - OnStart: 初期帯 K チャンク生成 + プレイヤー中央配置     │
│     - Update : DungeonState 共通処理 + maybeShift            │
│     - maybeShift: ターン境界でシフト判定                      │
│   factories.go : デバッグメニュー起動動線                     │
└───────────────┬─────────────────────────────────────────────┘
                │ uses (ChunkGen 注入)
┌───────────────▼─────────────────────────────────────────────┐
│ internal/overworld                                           │
│   ChunkGen : Band と実 mapgen の橋渡し                        │
│     - ChunkSeed(runSeed, chunkIndex): 決定的 seed 導出         │
│     - NewChunkGen: Plan → SpawnAt(offsetX) の注入実装          │
└───────┬──────────────────────────────┬──────────────────────┘
        │ uses                          │ uses
┌───────▼──────────────┐   ┌────────────▼─────────────────────┐
│ internal/worldstream │   │ internal/mapplanner              │
│  純 ECS・帯管理        │   │  OverworldField : 通行保証生成    │
│   - Band (shift)      │   │   - open-default(全面通行可能)    │
│   - AbsTileX (座標)    │   │   - OverworldBarriers(まばら障壁) │
│   - 原子操作           │   │   - 「どの列も塞がない」保証       │
│     (translate/remove)│   │ mapspawner.SpawnAt : offset 配置  │
└──────────────────────┘   └──────────────────────────────────┘
```

**依存の向き**: `states → overworld → {worldstream, mapplanner, mapspawner}`。`worldstream` は最下層で mapplanner/mapspawner に依存しない（`ChunkGen` 注入で分離）。

## 3. 座標モデル — 絶対軸と帯ローカルの分離

CDDA の `abs`/`bub` 型分離を 1D で軽量に再現（`worldstream/coord.go`）。

```
絶対軸 (無限, AbsTileX)     0        100       200       300       400 ...
                          ├─────────┼─────────┼─────────┼─────────┤
                                    │◄──── アクティブ帯 ────►│
帯ローカル (有界, consts.Tile)      0        100       200      (=K*chunkW)
                                    ├─────────┼─────────┤
                                    ▲
                          bandOriginX = eastIndex*chunkW  (例: eastIndex=1 → 100)

  変換:  absX  = localX + bandOriginX      (ToAbs)
         localX = absX  - bandOriginX      (ToLocal)
```

- **`GridElement{X,Y}` は常に帯ローカル・有界**（`0..K*chunkW`）。ECS・描画・AI・SpatialIndex はこれだけを見る＝**既存コードを一切変えない**。
- **`AbsTileX`** は `consts.Tile` と**別型**。前線 `frontEast`・`bandOriginX`・到達距離スコアはこの絶対軸で扱い、ローカル座標との取り違えをコンパイルで弾く。
- 南北はストリーミングしない（高さ固定の帯）ため絶対軸は **X のみ**。

## 4. シフト機構 — 東へ1回スライドする流れ

`Band.ShiftEast`（`worldstream/band.go`）が設計 §2.2 の擬似コードを実装。

```
【Before】P が中央チャンク東端(localX=2w)を越えた            eastIndex = n
   [west]          [center]         [east]
   ┌────────┬────────────────┬────────┐
   │ 破棄→  │              P►│        │
   └────────┴────────────────┴────────┘
   0        w               2w  P     3w   (w = chunkW)

  ShiftEast の手順:
   ① RemoveEntitiesInXRange(0, w, keep=Player|Squad)   西端チャンク破棄
   ② TranslateAllEntities(dx = -w)                     全エンティティを西へ(リベース)
   ③ rebaseCoordMaps(-w):
        - ExploredTiles     : キーを -w 付け替え(帯外は破棄)    ← 永続記憶
        - VisibleTiles      : クリア(次フレーム再構築)          ← 揮発
        - LightSourceCache  : クリア                            ← 揮発
        - SpatialIndex      : 無効化                            ← 揮発
   ④ eastIndex++  (= n+1) , bandOriginX += w
   ⑤ gen(chunkIndex = eastIndex+K-1, offsetX = (K-1)*w)  東端を決定的生成

【After】P は再び中央、帯座標は 0..3w のまま               eastIndex = n+1
   [center']        [east']          [new]
   ┌────────┬────────────────┬────────┐
   │        │              P │ 新規生成 │
   └────────┴────────────────┴────────┘
   0        w               2w        3w
   （GridElement は常に 0..K*chunkW ＝ 無限座標にならない）
```

**ヒステリシス（スラッシング防止, §2.1）**: 中央チャンク `[centerSlot*w, (centerSlot+1)*w)` にプレイヤーが居る間はシフトしない。境界ちょうどで往復しても生成/破棄を繰り返さない。`ShiftWest` は対称（東端破棄・`+w` リベース・西端生成）。

**CDDA との相違**: CDDA は submap ポインタをスライドさせる（データ不変）が、本作は**全エンティティを平行移動**する。帯が小さい（数千エンティティ）ため単純な平行移動で十分で、`SpatialIndex`/`CanMoveTo`/描画/AI をチャンク合成対応に書き換える破壊的変更を回避できる。

## 5. ゲームループの発火点

```
  プレイヤー移動(1タイル入力)
        │
        ▼
  DungeonState.Update (埋め込み) : 入力→アクション→システム列
        │   AnimationSystem / TurnSystem / VisionSystem / CameraSystem / HUD …
        ▼
  OverworldState.Update : trans==None なら maybeShift へ
        │
        ▼
  maybeShift (§2.5 ターン境界フック)
        │  Phase == Player かつ プレイヤー非アクティビティ か?
        ├─ No ──► 何もしない（アニメ/Activity 中の座標平行移動を避ける）
        └─ Yes ─► localX が中央チャンクを出た?
                    ├─ 東へ ─► Band.ShiftEast(world, gen)
                    └─ 西へ ─► Band.ShiftWest(world, gen)
```

安定点（Player フェーズ・非アクティビティ）でのみシフトするので、移動アニメ補間中・走行 Activity 実行中に全座標を平行移動して壊すことがない。

## 6. 決定的生成と通行保証

### 決定的チャンク生成（`overworld/chunkgen.go`）
```
chunkIndex ──► ChunkSeed(runSeed, chunkIndex)  [splitmix64 混合]
                     │  同じ (runSeed, chunkIndex) → 常に同じ seed
                     ▼
              mapplanner.Plan(chunkW, chunkH, seed, planner)
                     │
                     ▼
              mapspawner.SpawnAt(plan, offsetX, 0)  [帯スラブへ配置]
```
破棄チャンクは保存せず、必要なら seed から再生成できる（非永続）。

### 通行保証（`mapplanner/overworld_field.go`）
従来のダンジョンは「壁デフォルト＋部屋を掘る」ため独立チャンクを継ぐと境界が壁で詰まりうる。オーバーワールドは逆に：
```
① NewFillAll("dirt")     : 全面を通行可能な地面で埋める（デフォルト通行可）
② OverworldBarriers      : まばらな障壁(稜線)を置く。各ブロブ高さ ≤ h/3
③ carveEastWestPath      : 西端→東端まで4連結した通路を1本掘る（蛇行時は縦も連結）
     → 障壁をどう置いても西端→東端が確実に4連結する（BFSでテスト検証）
```
注意: 「各列に通行可能タイルが1つ以上」だけでは E-W 経路を保証できない（列ごとに通行可能な行がバラけて縦に途切れると連結しない）。よって per-column ではなく**連結した通路を constructive に掘る**ことで真の連結性を保証する。これにより継ぎ目のオートタイルは"見た目だけ"の問題に格下げされ、独立生成チャンクを継いでも東へ抜けられる。

## 11. 設計の核（なぜこの形か）

3つの調査エージェントが「無限座標・SpatialIndex 複数化・無限セーブ」を🔴高難度と評価したが、**アクティブ領域を「常に一定幅の単一マップ」に畳む（スライディング帯＋リベース）**ことで、これらを**そもそも発生させない**構造にした。本作固有の「一方向・非永続」が CDDA 最大の複雑さ（後端の永続化・再訪整合）をまるごと免除し、破棄チャンクは seed 決定的再生成で足りる。58.md のカリング（AI・描画）が「帯を広げても実処理を視界/活性半径内に抑える」性能前提を先に整えていたことが、この設計を成立させている。
