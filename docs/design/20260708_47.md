# グローバルなユーザー設定の永続化

## 背景

解像度・音量・キーバインドのような「プレイヤーがゲーム内で変更し、次回起動時も保持したい」設定を保存する仕組みが存在しない。

現在 `internal/config` は `caarlos0/env/v11` による**環境変数ベースの起動設定**であり、以下の責務を持つ。

- プロファイル（production/development）に応じたデフォルト値
- アセットパス、デバッグフラグ、プロファイリング設定
- ウィンドウサイズ（`RUINS_WINDOW_WIDTH` / `RUINS_WINDOW_HEIGHT`）の環境変数上書き

しかしこれは「開発者が起動時に渡す」ための仕組みであり、プレイヤーがオプション画面から変更してファイルに書き戻す用途には向かない。環境変数はゲーム内から永続化できないためである。

解像度はハードコードのデフォルト（960x720、`config.go:89-94`）で、変更しても次回起動時に失われる。将来的に音量設定を追加する予定があり、この「永続化されるユーザー設定」の器を用意する必要がある。

## 要件

- プレイヤーが変更した設定を**ファイルに永続化**し、次回起動時に復元する
- ゲーム内のオプション画面から**read/write**できる
- 将来の音量設定・キーバインド追加が**フィールド追加だけ**で済む
- 保存先は Steam 配信で問題が起きない場所にする
- 起動時に設定ファイルが存在しなくても**デフォルト値で動作**する
- 設定の保存失敗で**ゲームをクラッシュさせない**
- 実行時の設定の真実を1箇所（`config.Config`）に保ち、二重管理を避ける
- デバッグ/プロファイル用フィールドが誤ってユーザー設定ファイルに漏れない構造にする
- テスト時にファイルI/Oを実ファイルに依存せず検証できる

本ドキュメントのスコープは「**設定を永続化する器**」（config への永続化機構の追加 + 起動時の配線）までとする。実際に解像度を切り替えるウィンドウリサイズ・論理解像度スケーリング（`maingame/game.go:50` の TODO）は設計が別問題のため対象外とする。

## 設計

### 方針: config に集約し、永続化対象をネストしたサブ構造体で表現する

独立した `settings` package を新設せず、`config` package に永続化機構を持たせる。実行時の設定の真実は `config.Config` 単一とし、既存の `cfg → ebiten → Resources` 配線をほぼ変えない。

「どのフィールドを永続化するか」は**構造体の境界で表現する**。`Config` から永続化対象だけを `UserConfig` サブ構造体に括り出し、ファイルには `cfg.User` のみをread/writeする。除外タグ（`toml:"-"`）を手作業で管理しないため、デバッグフラグが漏れる事故が原理的に起きない。

### Config の再構成

現在 `Config` 直下にある `WindowWidth` / `WindowHeight` / `Fullscreen` を `UserConfig` へ移す。

```go
type Config struct {
    Profile Profile `env:"RUINS_PROFILE" envDefault:"production"`

    // 永続化されるユーザー設定
    User UserConfig

    // 以下、環境変数ベースの起動専用フィールド（永続化しない）
    Debug         bool `env:"RUINS_DEBUG"`
    // ... 既存のデバッグ/プロファイル/演出フィールド ...
}

// UserConfig はプレイヤーが変更・永続化するユーザー設定を表す。
// ここに含めたフィールドだけが settings ファイルへ書き出される。
type UserConfig struct {
    WindowWidth  int  `env:"RUINS_WINDOW_WIDTH"`
    WindowHeight int  `env:"RUINS_WINDOW_HEIGHT"`
    Fullscreen   bool `env:"RUINS_FULLSCREEN"`
    // 将来: Volume int
}
```

`caarlos0/env` はネストした構造体を再帰的に読むため、環境変数タグは既存のまま `UserConfig` のフィールドへ移すだけで動作する。

### 永続化対象のデフォルト値

ウィンドウのデフォルト（960x720）はプロファイルで差が無いため、`ApplyProfileDefaults` から切り出して1箇所にまとめる。

```go
// DefaultUserConfig は永続化設定のデフォルト値を返す。
// ファイル欠損時・新規フィールド追加時の補完に使う。
func DefaultUserConfig() UserConfig {
    return UserConfig{WindowWidth: 960, WindowHeight: 720, Fullscreen: false}
}
```

### 保存先

`os.UserConfigDir()` 配下にアプリ名のサブディレクトリを1つ切り、その中に置く。

```go
// UserConfigPath は永続化設定ファイルの絶対パスを返す
func UserConfigPath() (string, error) {
    dir, err := os.UserConfigDir()
    if err != nil {
        return "", err
    }
    return filepath.Join(dir, "ruins", "settings.toml"), nil
}
```

- Linux: `~/.config/ruins/settings.toml`
- Windows: `%AppData%\ruins\settings.toml`
- macOS: `~/Library/Application Support/ruins/settings.toml`

ユーザーから見て増えるのは `ruins/` フォルダ1個のみで標準作法に沿う。バイナリ横（インストールディレクトリ）は Steam の整合性チェックやアップデートで削除・上書きされ得るため避ける。将来 Steam Cloud を使う場合も、SteamworksSDK統合ではなく Auto-Cloud のパスパターン登録（コード不要）でこのパスをそのまま同期できる。

### 読み込み（cfg.User を上書き）

```go
// LoadUserConfig はファイルから c.User を上書きする。
// ファイルが存在しない場合は何もしない（デフォルトのまま）。
func (c *Config) LoadUserConfig() error {
    path, err := UserConfigPath()
    if err != nil {
        return err
    }
    data, err := os.ReadFile(path)
    if errors.Is(err, os.ErrNotExist) {
        return nil // 初回起動時はデフォルトで動作する
    }
    if err != nil {
        return err
    }
    // c.User を土台に Unmarshal するため、ファイルに無いフィールドはデフォルトが残る
    return toml.Unmarshal(data, &c.User)
}
```

`c.User` を土台に `Unmarshal` することで、設定ファイルに存在しないフィールド（音量追加直後など）はデフォルトが残る。

### 書き込み（アトミック + 失敗耐性）

```go
// SaveUserConfig は c.User をファイルへ書き込む。ディレクトリが無ければ作成する。
func (c *Config) SaveUserConfig() error {
    path, err := UserConfigPath()
    if err != nil {
        return err
    }
    if err := os.MkdirAll(filepath.Dir(path), 0o755); err != nil {
        return err
    }
    var buf bytes.Buffer
    if err := toml.NewEncoder(&buf).Encode(c.User); err != nil {
        return err
    }
    // 一時ファイルへ書いてから rename することで、途中クラッシュでの破損を防ぐ
    tmp := path + ".tmp"
    if err := os.WriteFile(tmp, buf.Bytes(), 0o644); err != nil {
        return err
    }
    return os.Rename(tmp, path)
}
```

呼び出し側（オプション画面）は `SaveUserConfig` の戻り値エラーを**ログに出すが握り潰し、ゲームは継続**する。設定保存の失敗でクラッシュさせない。

### 優先順位（3層）

現在の `Load()` は「デフォルト適用 → 環境変数上書き」の2層。ここにファイル層を中間に挟み、**環境変数 > ファイル > デフォルト**とする。

```go
func Load() (*Config, error) {
    cfg := &Config{}
    // 1. プロファイルのデフォルトを敷く（cfg.User は DefaultUserConfig で補完）
    cfg.ApplyProfileDefaults()
    // 2. ファイルで cfg.User を上書き
    if err := cfg.LoadUserConfig(); err != nil {
        // 読み込み失敗はログのみ。デフォルトで継続する
        log...(err)
    }
    // 3. 環境変数で明示指定された値を上書き（最優先、開発・デバッグ用）
    if err := env.Parse(cfg); err != nil {
        return nil, ...
    }
    // Seed/RNG/Validate は従来どおり
}
```

`env.Parse` を最後に置くことで、環境変数が指定されたフィールドはファイル値より優先される。ファイル読み込み失敗は握り潰してデフォルトで継続する。

### 呼び出し側の変更

`cfg.WindowWidth` 等の参照を `cfg.User.WindowWidth` に変更する。参照箇所は以下の3つのみで、コンパイラが漏れなく検出する機械的な変更である。

- `internal/cmd/play.go:56` `ebiten.SetWindowSize(cfg.User.WindowWidth, cfg.User.WindowHeight)`
- `internal/cmd/play.go:60` `if cfg.User.Fullscreen`
- `internal/maingame/game.go:166` `SetScreenDimensions(cfg.User.WindowWidth, cfg.User.WindowHeight)`

オプション画面から設定を変更する際は `cfg.User` を更新し `cfg.SaveUserConfig()` を呼ぶ。`cfg` は既に `world.Config` として参照可能なため、Resources への追加配線は不要。

### TOML ライブラリ

`raw.toml` で既に使用している `github.com/BurntSushi/toml`（v1.6.0）を用い、依存を増やさない。エンコードは `toml.NewEncoder(w).Encode`、デコードは `toml.Unmarshal` を使う。

## 採用しなかった方法

- **独立した `settings` package を新設する**: 責務分離は明快だが、解像度の値が `config.WindowWidth` と `settings.Display.Width` に二重化し、実行時の真実が2箇所になる。既存の `cfg → ebiten → Resources` 配線に settings を割り込ませる追加配線も必要。config に集約する方が二重管理を避けられる
- **flat な config のまま小DTOへ射影する**: 永続化する部分だけ別structへ手動で射影する方法。config構造を変えずに済むが、Load/Save の射影コードを手書きし、フィールド追加のたびに射影も直す二重管理になる。ネストなら射影不要で `cfg.User` をそのままread/writeできる
- **flat な config のまま `toml:"-"` で除外する**: 永続化しない約30フィールド全てに除外タグを付ける方法。1つ付け忘れるとデバッグフラグやプロファイル設定がユーザー設定ファイルに漏れる。最も脆く却下
- **環境変数だけで対応する（`RUINS_VOLUME` 等）**: 実装は最小だが、ゲーム内で変更して永続化できない。プレイヤーが環境変数を編集する前提は非現実的でオプション画面と両立しない
- **セーブデータに相乗りさせる**: 設定がセーブスロット依存になり「どのセーブでも音量は共通」という自然な期待に反する。初回起動やタイトル画面で使えない
- **保存先をバイナリ横にする**: ポータブル運用には向くが、Steam の整合性チェック・アップデートで削除/上書きされ得る、書き込み権限が無い構成があり得るという実害リスクがある

## コメント

- 本スコープは「永続化の器」まで。実際の解像度切り替え（ウィンドウリサイズ・論理解像度スケーリング）は `maingame/game.go:50` の `Layout()` 設計と絡むため別タスクとする
- 永続化対象は `UserConfig` の境界で決まる。将来 `Volume` を足す際は `UserConfig` にフィールドと `DefaultUserConfig` の該当行を追加するだけで、ファイル入出力コードは変更不要
- `config` package の責務が「環境変数からの起動設定の読み込み」に加えて「ユーザー設定の永続化」へ広がる。doc.go に両者の使い分けを明記する
- Steam Cloud 同期は将来 Auto-Cloud のパスパターン登録で対応でき、本設計の保存先はそのまま載せられる
- 設定保存・読み込みの失敗はログのみでゲームを継続し、書き込み権限問題等でクラッシュさせない
- テスト容易性のため、ファイルI/Oの本体をパスを引数に取る非公開ヘルパー（`loadUserConfigFrom`/`saveUserConfigTo`）へ分離した。公開メソッドは `UserConfigPath()` を解決して委譲する。これによりテストは環境変数（`XDG_CONFIG_HOME`）を差し替えず、一時ディレクトリのパスを直接渡せる。`t.Setenv` は並列テストと両立しない（paralleltest linter と衝突する）ため、この分離で全テストを並列・env非依存にできた
- ネストした `UserConfig` への環境変数パースは `caarlos0/env` が構造体を再帰的に処理するため、既存の env タグを移すだけで機能することを確認した
- 初回起動時のファイル生成（`EnsureUserConfigFile`）は `config.Load()` 内ではなくアプリ起動側（`runPlay`）に置いた。`Load()` に書き込み副作用を持たせると、`Load()` を呼ぶ `TestLoad` が実行のたびに実際の `~/.config/ruins/settings.toml` を生成してしまう。ファイル生成を起動時の副作用として分離することで `Load()` を読み取り専用に保ち、テストがユーザーのホームを汚さないようにした。責務としても `Load()`=設定解決、`runPlay`=起動時の副作用（ウィンドウ設定・プロファイラ起動など）と整合する
- 生成するファイルは `DefaultUserConfig()` の値を書き出す。環境変数による上書き前のデフォルトを書くため、開発用の一時的な環境変数値がファイルに焼き付かない
