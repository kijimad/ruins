# シームレスワールドの実装 ── スライディング帯によるチャンクストリーミング

## 背景

57.md で「メタマップを廃止し、単一解像度のシームレスワールドへ」という方向を確定した。母港・集落・遺跡・野営地・目標地点をすべて同じ座標空間・同じタクティカル解像度のタイル地図上に置き、1歩ずつ歩いて広域を横断する。しかし 57.md §6 は具体的な生成方式を「未解決の設計判断」として残していた:

- ワールドの生成方式。有限の事前生成か、チャンクストリーミングで無限生成か
- 寒波前線の空間表現
- 「一方向・戻れない」の空間的強制
- セーブ。連続マップの容量・チャンク管理をどうするか
- 既存 `DungeonState` の 50×50 マップとの解像度整合

本ドキュメントは、design.md が要求する **「延々と進める」シームレス東進** を実装するための具体設計を定める。すなわち 57.md §6 の生成方式を「**チャンクストリーミング**」に確定し、既存アーキテクチャへの接続方法を示す。この方式は無限・決定的再生成・非永続である。

58.md は AI カリングと SpatialIndex 増分更新を扱う。それと描画のビューポートカリングは、本設計の「アクティブ領域だけを処理する」性能前提を先に整えたものであり、本設計はその上に立つ。

### 現状実装の要点（調査結果）

- **地図生成**: `mapplanner.Plan(world, w, h, seed, type) → *MetaPlan` は seed 決定的。`MetaPlan.Tiles[]` は `w×h` のフラット1次元配列。`mapspawner.Spawn(world, metaPlan) → gc.Level` が ECS 化する（1タイル=1エンティティ、床=`GridElement`+`SpriteRender`、壁/void=+`BlockPass`+`BlockView`）。
- **寸法**: `consts.MapTileWidth/Height = 50` を `DungeonState.OnStart` がハードコード。
- **座標系**: `gc.GridElement{X,Y consts.Tile}`、`Tile` は `int`（負・大座標は型上は可）。原点 (0,0) 左上。
- **シングルトン**: `Dungeon`（`Level`/`Depth`/`ExploredTiles`/`VisibleTiles`/`LightSourceCache`）、`SpatialIndex`（`MapWidth/Height` スカラー・`BlockPass`/`Characters` マップ・毎ターン再構築・増分 `MoveCharacter`）、`TurnState`。すべて **単一マップ前提**。
- **フロア遷移**: ポータル→`RequestStateChange`→フェード→`TransReplace` で新 `DungeonState`。`OnStop` が Player/SquadMember 以外を全削除し、`OnStart` が全再生成する（＝マップは常に1枚・作り直し）。
- **移動/境界**: `CanMoveTo` が `0<=x<MapWidth` と `SpatialIndex.BlockPass` で判定。境界外は void タイル。
- **カリング（既存）**: AI は `activationRadius`（`VisionRadiusTiles+2`）で距離カリング、描画は `viewportTileBounds` でビューポートカリング。
- **セーブ**: `ark-serde` でワールド全体を JSON 化。`SpatialIndex`/`Position` 等は `skipComponents` で除外。**ワールド全体を1ファイルに直列化**するため無限マップには耐えない。

### CDDA の参考（裏取り済み）

design.md/57.md が範とする CDDA は「ミクロで広域を表現する（ミクロとマクロに壁がない）」構造。その地図実装:

| 単位 | サイズ | 役割 |
|---|---|---|
| submap | 12×12 タイル | **ストリーミングの単位**（load/save） |
| overmap terrain (OMT) | 2×2 submap = 24×24 | **mapgen の単位**（1回の生成） |
| overmap | 180×180 OMT | 地域生成の単位 |
| reality bubble (map) | 約11×11 submap ≈ 120タイル四方 | プレイヤー中心の**アクティブ領域**。ここ**だけ**が実シミュレーション対象。外は存在しない |

- `map::shift()`: プレイヤーが submap 境界を越えると、`grid` ポインタをずらしてプレイヤーを中央に保つ。先頭辺を生成（未訪問なら overmap 地形種別から）、後端を MAPBUFFER/ディスクへ退避。
- **reality bubble** の外はモンスターを overmap 上の抽象グループに退避し、実シミュレーションしない。

本作が引き継ぐ／引き継がない点は §設計「7. CDDA から引き継ぐもの・引き継がないもの」に整理する。

## 要件

- **延々と東へ歩ける**。マップ端で行き止まらず、前方が継ぎ足され続ける。専用画面・ロード停止を挟まない（シームレス）。
- 57.md の単一解像度原則を守る: 別解像度の広域画面を作らない。アクティブ領域は既存のタクティカル移動・戦闘・stats・描画・AI をそのまま使う。
- **既存の単一マップ機構（`Dungeon.Level`/`SpatialIndex`/`CanMoveTo`/描画/AI/ターン）への破壊的変更を最小化する**。3エージェント調査が「🔴 高難度」と評価した「無限座標」「SpatialIndex 複数化」「無限セーブ」を、設計で回避する。
- 生成は **seed 決定的**（既存 `Plan` を再利用）。同じ run seed・同じ東進位置なら同じ地形。
- **一方向**（戻れない）と**寒波前線が背後を呑む**を空間的に表現する。
- セーブ容量は東進距離に依存せず**有界**に保つ。
- CLAUDE.md 原則（シンプルさ・テストしやすさ）。生成・帯管理・前線計算は ECS/Ebiten 非依存の純ロジックに寄せ、テスト可能にする。
- 遺跡（Z方向の別マップ）は既存のポータル＋`TransPush` をそのまま使う（57.md §「単一解像度は Z 方向を否定しない」）。

## 設計

### 1. 中核モデル: 固定高さの帯 × E-W 1次元ストリーミング

- **ワールド ＝ 高さ固定・東西に無限の帯**。design.md の「東西に長い横長」に対応する。高さ `H` は固定で、南北は経路選択の幅として分岐→合流を収める程度、目安 50〜80 タイル。**南北方向のストリーミングは行わない**ので、CDDA の 2D ストリーミングより大幅に単純になる。
- **チャンク、すなわち区画** ＝ 幅 `chunkW` × 高さ `H` の縦スラブ。1チャンク ＝ **1回の `Plan()` 出力**。各チャンクは `(runSeed, eastIndex)` から決定的に生成する。`chunkW` は調整可能パラメータで目安 50〜100。
- **アクティブ帯**、reality bubble 相当 ＝ 東西に隣接する `K` 個のチャンクを、**ひとつながりの連続タイル座標空間**に並べたもの。幅 `K*chunkW` × 高さ `H`。`K` は「西に1・現在・東に1〜2」で最低 3、ビューポート余裕を見て 3〜4 に調整できる。
- **この帯こそが、既存エンジンから見える「単一のマップ」である**。既存の `Dungeon.Level`・`SpatialIndex`・`CanMoveTo`・描画・AI・ターンは、幅 `K*chunkW`・高さ `H` の1枚のマップであるこの帯に対してそのまま動く。50×50 の単一マップ機構が、`K*chunkW × H` の単一マップ機構に「広がるだけ」で、複数マップ管理は導入しない。

### 2. スライドとリベース（CDDA `map::shift` 相当）── 無限座標の回避

- プレイヤーは常に帯の中央付近（現在チャンク）に居るよう保つ。
- プレイヤーが**東隣チャンクの領域へ踏み込んだ**ら「**シフト**」を1回行う:
  1. **東端に新チャンクを生成**: `Plan((runSeed, eastIndex+K-1) …)` → `mapspawner.Spawn`（東スラブへオフセット配置。§5）。
  2. **西端チャンクを破棄**: `GridElement.X < chunkW` のエンティティ（タイル・敵・アイテム等）を `RemoveEntity`。プレイヤー・隊員は中央に居るため対象外。破棄物は「前線に呑まれた」ものとして**保存しない**（一方向・非永続）。
  3. **リベース（全エンティティを西へ平行移動）**: 残る全エンティティの `GridElement.X -= chunkW`。これでプレイヤーは再び中央へ戻り、**帯のタイル座標は常に `0..K*chunkW` に収まる**。
  4. `eastIndex++`。
- 効果: **`GridElement.X` は無限に増大しない**。`SpatialIndex.MapWidth`・`Level.TileWidth` は常に `K*chunkW`（定数）。→ 調査が「🔴 高難度」とした「無限座標」「SpatialIndex 複数化」が**発生しない**。既存の単一マップ機構を温存できる。
- コスト: シフト1回につき「1チャンク生成＋1チャンク削除＋全アクティブエンティティの X 補正」。頻度は「chunkW タイル前進ごとに1回」と低く、各操作はアクティブ帯（有界）に対して O(帯エンティティ数)。
- **東進の絶対位置は `eastIndex`（スカラー）で別に持つ**。前線・生成 seed・到達スコア（distance）はこの `eastIndex`（＋帯内ローカル X）から算出し、タイル座標には載せない。

#### 2.1 シフトのトリガ（ヒステリシス ── CDDA `update_map` に倣う）

CDDA は「プレイヤーを中央 submap 帯に保つ」ことで境界での往復スラッシングを防ぐ（実定数: `SEEX=12`, `MAPSIZE=11`, `HALF_MAPSIZE_X=60`, `MAPSIZE_X=132`）。判定は概念的に:

```
while playerLocalX <  60:      playerLocalX += 12; shift--   // 西へ1 submap
while playerLocalX >= 60+12:   playerLocalX -= 12; shift++   // 東へ1 submap
```

本作（1D）でも同様に、**中央チャンクの幅ぶんのヒステリシス帯**を設ける。プレイヤーのローカル X が**中央チャンクを東へ出たら東シフト、西へ出たら西シフト**。境界ちょうどで往復しても、中央帯を出るまでシフトしないため、生成/破棄を無駄に繰り返さない（CDDA が中央 submap 帯で防いでいるのと同じ）。これは 58.md 以前の設計に欠けていた重要点。

#### 2.2 シフトの擬似コード（本作 1D 版。`K` チャンク、中央は index `west`）

```
onPlayerMoved():
  localX = player.GridElement.X
  if   localX >= chunkW*(west+1):  shiftEast()   // 中央チャンクを東へ出た
  elif localX <  chunkW*west:       shiftWest()   // 西へ出た（短い寄り道）

shiftEast():
  removeEntitiesInXRange(0, chunkW, keep=player|squad)   // 西端チャンク破棄（前線が呑む）
  translateAllEntities(dx = -chunkW)                     // リベース：全エンティティを西へ
  eastIndex++
  plan  = Plan(runSeed, eastIndex+(K-1), chunkW, H)      // 東端チャンクを決定的生成
  spawnChunkAt(plan, offsetX = chunkW*(K-1))
  recalcSeamAutotile(boundaryX = chunkW*(K-1))           // 継ぎ目のオートタイル再計算
  invalidateSpatialIndex()                               // 次ターン開始で再構築
```

`shiftWest` は対称（東端破棄・`+chunkW` リベース・西端生成・`eastIndex--`）。一方向ゲームゆえ西シフトは短い寄り道の復帰時のみ稀に起きる。**`eastIndex=0`（ラン開始点）より西へはシフトしない**——開始点より西には何も生成されておらず、プレイヤーは帯西端（`localX=0` の境界）で自然に止まる。前線（§3）実装前の暫定的な西端制約でもある。1タイル/ターン移動なので1回の移動で跨ぐチャンクは高々1つ、シフトは1回で足りる（CDDA も ±1 submap 単位に分割してループする）。

**隊員の安全レール（Q7）**: `removeEntitiesInXRange` は隊員を削除しない（`keep=player|squad`）が、それだけでは不十分。隊員が戦闘中・地形で足止め・AP 切れで**不本意に遅れ**、リベース後に破棄ライン西（進入不可の極低温ゾーン）へ落ちることがある。これは design.md の「opaque な即死を避ける／損失は"選んだもの"にする」に反する。よって:
- リベース後、破棄ライン西に来た **keep 対象（隊員）は安全な東側タイルへ強制ワープ**させる（不本意な取り残しを救済）。
- 極低温ゾーンでの凍死は「プレイヤーが明示的に待機/居残り命令を出した」場合にのみ起きる＝**選んだ損失**に限定する。デフォルトの追従/護衛 AI は隊員をプレイヤー近傍に保ち、通常は破棄ラインに触れさせない。

#### 2.3 CDDA との相違：ポインタスライドを採らず「全エンティティ平行移動」にする理由

CDDA の `map::shift` は **submap ポインタ配列をずらすだけ**で、submap 内のタイルデータ（submap ローカル座標 0..11）は不変・非コピー（O(MAPSIZE²) のポインタ移動）。巨大な永続 2D ワールドをディスク I/O 付きで流すための最適化である。

本作はこれを**採らない**:
- アクティブ帯は数チャンク（タイル＋敵で数千エンティティ）と小さく、`translateAllEntities`（全 `GridElement.X` を減算）は shift 1回あたり O(帯エンティティ数)＝サブ ms。shift 頻度は「chunkW タイル前進ごと」で低い。
- ポインタスライド方式は「タイルをチャンクローカル座標＋チャンクスロットで持つ」ことを要求し、`SpatialIndex`/`CanMoveTo`/描画/AI を軒並みチャンク合成対応へ書き換える必要がある（本設計が避けたい破壊的変更・調査の🔴課題そのもの）。
- 全エンティティ平行移動なら **`GridElement` は帯ローカルの単一座標のまま**で、既存の単一マップ機構を一切変えずに済む。

＝ CDDA の賢さは「巨大永続ワールド」ゆえの必要悪。本作の小さな非永続帯では**素朴な平行移動の方がむしろ単純で正しい**（57.md §3「CDDA を全模倣しない」の姿勢を実装レベルで適用）。

#### 2.4 リベースが触る「座標キー Map 群」（エンティティ以外）

`translateAllEntities` は `GridElement`（エンティティ）を動かすだけで、**座標をキーに持つシングルトンの Map 群**は動かない。リベース時にこれらも処理しないと、キーがズレて破綻する。

- **座標キー Map は全てキー付け替え（クリアしない）**。リベースは純粋な座標シフトなので、`ExploredTiles`・`VisibleTiles`・`LightSourceCache` を**すべて同じ dx で付け替える**（`new[k.translate(dx)] = v`、破棄ゾーンに落ちたキーは捨てる）。
  - `ExploredTiles`（探索済みフォグ・永続）を全クリアすると生き残ったチャンクが再フォグ化し、「隊員は探索済みエリアのみ行動」（design.md）ゆえ隊員が動けなくなる。
  - `VisibleTiles`/`LightSourceCache`（視界・揮発）を**クリアするとチャンク境界越えのフレームで視界が空になり1フレーム暗転する（チラつき）**。付け替えれば同一フレームの描画で有効なまま保てる（次フレームの `VisionSystem` がどのみち再計算するが、その1フレームの穴を無くす）。当初「揮発はクリアでよい」としていたが、実プレイでチラつきが出たため撤回。
  - コストは map の再構築（O(タイル数)）で、シフト頻度が低いため無視できる。
- **`SpatialIndex`** は描画に使わない論理用途なので `InvalidateSpatialIndex`（ターン開始で再構築）でよい。
- **カメラも要再センタリング**: `CameraSystem` は `Update` 内でシフト前のプレイヤー位置に合わせるため、シフト後にプレイヤーが中央へリベースされると視点がズレてジャンプする。シフトが起きたら `CameraSystem` を再実行して再センタリングする（`OverworldState.maybeShift`）。

#### 2.5 シフトの発火タイミング（確定）

シフトは全座標を書き換える破壊的操作なので、発火点を1つに固定する:

1. 境界を跨いだプレイヤーの一歩が**完全に解決**（移動アニメ完了・その Activity 終了）。
2. `End` フェーズ（ターン境界）で境界判定。
3. 跨いでいればシフト（生成/破棄/リベース、プレイヤーごと中央へ引き戻す）。
4. 次の `Player` フェーズへ。

- **アニメ補間中・Activity 実行中には絶対にシフトしない**（座標が飛ぶ）。ターン境界＝アニメも Activity も静止している唯一の安全点。
- 連続 Activity（走行など）は**毎ターン live な現在位置から次の一歩を再計算**し、絶対目的地をターン跨ぎでキャッシュしない設計にする（既存 `RunActivity` はそう。junction 判定も毎ターン）。これによりステップ間にシフトが挟まっても、リベース後の位置から走行が継続する。

### 3. 寒波前線 ＝ 移動する極低温ゾーン（破棄と圧の統合）

前線を「点の閾値」でなく **幅を持つ極低温ゾーン**として表現する。これにより §2 のシフト破棄と前線を**単一の破棄ライン**に統合できる。

```
| マップ外(破棄・進入不可) | 極低温ゾーン(数チャンク) | ←frontEast | 通常エリア | プレイヤー | 先読み |
        ↑破棄/進入不可ライン = frontEast - coldWidth              （実ターン経過でゆるやかに東進）
```

- **`frontEast`（絶対 X）** ＝ 極低温ゾーンの東端。実ターン経過でゆるやかに東進する（57.md §2「区画単位・間欠的で緩い」）。
- **極低温ゾーン `[frontEast - coldWidth, frontEast]`（数チャンク幅）** ＝ 生存不能な極寒。`tile_temperature` を極低に。プレイヤーはここに飲まれると死ぬ（居座りの罰）。数チャンク手前から「迫る壁」として見える（design.md「常に感じられる遅い地平」）。
- **破棄/進入不可ライン ＝ `frontEast - coldWidth`（ゾーン西端）**。ここより西は**破棄＋進入不可**。プレイヤー移動でも前線前進でも、このラインより西に落ちたチャンクを破棄する。＝ §2 のシフト破棄と前線が**1本のラインに統合**され、2つの破棄機構は存在しない。
- **一方向の強制** ＝ 破棄ラインが物理的な進入不可壁（西へは戻れない。短い寄り道は破棄ラインより東でのみ可）。
- **区画単位の状態**（未到達／侵食中＝極低温ゾーン内／破棄）を持つ（57.md §2）。温度書き込みは**チャンク単位**（ゾーンに入ったチャンクの `tile_temperature` を極低に設定）で足り、タイル単位の連続勾配計算は不要。タイル単位の霜演出は「侵食中」チャンク内に閉じる。
- 原則は「滅多に効かないが居座れば確実に効く背景の事実」。主緊張源は前線でなく空腹・燃料・戦闘（57.md §2）。プレイヤーが普通に東進する限り極低温ゾーンは背後に留まり、めったに触れない。

### 4. 生成の決定性と非永続（セーブ有界化）

- 各チャンクは `(runSeed, eastIndex)` から決定的生成。**破棄チャンクは保存しない**——一方向で二度と訪れないため。理論上再訪が要れば seed から再生成できる（CDDA の MAPBUFFER/ディスク保存は不要）。
- **セーブ内容（有界）＝ 帯を丸ごと保存する（採用: 全保存）**: `runSeed` ＋ `eastIndex` ＋ `frontEast` ＋ **アクティブ帯の全エンティティ（18,000 前後のタイルを含む）** ＋ メタ進行。既存の `ark-serde` ワールド直列化を**帯のみ**に適用すればよく、除外設定（`skipComponents`）はそのまま流用。
  - **「有界」は「一定サイズ」の意味**（東進距離に非依存）。帯サイズ相当の数十〜数百 KB を毎回保存する。小さくはないが、**東へ何万タイル進んでもセーブは増えない**。
  - **改変 diff 追跡は不要**。全タイルを保存するので「掘った壁・開けた扉・拾ったアイテム」もそのまま直列化される。seed 再生成＋diff という複雑な機構を持たずに済む（Q6 の判断: 単純さのため全保存を採る）。
- ロードは「直列化された帯エンティティをそのまま復元」＋`bandOriginX`/`frontEast` 復元。未読込チャンク（破棄済み）は一方向ゆえ復元不要。→ 調査が「🔴 高難度」とした「無限セーブ」を回避（有界化）。

### 5. 既存生成器の再利用と継ぎ目（オートタイル連続性）

- `mapplanner.Plan` は seed・幅高さ引数を持つため **1チャンク＝1 Plan 呼び出し**にそのまま対応する（CDDA の「OMT ごとに mapgen」と同型）。
- **`mapspawner.Spawn` に X オフセットを追加**する（現状は絶対 0..W にスポーン）。東スラブへは `X += (K-1)*chunkW` して配置。案: `Spawn` にオフセット引数を足すか、スポーン後に帯へ平行移動するヘルパを噛ませる。
- **継ぎ目のオートタイル**: オートタイルは4近傍ビットマスク（`autotile.go`）。チャンク結合時、境界列は隣接チャンクのタイルを見て**接合後に再計算**する（境界2列だけの局所再計算で十分）。地形種別の連続性（雪原→山地の急な切替回避）は、生成時に隣接チャンクの地形種別を入力する（`eastIndex-1` の種別を渡す）ことで緩和できる。

#### 5.1 通行保証：開けた地形をデフォルトにする専用ワールドプランナー（CDDA 準拠）

**現行の `SmallRoom`/`BigRoom` プランナーは「壁がデフォルトで部屋を掘る」方式で、オーバーワールドに流用すると「独立生成した隣チャンクの境界が両側とも壁＝東進で詰む」問題を必ず起こす**。CDDA の原野（field/forest）は逆に「**通行可能がデフォルト、障壁は例外**」で、これが継ぎ目の通行を構造的に保証している（調査: `overmap_terrain` の open land は素で通行可、壁は明示配置のみ）。

本作は**オーバーワールド専用プランナー**を新設し、この原則を採る:
- **雪原/森など開けた地形＝素で通行可能をデフォルト**にする。到達不可の**稜線・断崖を"例外"としてまばらに配置**する（design.md「到達不可地形が自然に合流点・分岐を作る」の実体）。
- **通行保証（連結した通路を1本掘る）**: 「各列に通行可能タイルが1つ以上ある」だけでは E-W 通行を保証できない（列ごとに通行可能な行がバラけて縦に途切れると4連結しない）。よって障壁配置後に、**西端から東端まで4連結した通路を constructive に1本掘る**（上下に蛇行する際は移動元・移動先の両タイルを通し縦の連結も確保）。これで障壁をどう置いても西端→東端が確実に4連結し、独立生成チャンクを東西に継いでも通行できる。継ぎ目のオートタイルは"見た目だけ"の問題に格下げされる。
  - 実装は BFS で「西端列→東端列が4連結」を検証してテスト固定する（per-column passability ではなく実連結性を検証）。
- CDDA の道路網（`overmap_connection` による接続点＋線形地形）は、大陸規模の道グラフが要る場合の上位互換。本作はまず「開けたデフォルト」で足り、将来 street/交易路を明示したくなったら接続点方式へ拡張できる（terrain に接続 ID タグを持たせておくと拡張が楽）。
- 遺跡・集落は「開けた地形の中に置かれた壁がデフォルトの区画（既存プランナー/テンプレート）」として配置する。オーバーワールドの通行保証（開けたデフォルト）と、区画内部の構造（壁デフォルト）は**別レイヤー**として両立する。

### 6. ステート構成（シーン遷移を挟まない東進）

- **オーバーワールドは持続する単一ステート**（新 `OverworldState`、または `DungeonState` を一般化）。東進中の帯シフトは `Update` 内で行い、**シーン遷移（`TransReplace`/フェード）を挟まない**。現状のフロア遷移（`OnStop` 全削除→`OnStart` 全再生成）は East 移動には使わない。
- **遺跡（Z方向の別マップ）への出入りは既存のポータル＋`TransPush` を維持**（57.md §「単一解像度は Z 方向を否定しない」）。オーバーワールドで遺跡区画に入る＝ポータルで別ダンジョンマップへ `TransPush`、脱出は `WarpEscape` で戻る。オーバーワールド帯は `TransPush` 中サスペンドされる（帯の状態は保持）。
  - **`OnPause`/`OnResume` は帯を触らない（`DungeonState` の no-op を継承）**。射撃・観察・ダンジョンメニューなどのオーバーレイも `TransPush` で載るため `OnPause`/`OnResume` が呼ばれるが、これらは同じ世界を描画・操作するだけで帯タイルはそのまま残す必要がある。ここで帯を退避/再生成すると、**オーバーレイ進入で帯タイルが消えて画面が黒くなり、復帰時の `MovePlayerToPosition` が隊員を再配置してしまう**（`OnPause` は載るステート種別を区別できないため、遺跡進入だけを狙って帯を退避することはできない）。
  - **将来ポータルで遺跡へ入れるようにする場合**、帯の退避は汎用フックの `OnPause` ではなく「遺跡進入」専用の経路（ポータル発動時）で行う。現状オーバーワールドにポータルは無く遺跡進入は発生しないため、帯退避は実装しない（YAGNI）。
- 母港・集落も帯上の地点として配置（遺跡と同様、区画内のテンプレート `TemplatePlanner` で生成）。

### 7. CDDA から引き継ぐもの・引き継がないもの

**引き継ぐ**:
- **reality bubble ＝ アクティブ帯**。帯の外は生成も実シミュレーションもしない（既存カリングと同一思想）。
- **`shift` によるリベース**でローカル座標を有界に保つ。
- **地形種別（overmap 相当）→ タイル生成（submap 相当）の 2 段**。本作では「`eastIndex` ごとの地形種別」＝ overmap 相当、「チャンクの実タイル」＝ submap 相当。

**引き継がない**:
- **2D ストリーミング → E-W 1D のみ**（高さ固定の帯）。南北の submap 管理を持たない。
- **submap と OMT の 2 単位分離 → チャンク 1 単位に統合**（一方向・非永続ゆえ細粒度 save 不要）。
- **MAPBUFFER/ディスク永続化 → 非永続（決定的再生成）**。後端は保存せず破棄。
- **恒久拠点/再訪 → 無し**（ローグライク・一方向）。母港はメタ層のみ（57.md §3）。

### 8. 既存の全走査箇所への影響

アクティブ帯は「一定幅の単一マップ」なので、全走査は**帯サイズで有界**に保たれる（無限化しない）:
- `buildSpatialIndex`（毎ターン全 BlockPass+Character）・`RestoreAllActionPoints`（全 TurnBased）・AI `ProcessAll`（全 SoloAI/SquadAI→距離カリング）は、対象が帯内に限られるため `K*chunkW×H` 相当で頭打ち。`K`・`chunkW` を過大にしなければ現状 50×50 の数倍規模に収まる。
- 描画・AI のカリング（58.md）で、帯を広げても実処理は視界/活性半径内に制限される。
- したがって「無限マップでの全走査爆発」は**帯設計そのものが防ぐ**。追加の空間分割（チャンク別インデックス）は不要。

### 9. 座標規律（絶対/ローカルの分離 ── CDDA の abs/bub に倣う）

CDDA は座標を型で分離する（`abs`＝無限絶対／`bub`＝reality bubble ローカル・有界／`rel`＝差分）。`map.abs_sub` が「bubble 左上の絶対 submap 位置」を保持し、変換は `bub = abs - abs_origin` の単純オフセット。異 origin・異 scale の加算はコンパイルエラーで防ぐ。

本作は **1D・単一 scale** なので大幅に軽く、規律だけ借りる:
- **`GridElement{X,Y}` は帯ローカル・有界**（常に `0..K*chunkW` × `0..H`）。ECS・描画・AI・`SpatialIndex`・`CanMoveTo` はこれだけを見る（現状のまま変更しない）。
- **帯の絶対原点 `bandOriginX = eastIndex*chunkW`（スカラー1つ）** を持つ。絶対位置 `absX = local.X + bandOriginX`。前線 `frontEast`・生成 seed・到達スコアはこの絶対軸で計算。
- 変換は加減算のみ: `absX = localX + bandOriginX` / `localX = absX - bandOriginX`。
- CDDA の型安全（abs と local を混同しない）は、Go では **絶対 X に別名型（例 `type AbsTileX int`）** を与え、絶対軸を扱う関数と `GridElement`（ローカル）を扱う関数を型で分けることで軽く再現する。取り違えをコンパイルで弾く。

これにより「無限の東進位置は `eastIndex`/`bandOriginX` が担い、実シミュレーションは常に有界な帯ローカルで回る」という CDDA reality bubble の核心を、既存コードをほぼ触らずに実現する。

### 10. CDDA との対応表

| CDDA | 本作 | 備考 |
|---|---|---|
| submap（12×12・load/save 単位） | チャンク（`chunkW×H`・生成/破棄単位） | 本作は 1 チャンク＝1 `Plan()` |
| overmap terrain（24×24・mapgen 単位） | ＝チャンク（統合） | 一方向・非永続ゆえ 2 単位に分けない |
| reality bubble（11×11 submap ≈132タイル） | アクティブ帯（`K` チャンク） | `K` は奇数（中央チャンクを持つ）。最小 3 |
| `MAPSIZE=11`（奇数・中央 submap） | `K`（奇数・中央チャンク） | 中央維持でヒステリシス |
| `map.abs_sub`（bubble 絶対原点） | `bandOriginX = eastIndex*chunkW` | 1D スカラー |
| `map::shift`（ポインタスライド） | `translateAllEntities`（平行移動） | §2.3 の理由で方式を変える |
| MAPBUFFER＋ディスク保存 | 非永続（seed 再生成） | uniform submap を保存しない CDDA の極端版 |
| bubble 外モンスター→overmap mongroup | 西端敵は despawn（抽象退避なし） | 一方向で二度と会わない |

## 採用しなかった方法

- **グローバル絶対座標を無限に伸ばす（リベースしない）** ── 実装は素直だが、`GridElement.X` が増大し続け、`SpatialIndex.MapWidth`（スカラー）・`Level.XYTileIndex`（`y*w+x` 線形化）・VisionSystem キャッシュなど**単一マップ前提が各所で破綻**する（3エージェントが「🔴 高難度」と評価）。リベースはこれらを温存できるため不採用にしない理由がない。
- **複数マップ/複数 SpatialIndex を同一座標空間に共存させる（チャンク別インデックス）** ── CDDA の grid ポインタ配列に近いが、`SpatialIndex`・`Dungeon.Level`・`CanMoveTo` を軒並みチャンク対応に書き換える必要があり、破壊的変更が大きい。「帯＝単一マップ」に畳めば不要。
- **2D（南北も）ストリーミング** ── CDDA 準拠だが、本作は「東西の帯」で南北有限のため過剰。1D に限定して大幅に単純化する。
- **破棄チャンクをディスク保存（CDDA MAPBUFFER 準拠）** ── 一方向で再訪しないため無駄。seed 決定的再生成で足り、セーブも有界化できる。
- **有限の広域マップを事前一括生成** ── 「延々と」の無限性と、初期生成コスト・セーブ容量に反する。57.md の「毎周回新規生成」とも整合しない。

## コメント

- 設計の勘所は「**アクティブ領域を常に一定幅の単一マップに畳む（スライディング帯＋リベース）**」の一点。これにより、3エージェント調査が高難度と見た3課題（無限座標・複数 SpatialIndex・無限セーブ）が**そもそも発生しない**構造にできる。既存の 50×50 単一マップ機構を捨てず、幅を広げて滑らせるだけになる。
- 58.md のカリング・増分インデックスは「帯を広げても実処理を視界/活性半径内に抑える」前提として本設計に不可欠。性能基盤が先に入っていたのは順序として正しかった。
- 一方向・非永続という本作固有の制約が、CDDA 最大の複雑さ（後端の永続化・再訪整合）を**まるごと免除**してくれる。CDDA を全模倣せず「引き継がない」判断（57.md §3 の姿勢）を地図実装にも適用した。
- 本ドキュメントは方式の確定と接続方針まで。`chunkW`/`K`/`H` の具体値、継ぎ目オートタイルの詳細、前線の速度式、遺跡/集落の配置アルゴリズムは次段（実装フェーズ）で詰める。

- 実装レビュー（20260717）の対応状況:

  - **B1 シフトの while ループ化（対応済み）**: 複数チャンク跨ぎでも中央に収まるまで連続シフト。停止性・非振動を確認。
