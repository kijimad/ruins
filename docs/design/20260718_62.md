# GitHub Actions CI のキャッシュ最適化（Docker / Go）

## 背景

CI（`.github/workflows/`）の実行が遅い。監査したところ、**キャッシュが効いていない/二重になっている**箇所が主因だった。

### ジョブ別キャッシュ状況（監査結果）

| ジョブ | 実処理 | Go module | Go build cache | Docker layer | 実測 |
| --- | --- | --- | --- | --- | --- |
| `crosscomple` | `make build`（build.sh） | ❌ 空マウント | ❌ 空マウント | ❌ base 毎回再build | 約4m |
| `build-steam` | build_steam.sh | ❌ | ❌ | ❌ | main のみ |
| `wasm`(WASM Build) | build.sh + 全cov test | ❌(build.sh側) | ⚠ setup-go(test側)のみ | ❌ | 全push・重い |
| `lint` | `make toolsinstall lint` | ✅ 二重 | ⚠ | — | 約3m |
| `test`/`flaky`/`generate` | make test 等 | ✅ 二重 | ✅ setup-go | — | — |
| `build`/`image` | Docker release | ✅ BuildKit | ✅ BuildKit | ✅ `type=gha` | 約7m |

### 主因（詳細）

- **F1**: `scripts/build.sh` の `docker build . --target base -t base` に**キャッシュ指定が無い**。fresh runner では base 段の apt install（X11/mesa/openal 十数個）＋upx ダウンロードを毎回ゼロから。`build`/`image` ジョブは同じ base 層を `type=gha` でキャッシュ済みなのに共有できていない。
- **F2**: `build.sh` の `cmd()` は `$HOME/go/pkg/mod` と `$HOME/.cache/go-build` をマウントするが、`crosscomple`/`build-steam`/`wasm` ジョブには**キャッシュ復元ステップが無い**ため空マウント → モジュールDL＋ebiten含む全ビルドが毎回コールド。
- **F3**: `lint` ジョブの `make toolsinstall` が `go install golangci-lint@v2.11.3`（巨大依存を毎回コンパイル）＋`npm install`（lint に不要）＋`setup-hooks.sh`（CI に不要）を実行。
- **F4**: `setup-go@v6`（`go-version-file` 指定＝デフォルト `cache: true`）が `~/go/pkg/mod` と `~/.cache/go-build` の**両方**を自動キャッシュしているのに、手動 `actions/cache`（module のみ）が重複。手動側は build cache を含まない下位互換で、純粋に冗長。
- **F5**: `wasm_build.yml` が `on: push`（全ブランチ全push）でフル Docker ビルド＋全カバレッジ test を回す。gh-pages デプロイは main のみ。

## 要件

- **正しさ・再現性を壊さない**。特に生成物ビルド（クロスコンパイル・Docker release）の成果物同一性を保つ。
- **段階的・低リスク順に**入れる。各段で `make check` と CI 実測で効果を確認する。
- **キャッシュキーは go.sum / Dockerfile 起点**にし、依存が変わったら自然に無効化される（stale キャッシュで誤結果を出さない）。
- 効果は `crosscomple` と `lint` の実測短縮で測る。

## 設計

低リスク・即効の順に4段階。

### Tier 1（低リスク・即効） — F4 / F3

**F4: Go モジュールの二重キャッシュ除去**
`test`/`flaky`/`lint`/`generate` の手動「Cache Go modules」（`actions/cache` on `~/go/pkg/mod`）を削除する。`setup-go@v6` の内蔵キャッシュ（module + build cache 両方、go.sum キー）に一本化する。内蔵キャッシュが効いていることを実測で確認する（cache hit ログ）。

**F3: `lint` ジョブの `toolsinstall` 脱却**
`make toolsinstall lint` をやめ、lint に必要なものだけ入れる。

- golangci-lint は `golangci/golangci-lint-action@v?`（プリビルドバイナリDL＋lint 結果キャッシュ）で導入し、`make lint` 内の `golangci-lint run` はそのバイナリを使う。ソースからの毎回コンパイルを避ける。
- `deadcode` は `go run golang.org/x/tools/cmd/deadcode@latest`（build cache 復元下で高速）か `go tool` 経由にする。
- `npm install`・`setup-hooks.sh` は lint と無関係なので実行しない。
- lint 内の `go build -o /dev/null .` は残す（ビルド健全性の前提確認）。

### Tier 2（中リスク） — F2

**F2: `build.sh` が使う Go キャッシュを CI で復元**
`crosscomple`/`build-steam`/`wasm` の各ジョブに、build.sh が `docker run` でマウントする host ディレクトリ（`~/go/pkg/mod`・`~/.cache/go-build`）を `actions/cache` で復元するステップを足す。キーは `go.sum`＋GOOS 群。これで Docker 内ビルドのモジュールDL・コンパイルが warm start になる。

### Tier 3（要設計判断） — F1

**F1: `build.sh` の base イメージを gha キャッシュで共有**
`crosscomple`/`build-steam`/`wasm` で base 段（apt+upx）を毎回再ビルドしないようにする。方針は次の一択を採る:

- ジョブ側に `docker/setup-buildx-action` を追加し、**`docker/build-push-action@v7`（`target: base`, `load: true`, `cache-from/to: type=gha`）で base イメージを先に構築**してローカルに load する。その後 build.sh を回すが、`build.sh` の `docker build` は既にローカルに `base` が存在すれば**再ビルドをスキップ**するよう改修する（`docker image inspect base` で存在確認）。
- これにより `build`/`image` ジョブと**同じ gha キャッシュ層を共有**でき、base 段のビルドがほぼ消える。

（注: 生の `docker buildx build --cache-to type=gha` を build.sh に直書きする案は、gha キャッシュバックエンドが Actions ランタイムトークンを要求し、シェルスクリプト単体では成立しにくい。ジョブ側で `build-push-action` に寄せる方が確実。）

### Tier 4（任意） — F5

**F5: `wasm_build.yml` のトリガー適正化**
デプロイが main 限定である一方でビルド/テストが全 push で走る点を見直す。`on: push: branches: [main]`（＋必要なら PR プレビュー用に paths フィルタ）へ絞るか、少なくとも Tier 2/3 のキャッシュを効かせる。トリガー変更は挙動影響があるためオーナー確認の上で行う。

| `.github/workflows/check.yml`（lint ジョブ） | F3: `make toolsinstall lint` を廃し、golangci-lint-action 導入＋`make lint` のみ。npm/hooks を除去 |

## 採用しなかった方法

- **`docker buildx build --cache-to type=gha` を build.sh に直書き**: gha キャッシュエクスポータが Actions ランタイムトークン（`ACTIONS_RUNTIME_TOKEN` 等）を要求し、ローカル実行や単体スクリプトでは成立しない。CI ジョブ側で `build-push-action` に寄せる方が堅牢なので不採用。
- **Docker 内 BuildKit cache mount に一元化して host マウントを廃止**: build.sh は `docker run`（build ではない）でコンパイルするため BuildKit の `--mount=type=cache` が効かない。`docker run` 経路を残す限り host マウント＋`actions/cache` が必要。build.sh 全体を `docker build` 内コンパイルへ作り替える大改修は範囲外。
- **セルフホストランナー / 独自キャッシュサーバ**: 効果は大きいが運用コスト増。まず標準アクションのキャッシュ最適化で足りるはずなので不採用。
- **golangci-lint を `go tool` 化**: 20260718_61.md で「巨大依存でアプリの依存グラフを汚すため tool 化しない」と決定済み。プリビルドバイナリ（action）導入がこの方針と両立し、かつコンパイルも消せるので優先。

## コメント

- Tier は独立に入れて各段で効果測定できる。まず Tier 1（F4/F3、低リスク・即効）→ Tier 2（F2）→ Tier 3（F1）の順。
- キャッシュ最適化の落とし穴は「stale キャッシュで誤った成果を出す」こと。キーを go.sum / Dockerfile 起点にし、依存変更で自然失効させる。効果測定は「変更前後の同一コミットでの実測」で行う。
