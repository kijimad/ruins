# Go コード品質ツールの調査と導入方針

## 背景

本プロジェクトは既に golangci-lint v2 で約39 linter を有効化し、`deadcode`・`benchstat`・
`go test -race -shuffle`・goldie（ゴールデン画像）・カバレッジレポート・`depguard`（禁止
パッケージ）・Dependabot・AI レビュー・oapi-codegen まで揃っている。**基本的な静的解析と
スタイルチェックは飽和状態**にある。

一方で「型カバー率は計測できないか」という問いをきっかけに、静的解析の網羅とは別軸の品質
ツールに未着手のギャップがあると分かった。本ドキュメントは、重複を避けつつ本プロジェクトに
効くツールを調査し、導入方針を残す。

### 現状ツール棚卸し

| カテゴリ | 既存 |
| --- | --- |
| lint / 静的解析 | golangci-lint v2（errcheck, govet, staticcheck, revive, gocritic, exhaustive, nilerr, errorlint, testifylint 等 約39） |
| 依存制御 | depguard（math/rand・logrus・pkg/errors を禁止）、Dependabot |
| デッドコード | deadcode, unused |
| テスト | testify, goldie, `-race`, `-shuffle=on`, `-cover` |
| ベンチ | benchstat（CI で base vs PR 比較） |
| 生成 | oapi-codegen, 独自 codegen |
| CI/レビュー | check.yml, Code Metrics, claude-code-review |

## 要件

- 既存の静的解析・スタイル系と**重複しない**こと。lint はこれ以上増やしても限界効用が低い
- **本プロジェクト固有の弱点**を突くこと。すなわち
  - 依存の「実効的な」脆弱性（到達可能なもの）の把握
  - 決定的生成・シリアライズ・座標変換など**入力空間の広いコード**の堅牢性
  - 行カバレッジでは測れない**テストスイートの実効性**
- 依存を無闇に増やさない。標準機能（`go test -fuzz` 等）を優先する
- 導入は小さく始められ、CI に載せられること

## 設計

優先度でティア分けする。

### Tier 1 — 明確なギャップ・高ROI

#### 1. `govulncheck`（公式・脆弱性スキャン）
- Go チーム公式。依存の既知脆弱性のうち**コールグラフで到達可能なものだけ**を報告する
- Dependabot は「依存に脆弱性がある」で件数が出るが到達性を見ずノイズが多い。govulncheck で
  実対応対象を絞れる。CI に数行、`toolsinstall` の系譜に馴染む
- 効果/コスト = 高/極小

#### 2. ファジング（`go test -fuzz`・標準機能・依存ゼロ）
- 決定的生成とラウンドトリップが多く、当たりが良い領域が揃っている
  - `save` の serde ラウンドトリップ（Save→Load 不変、壊れた JSON で panic しない）
  - `mapplanner` の seed→生成の決定性・接続性不変条件
  - `worldstream` の座標変換 `ToAbs`/`ToLocal` の往復、`overworld` の autotile ビットマスク
  - TOML/raw パース
- 効果/コスト = 高/小（標準機能。CI は短時間シード実行、ローカルで長時間）

#### 3. プロパティベーステスト（`pgregory.net/rapid`）
- シームレス周りは**不変条件が言語化しやすい数学的コード**で、例ベースより網羅が効く
  - `band.ShiftEast` 後に `ShiftWest` で元に戻る（可逆）
  - 継ぎ目が**生成順に依存しない**（既存の例ベーステストを全域プロパティ化）
  - `Percent.ApplyInt` の単調性・`0→0`・`PercentBase→恒等`
  - 座標往復 `ToLocal(ToAbs(x)) == x`
- 効果/コスト = 高/小〜中（testify と併用可）

### Tier 2 — テスト品質の深掘り

#### 4. ミューテーションテスト（`gremlins` または `avito-tech/go-mutesting`）
- コードを機械的に改変し「テストが落ちるか」で**テストスイートの実効性**を数値化する。
  行カバレッジより本質的な「テストの強さ」を測れる
- goldie とユニットが厚いので、「そのテストは本当にバグを捕まえるか」を可視化する価値が高い
- コスト = 中〜大（遅い）。変更パッケージに絞って定期実行が現実的

#### 5. `nilaway`（Uber・nil panic 静的解析）
- Ark の `Get()` はポインタを返し、防御的 nil チェックの方針もあるため相性が良い。nil 由来の
  panic を静的に炙り出せる。golangci-lint の**モジュールプラグイン方式**で組み込める
- コスト = 中（誤検知が多め・実験的。まず advisory 実行→段階導入が無難）

#### 6. `go.mod` の `tool` ディレクティブ（Go 1.24+・再現性）
- `toolsinstall` の `go install ...@latest`（非再現）を `go get -tool` でバージョン固定に置換
- golangci-lint は固定なのに goimports/deadcode/benchstat は `@latest`。ツールの
  バージョンドリフトを防ぎ CI とローカルを一致させられる。Go 1.25 使用中で即利用可
- コスト = 小

### Tier 3 — 状況次第

| ツール | 用途 | 位置づけ |
| --- | --- | --- |
| `gofumpt` | より厳格な整形 | golangci-lint 経由で低コスト |
| `gocognit` / `cyclop` | 認知的複雑度 | 既存 `gocyclo` を補完 |
| 依存レイヤ強制（`depguard` 拡張 or `go-arch-lint`） | パッケージ責務境界の強制 | 「責務で package 分割」方針のアーキ番人。例: `components` が `states` を import しない |
| 統合カバレッジ（`go build -cover`, Go1.20+） | 実プレイ中のコード網羅 | 手動プレイ/golden で到達した実行パスを計測 |
| `gosec` | セキュリティ | オフラインゲームで優先度低。Steam/ネット部分があれば一考 |

### あえて採用しない

- 追加のスタイル系 linter: 既に飽和。限界効用が低い
- `testing/synctest`（Go1.25）: 並行性が薄く出番が少ない
- SaaS カバレッジ（Codecov 等）: Code Metrics Report で足りている
- `gosec` の常時有効化: オフラインゲーム主体で誤検知ノイズが上回りやすい

## 変更箇所

| ファイル | 変更内容 |
| --- | --- |
| `Makefile` | `lint` ターゲットに `govulncheck ./...` を統合する（CI の lint ジョブが `make lint` を実行するため専用ジョブ不要） |
| `go.mod` | `go 1.26.5` を Go 版の唯一の源にする。標準ライブラリの脆弱性を修正版で解消。あわせて `tool` ディレクティブで開発ツールを固定する |
| `.github/actions/setup-go-with-x` | `go-version-file: go.mod` にし CI のハードコードを排除、go.mod に自動追従させる |
| `Dockerfile` | ベースを `golang:1.26` にし go.mod + GOTOOLCHAIN=auto で実効版を決める（go.mod 更新に自動追従） |
| `internal/save/*_test.go` | serde ラウンドトリップの Fuzz テストを追加する |
| `internal/mapplanner/*_test.go` | seed→生成の決定性・接続性の Fuzz/プロパティテスト |
| `internal/worldstream/*_test.go` | 座標往復・帯シフト可逆性の rapid プロパティテスト |
| `internal/overworld/*_test.go` | 継ぎ目の生成順非依存を全域プロパティ化 |
| `internal/consts/percent_test.go` | `Percent.ApplyInt` 単調性等のプロパティテスト |
| CI（任意） | gremlins を変更パッケージ限定で定期実行するジョブ |

## 採用しなかった方法

- **すべてを一度に導入**: ツールを一気に足すと CI 時間とノイズが増え、定着しない。Tier 1 から
  小さく入れる
- **lint をさらに増やす**: 静的解析は飽和済み。別軸（脆弱性・fuzz・mutation）の方が費用対効果が高い
- **SaaS 型の品質基盤**: 単独開発かつ既存の Code Metrics で足りるため、外部サービス依存を避ける

## コメント

- 本プロジェクトの弱点は「静的解析の網羅」ではなく **①依存の実効脆弱性 ②生成/シリアライズ/
  座標系の入力空間 ③テストの実効性の可視化**。追加すべきは linter ではなく別軸のツール
- 特に**シームレス実装は不変条件が明確**なので、rapid のプロパティテストと fuzz が最も効く。
  例ベースで拾えない「生成順・seed・境界」のバグに強い
- 進行系倍率が高スキルで 0 停止しうる件（PR #491 レビュー）は本調査の範囲外。バランス設計時に別途扱う

### まず3つ選ぶなら
1. `govulncheck` を CI に追加（ノイズ削減、即効）
2. fuzz + rapid（seamless/serde/mapgen の不変条件を叩く。標準機能中心で依存増を抑える）
3. gremlins をたまに回す（テストの実効性を数値化）
