# 型ユーティリティによる堅牢化・利便化の調査と適用方針

## 背景

「よく使われる型ユーティリティで、より堅牢・便利にできる余地はないか」を4観点（ドメイン型 / 標準ライブラリ generics / 独自ジェネリックヘルパ / 型安全性）で徹底調査した。Go 1.26.5。既に `internal/consts` に newtype 群（`WorldPixel`/`ScreenPixel`/`Tile`/`Chunk`/`Percent`/`Turn`/`AbsTileX`/`Coord[T]`/`Rect`）があり、newtype パターンの下地は整っている。

### 調査で分かった全体像

- **未チェック型アサーションの panic リスクは既に潰れている**: `.golangci.yml` で `forcetypeassert` 有効。手書きコードの未チェック `.(T)` はゼロ（131という数字は生成コード `server.gen.go` 込み）。→ 「アサーション panic」対策は不要。
- **機会は2方向に集約される**: ①**既存ドメイン型を使っていない不整合の解消**、②**Go 1.26 stdlib（slices/maps/cmp）への近代化**（現状わずか4ファイルでしか未使用）。新しい汎用機構の乱造ではない。
- 汎用 util パッケージは無い（`testutil` のみ）。ヘルパを足すなら CLAUDE.md の「責務で package 分割」に従い、汎用 util でなく該当 package（例: `query`）に置く。

## 要件

- CLAUDE.md の「シンプルさ最優先・YAGNI」を厳守。3箇所以上で反復し、かつ堅牢性/利便が具体的なものだけ採用する。
- 既存型・stdlib で足りるものは新設せず適用に留める。
- 各項目は独立に入れ、都度 `make check` で検証・コミット分割する。

## 採用しなかった方法

- **新規 `Set[T]` / find-first / find-nearest / map get-or-default ヘルパ**: いずれも反復1〜2箇所、または述語・戻り型が毎回異なり抽象化コストが DRY 利得を上回る。YAGNI で見送り。
- **`ActionPoints`/`Speed`/`Currency`/`ItemValue` などの量 newtype**: 理論上の取り違えは防げるが実バグ実例が無く、`Pool[T]` ジェネリクスや広範囲へ波及して破壊的。費用対効果が低く保留。
- **マスタ参照キー（RecipeInput.Name 等）の identifier newtype**: 参照先がランタイム map で「存在するキーか」は結局実行時検証。閉じた列挙でもなく、newtype の安全性が小さい割に波及が広い（66箇所超）。見送り。
- **`cmp.Or` / `maps.Keys` の無理な導入**: フォールバックが多行で可読性が落ちる／該当ループが実質無く、置換価値なし。
- **`Mean` 総和ループの stdlib 化**: Go stdlib に sum が無く置換不可。現状維持。

## コメント

- Tier A は「既存型・stdlib を当てるだけ」で新機構ゼロ、実バグ耐性（A1 silent failure / A2 ちらつき / A4 反復中変異）が具体的、範囲も局所。最優先。
- Tier B は低リスク近代化としてまとめて入れられる。
- Tier C は影響範囲が広く設計判断が要るため、A/B 適用後に別 doc で個別検討する。特に C1（hooks）と C2（exhaustive 設定）は全体波及。
- 前提の重要事実として、`forcetypeassert` により未チェックアサーション panic は既に構造的に解決済み。この方向の追加対策は不要。
