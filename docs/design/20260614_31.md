# ロケーションタグの汎用化

## 背景

ロケーションタグはアイテム専用で設計されていた。コンポーネント名に `Item` プレフィックスが付き (`ItemLocationInPlayerBackpack`, `ItemLocationEquipped`, `ItemLocationOnField`)、ヘルパー関数もアイテムを前提としていた。

この制約により、アイテム以外のエンティティがロケーションシステムを利用できなかった。たとえばPropはバックパックに入れる際に `Item` コンポーネントを後付けするハックが必要だった。

```go
// 旧: PropをItem化してからロケーション移動する（pickup.go:168-171）
if !propEntity.HasComponent(world.Components.Item) {
    propEntity.AddComponent(world.Components.Item, &gc.Item{Count: 1})
}
worldhelper.MoveToBackpack(world, propEntity, actor)
```

ロケーションタグを汎用化し、任意のエンティティが参加できるようにする。

## 設計

### 案A: Itemプレフィックス除去 + Owner追加（採用）

コンポーネント名から `Item` プレフィックスを除去し、`InBackpack` に `Owner` フィールドを追加する。

#### コンポーネント変更

```go
// Before
ItemLocationInPlayerBackpack *ecs.NullComponent  `save:"true"`
ItemLocationEquipped         *ecs.SliceComponent `save:"true"`
ItemLocationOnField          *ecs.NullComponent

// After
LocationInBackpack *ecs.SliceComponent `save:"true"`  // NullComponent → SliceComponent（Ownerを持つため）
LocationEquipped   *ecs.SliceComponent `save:"true"`
LocationOnField    *ecs.NullComponent
```

```go
// Before
type LocationInPlayerBackpack struct{}

// After
type LocationInBackpack struct {
    Owner ecs.Entity // バックパックの所有者。プレイヤー、NPC、収納箱など
}
```

`LocationEquipped` と `LocationOnField` は構造変更なし。名前のみ変更する。

### 案B: 単一Locationコンポーネント

**不採用の理由**: ECSのJoinクエリでロケーション種別によるフィルタができなくなる。`Manager.Join(world.Components.LocationOnField)` のようなクエリが書けず、排他コンポーネントパターンのECS上の強みが失われる。

### 案C: ロケーションをインターフェース化

**不採用の理由**: goecs の SliceComponent は `interface{}` で値を格納するため、インターフェース型にしても型安全性の向上がない。追加の抽象化は複雑さだけ増す。

## 実施結果

### Phase 1: リネーム（実施済み）

45ファイルに対してsedで一括置換を実行した。

| 旧名 | 新名 |
|------|------|
| `ItemLocationInPlayerBackpack` | `LocationInBackpack` |
| `ItemLocationEquipped` | `LocationEquipped` |
| `ItemLocationOnField` | `LocationOnField` |
| `ItemLocationType` | `LocationType` |
| `item_location.go` | `location.go` |

enum変数名の衝突を回避するため、`fmt.Stringer` 型の enum 変数名を `LocationType*` プレフィックスに変更した。

```go
var (
    LocationTypeInBackpack LocationType = LocationInBackpack{}
    LocationTypeEquipped   LocationType = LocationEquipped{}
    LocationTypeOnField    LocationType = LocationOnField{}
)
```

### Phase 2: Owner追加（実施済み）

`LocationInBackpack` を NullComponent → SliceComponent に変更し、`Owner` フィールドを追加した。

変更箇所:
- `components.go`: 構造体に Owner フィールド追加、NullComponent → SliceComponent
- `location.go`: `MoveToBackpack` で `LocationInBackpack{Owner: owner}` を設定
- `spawn.go`: `SpawnItem` 内でAddEntities後に player entity の Owner を設定
- `save/manager.go`: 保存時に OwnerRef をシリアライズ、復元の第3段階で Owner を解決
- `oas/typespec/save.tsp`: `MarkerComponent` → `LocationInBackpackComponent{OwnerRef}` に変更
- `oas/openapi.yml`, `oapi/models.gen.go`: TypeSpec → OpenAPI → oapi-codegen で再生成

### Phase 3: Propのハック解消（実施済み）

`collectProp()` から `Item` コンポーネントの後付けハックを除去し、インベントリUIクエリを `Join(LocationInBackpack)` ベースに変更した。

変更箇所:
- `activity/pickup.go`: `collectProp()` から `Item` コンポーネント後付けを除去した。`MoveToBackpack()` の呼び出しのみに簡素化した
- `states/inventory_menu.go`: `queryMenuItem`, `queryMenuWeapon`, `queryMenuWearable` の Join クエリから `Item` を除去した。Stackable の Count 表示に `HasComponent(Item)` ガードを追加した
- `states/shop_menu.go`: `createSellItems` の Join クエリから `Item` を除去した。同様に `HasComponent(Item)` ガードを追加した
- `states/place_state.go`: Join クエリから `Item` を除去した
- `worldhelper/run.go`: `collectBackpackItems` の Join クエリから `Item` を除去した。`Item.Count` の取得に `HasComponent` ガードを追加した
- `activity/pickup_test.go`: Prop拾得テストのアサーションを更新し、PropにItemコンポーネントが付与されないことを確認するようにした

## コメント

- Phase 1では、sed一括置換で enum 変数と struct 型名が衝突する問題が発生した。`gc.LocationInBackpack` が enum 変数と struct 型の両方に使われていたため、enum 変数を `LocationType*` プレフィックスに変更して解決した
- Phase 2でNullComponent → SliceComponentに変わるため、Joinクエリのパフォーマンス特性が微妙に変わる可能性がある。ただし現在のエンティティ数（数百〜数千）では無視できる差である
- セーブデータの後方互換性は破壊される。`LocationInPlayerBackpack` (MarkerComponent) → `LocationInBackpack` (OwnerRef付き) への形式変更のため、旧セーブデータはロードできない
- 将来の拡張として `LocationInShop`, `LocationInStorage` などの追加が容易になる。追加時は新しいSliceComponent/NullComponentを定義し、`setLocation()` の排他制御に追加するだけでよい
