# Propメカニクス設計

## 背景

現在Propはマーカーコンポーネント（`Prop struct{}`）で、blockPass/blockViewとスプライト表示のみの機能を持つ。アイテムとの差がほぼなく、ランダム配置しても「邪魔な装飾」にしかならない。Propにゲームメカニクスを与える。

## 核心: PropはItemの「設置状態」である

PropとItemは排他的なカテゴリではない。同じエンティティが「フィールドに設置されているか」「インベントリにあるか」で振る舞いが変わる。

```
[設置状態]                    [所持状態]
フィールド上のProp  ──拾う──→  インベントリ内のItem
  - blockPass/blockView        - 使用/装備
  - 移動コスト修正             - 分解
  - 光源                       - 売却
  - 破壊可能（攻撃で壊せる）
        │
        ↓ 破壊
  ドロップアイテム出現
```

CDDAも同じ設計思想を持つ。家具には`deployed_item`（設置元アイテム）と`base_item`（基底アイテム）があり、家具とアイテムは相互変換できる。

### PropがItemと異なる唯一の点

**フィールドに置かれたとき環境に影響を与える。**

| 環境への影響 | 設置状態（Prop） | 所持状態（Item） |
|---|---|---|
| 通行を塞ぐ | blockPass | なし（コンポーネントは保持するが、GridElementがないため無効） |
| 視線を遮る | blockView | なし（同上） |
| 通行コスト加算 | passCost | なし（同上） |
| 光源 | lightSource | なし（同上） |
| 破壊→ドロップ | breakable + dropGroup | 分解 |

拾う・設置するの操作で状態が切り替わる。

## 追加する3つのメカニクス

### 1. 破壊可能性（Breakable）

Propに耐久値（HP）を与え、プレイヤーや敵の攻撃で破壊できるようにする。

```
[TOMLデータ例]
[[props]]
name = "crate"
breakable = { hp = 20 }
```

- Propに対して攻撃コマンドを使うと、武器のダメージ分HPが減少する
- HP 0で破壊される
- 破壊時にドロップアイテムを出す（後述）
- 破壊不能なPropはbreakableを省略する（ドア、ワープポータルなど構造物）

CDDAとの違い: CDDAのstr_min/str_max累積モデルは複雑すぎる。ruinsは既存の攻撃ダメージシステムをそのまま使い、HP制にする。

### 2. 破壊時ドロップ（DropGroup）

破壊されたPropがアイテムをドロップする。探索のインセンティブになる。

```
[TOMLデータ例]
[[props]]
name = "crate"
breakable = { hp = 20 }
dropGroup = "crate_loot"
```

- dropGroupはアイテムテーブル名を参照する
- 木箱を壊すと中から素材やアイテムが出る
- 樽を壊すと回復アイテムが出る
- dropGroupが未設定なら何もドロップしない

CDDAとの違い: CDDAのbase_itemからのuncraft分解は実装しない。ドロップテーブルで直接定義する。

### 3. 通行コスト加算（PassCost）✅ 実装済み

Prop上を通過する際に移動APコストを加算する。blockPass=falseでも通りにくいPropを表現できる。

```
[TOMLデータ例]
[[props]]
name = "carpet"
passCost = 0     # コスト変化なし
blockPass = false

[[props]]
name = "rubble"
passCost = 50    # ベースコスト+50
blockPass = false
```

- passCostはベース移動コスト（StandardActionCost=100）への絶対値加算で表現する
- 0なら通常通り、50ならコスト150
- blockPass=trueのPropでは設定不可（バリデーションエラー）
- キャラクター側のCharModifiers.MoveCost倍率と組み合わせて最終コストが決まる

## 拾う・置く操作

フィールド上のアイテムやPropを拾い、バックパック内のアイテムをフィールドに置く。ItemもPropも同じ「拾う」「置く」操作で扱う。隣接タイルへの方向指定で操作する。

### 操作フロー

#### 拾う（フィールド → バックパック）

```
[DungeonState]
    │ Gキー押下
    ▼
[PickupState] ← モーダルステート
    │ 方向キー（8方向）で対象タイルを選択
    │ 同タイル上のアイテム/Propを一覧表示
    │ Enter で拾得実行
    │ Escape でキャンセル → DungeonStateに戻る
    ▼
[PickupActivity実行]
    │ 対象タイルのアイテム/Propをバックパックに移動
    │ Propの場合はProp→Item変換してから移動
    ▼
[DungeonState]
```

#### 置く（バックパック → フィールド）

```
[DungeonState]
    │ Pキー押下
    ▼
[PlaceState] ← モーダルステート
    │ バックパック内のアイテム一覧を表示
    │ Tab でアイテムを選択
    │ 方向キー（8方向）で配置先タイルを選択
    │ Enter で配置実行
    │ Escape でキャンセル → DungeonStateに戻る
    ▼
[DropActivity実行]
    │ バックパックからアイテムを取り出す
    │ 指定タイルにアイテムとして配置する
    │ Pickable付きアイテムの場合はItem→Prop変換して配置する
    ▼
[DungeonState]
```

### PickupState の設計

ShootingStateと同じモーダルステートパターンを使う。

```
状態:
  selectedDirection: Direction  // 現在選択中の方向（初期値: DirectionNone = 足元）
  itemsAtTarget: []ecs.Entity  // 選択方向のタイルにあるアイテム/Prop一覧

キー操作:
  方向キー（8方向） → selectedDirectionを更新、itemsAtTargetを再取得
  Enter             → itemsAtTargetの全アイテムを拾得
  Escape            → キャンセルしてDungeonStateに戻る

表示:
  - 選択中のタイルにカーソルを描画する（ShootingStateのターゲットカーソルと同様）
  - 情報パネルに拾得可能なアイテム名一覧を表示する
  - 拾えるものがない場合はその旨を表示する
```

### PlaceState の設計

```
状態:
  selectedItem: int             // バックパック内の選択中アイテムインデックス
  selectedDirection: Direction  // 配置先の方向

キー操作:
  Tab               → バックパック内のアイテムを巡回選択する
  方向キー（8方向） → selectedDirectionを更新
  Enter             → 配置先が有効ならDropActivityを実行
  Escape            → キャンセル

検証:
  - 配置先タイルにblockPass=trueのエンティティがないことを確認する
```

### 拾得対象の判定

拾えるものは2種類ある。

| 対象 | 条件 | 処理 |
|---|---|---|
| フィールドアイテム | Item + ItemLocationOnField + GridElement | そのままバックパックに移動する。既存のPickupActivityと同じ処理である |
| 拾えるProp | Prop + !Interactable + GridElement | Itemコンポーネントを追加してバックパックに移動する |

すべてのPropが拾えるわけではない。Interactableを持つProp（ドア、樽など攻撃対象、ワープポータル）は拾えない。Interactableを持たないPropが拾得対象となる。

### Prop ↔ Item 変換

設計時にはPickableコンポーネントでProp定義名を保持し、コンポーネントの除去/復元で変換する案だったが、実装ではより単純な方式を採用した。

#### 実装した方式: コンポーネント保持

Prop系コンポーネント（Prop, BlockPass, BlockView, PassCost等）は拾っても削除せず保持する。GridElementがないためフィールド上では無効となり、バックパック内で副作用は生じない。

#### 拾う（フィールド → バックパック）

1. Itemコンポーネントを追加する（未付与の場合）
2. ItemLocationInPlayerBackpackに移動する
3. GridElementを削除する（フィールドから消える）
4. Prop, BlockPass, BlockView等はそのまま保持する

#### 置く（バックパック → フィールド）

1. ItemLocationOnFieldに移動する
2. 指定座標のGridElementを追加する
3. Prop, BlockPass等が保持されているため、フィールドに置いた時点で自動的にPropとして機能する

#### Pickableコンポーネントは不採用

設計時にはPickableコンポーネントを導入してProp定義名を保持し、置くときにコンポーネントを復元する案だった。しかしコンポーネントを保持する方式にしたことで復元が不要になり、Pickableコンポーネントも不要になった。拾えるかどうかの判定は `Prop && !Interactable` で十分である。

### 既存Activityの変更

#### PickupActivity

- ActionParams.Destination にターゲットタイル座標を渡す。Destinationは必須である
- performPickupActivity で Destination の座標にあるアイテム/Propを検索する
- `Prop && !Interactable` の条件でPropの拾得判定を行う
- collectPropメソッドでPropをItem化してバックパックに移動する。Prop系コンポーネントは保持する

#### DropActivity

- ActionParams.Destination に配置先タイル座標を渡す。Destinationは必須である
- Prop由来アイテムもそのままフィールドに配置する。コンポーネントが保持されているため、特別な変換処理は不要である

### 分解する（所持状態での破壊）

- 未実装
- インベントリ内のアイテムを分解して素材を得る
- dropGroupと同じテーブルを参照する
- 設置して攻撃で壊す場合と同じ結果が得られる

## 採用しなかった機能

| 機能 | 不採用の理由 |
|---|---|
| 押して移動 | STR判定と衝突処理が複雑。探索ループへの寄与が薄い |
| 収納（max_volume） | アイテムがタイル上に直接存在するruinsの設計と合わない |
| 開閉 | 既にDoorコンポーネントで実装済み |
| comfort/睡眠 | ruinsに休息メカニクスがない |
| 作業台 | ruinsにクラフトシステムがない |
| 可燃性 | 火災システムがない |

## データ構造の変更

OpenAPIスキーマに以下を追加する。

```yaml
# Propに追加するフィールド
breakable:
  type: object
  properties:
    hp:
      type: integer
      description: 耐久値
  required: [hp]

dropGroup:
  type: string
  description: 破壊時ドロップおよび分解時のアイテムテーブル名

passCost:
  type: integer
  default: 0
  description: 通行コスト加算値
```

ECSコンポーネントに追加する。

```go
// Breakable は破壊可能なエンティティを表す
type Breakable struct {
    HP Pool // 耐久値
}

// DropGroup は破壊・分解時にアイテムをドロップするエンティティを表す
type DropGroup struct {
    TableName string // アイテムテーブル名
}

// PassCost はフィールド上のタイルの移動コストを修正する。
// ベースコストへの加算値で表現する
type PassCost struct {
    Value int // 移動コスト加算値。0で変化なし、50でコスト+50
}
```

## 実装の優先順位

段階的に実装する。各段階で動作確認できる。

1. **PassCost**: ✅ 実装済み。移動システムにPropチェックを1箇所追加するだけ
2. **Breakable**: ✅ 実装済み。PropにHP+MeleeInteractionを付与し、既存の攻撃システムで破壊可能にした。専用のBreakableInteractionではなくMeleeInteractionを再利用する設計に変更した
3. **DropGroup**: 未実装。Breakable完成後に追加。既存のアイテムスポーンシステムを再利用する
4. **拾う/置く**: ✅ 実装済み。PickupState/PlaceStateモーダルステート + PickupActivity/DropActivity拡張。Pickableコンポーネントは不採用とし、コンポーネント保持方式で実装した

## ランダム配置への影響

メカニクスが実装されると、PropPlannerのランダム配置が意味を持つようになる。

- 木箱/樽: 壊すとアイテムがドロップする。探索のインセンティブ
- 瓦礫: 移動コスト増加で戦術的な地形効果を生む
- ランタン/蝋燭: 光源として暗闇フロアで重要。壊すと暗くなる。拾って持ち運べる

## 進捗

- PassCost: 実装済み
- Breakable: 実装済み。PropにHP+MeleeInteractionを付与する方式。Spaceメニューから「攻撃する」で破壊できる。dead_cleanup_systemで破壊後のエンティティ除去も実装済み
- DropGroup: 未実装
- 拾う/置く: 実装済み。PickupState（Gキー）/PlaceState（Pキー）モーダルステート + PickupActivity/DropActivity拡張

## 設計時の想定と実装の差異

| 項目 | 設計時 | 実装 |
|---|---|---|
| 拾得判定 | Pickableコンポーネント | `Prop && !Interactable` |
| Prop→Item変換 | コンポーネント除去+復元 | コンポーネント保持（GridElement除去のみ） |
| Item→Prop変換 | Pickable.PropDefNameでコンポーネント復元 | 不要（コンポーネントが保持されている） |
| 足元BlockPass制限 | 検討なし | 検討したが不採用。足元にBlockPassのPropを置いても移動は制限しない |

コンポーネント保持方式にしたことで、Pickableコンポーネントが不要になり、変換ロジックも大幅に単純化された。

## コメント

- Breakableの攻撃対象選択は実装済み。Spaceメニュー（action_handlers.go）でMeleeInteractionを持つPropが「攻撃する(名前)」として選択肢に表示される
- セーブ/ロードにBreakable.HPの永続化が必要になる
- PropPlannerの再有効化はPassCost実装後が適切である
- PickupState/PlaceStateはShootingStateのモーダルパターンを踏襲した。方向キーでタイル選択（チェビシェフ距離1以内）、Enter確定、Escapeキャンセルの統一UI
- Interactableの有無で拾えるPropと拾えないPropを区別する。ドア・ポータル・破壊可能オブジェクトなどInteractableを持つ構造的Propは拾えない
- PickupActivity/DropActivityのDestinationは必須引数に変更した。足元操作の後方互換性は不要になった
- ItemもPropも同じ「拾う」「置く」で操作する。PlaceActivityは新設せず、DropActivityをそのまま使う
