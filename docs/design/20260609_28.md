# Propメカニクス設計

## 背景

現在Propはマーカーコンポーネント（`Prop struct{}`）で、blockPass/blockViewとスプライト表示のみの機能を持つ。アイテムとの差がほぼなく、ランダム配置しても「邪魔な装飾」にしかならない。Propにゲームメカニクスを与える。

## 核心: PropはItemの「設置状態」である

PropとItemは排他的なカテゴリではない。同じエンティティが「フィールドに設置されているか」「インベントリにあるか」で振る舞いが変わる。

```
[設置状態]                    [所持状態]
フィールド上のProp  ──拾う──→  インベントリ内のItem
  - blockPass/blockView        - 使用/装備
  - 移動コスト修正             - 分解
  - 光源                       - 売却
  - 破壊可能（攻撃で壊せる）
        │
        ↓ 破壊
  ドロップアイテム出現
```

CDDAも同じ設計思想を持つ。家具には`deployed_item`（設置元アイテム）と`base_item`（基底アイテム）があり、家具とアイテムは相互変換できる。

### PropがItemと異なる唯一の点

**フィールドに置かれたとき環境に影響を与える。**

| 環境への影響 | 設置状態（Prop） | 所持状態（Item） |
|---|---|---|
| 通行を塞ぐ | blockPass | なし |
| 視線を遮る | blockView | なし |
| 通行コスト加算 | passCost | なし |
| 光源 | lightSource | なし |
| 破壊→ドロップ | breakable + dropGroup | 分解 |

拾う・設置するの操作で状態が切り替わる。

## 追加する3つのメカニクス

### 1. 破壊可能性（Breakable）

Propに耐久値（HP）を与え、プレイヤーや敵の攻撃で破壊できるようにする。

```
[TOMLデータ例]
[[props]]
name = "crate"
breakable = { hp = 20 }
```

- Propに対して攻撃コマンドを使うと、武器のダメージ分HPが減少する
- HP 0で破壊される
- 破壊時にドロップアイテムを出す（後述）
- 破壊不能なPropはbreakableを省略する（ドア、ワープポータルなど構造物）

CDDAとの違い: CDDAのstr_min/str_max累積モデルは複雑すぎる。ruinsは既存の攻撃ダメージシステムをそのまま使い、HP制にする。

### 2. 破壊時ドロップ（DropGroup）

破壊されたPropがアイテムをドロップする。探索のインセンティブになる。

```
[TOMLデータ例]
[[props]]
name = "crate"
breakable = { hp = 20 }
dropGroup = "crate_loot"
```

- dropGroupはアイテムテーブル名を参照する
- 木箱を壊すと中から素材やアイテムが出る
- 樽を壊すと回復アイテムが出る
- dropGroupが未設定なら何もドロップしない

CDDAとの違い: CDDAのbase_itemからのuncraft分解は実装しない。ドロップテーブルで直接定義する。

### 3. 通行コスト加算（PassCost）✅ 実装済み

Prop上を通過する際に移動APコストを加算する。blockPass=falseでも通りにくいPropを表現できる。

```
[TOMLデータ例]
[[props]]
name = "carpet"
passCost = 0     # コスト変化なし
blockPass = false

[[props]]
name = "rubble"
passCost = 50    # ベースコスト+50
blockPass = false
```

- passCostはベース移動コスト（StandardActionCost=100）への絶対値加算で表現する
- 0なら通常通り、50ならコスト150
- blockPass=trueのPropでは設定不可（バリデーションエラー）
- キャラクター側のCharModifiers.MoveCost倍率と組み合わせて最終コストが決まる

## 拾う・設置する操作

PropとItemの状態遷移を可能にする。

### 拾う（Prop → Item）

- フィールド上のPropに対して「拾う」コマンドを使う
- Propコンポーネントが外れ、Itemコンポーネントが付与される
- フィールドからの環境効果（blockPass, 光源など）が消える
- インベントリに入る

### 設置する（Item → Prop）

- インベントリ内のアイテムに対して「設置」コマンドを使う
- 隣接タイルを選択して配置する
- Itemコンポーネントが外れ、Propコンポーネントが付与される
- フィールドでの環境効果が発生する

### 分解する（所持状態での破壊）

- インベントリ内のアイテムを分解して素材を得る
- dropGroupと同じテーブルを参照する
- 設置して攻撃で壊す場合と同じ結果が得られる

## 採用しなかった機能

| 機能 | 不採用の理由 |
|---|---|
| 押して移動 | STR判定と衝突処理が複雑。探索ループへの寄与が薄い |
| 収納（max_volume） | アイテムがタイル上に直接存在するruinsの設計と合わない |
| 開閉 | 既にDoorコンポーネントで実装済み |
| comfort/睡眠 | ruinsに休息メカニクスがない |
| 作業台 | ruinsにクラフトシステムがない |
| 可燃性 | 火災システムがない |

## データ構造の変更

OpenAPIスキーマに以下を追加する。

```yaml
# Propに追加するフィールド
breakable:
  type: object
  properties:
    hp:
      type: integer
      description: 耐久値
  required: [hp]

dropGroup:
  type: string
  description: 破壊時ドロップおよび分解時のアイテムテーブル名

passCost:
  type: integer
  default: 0
  description: 通行コスト加算値
```

ECSコンポーネントに追加する。

```go
// Breakable は破壊可能なエンティティを表す
type Breakable struct {
    HP Pool // 耐久値
}

// DropGroup は破壊・分解時にアイテムをドロップするエンティティを表す
type DropGroup struct {
    TableName string // アイテムテーブル名
}

// PassCost はフィールド上のタイルの移動コストを修正する。
// ベースコストへの加算値で表現する
type PassCost struct {
    Value int // 移動コスト加算値。0で変化なし、50でコスト+50
}
```

## 実装の優先順位

段階的に実装する。各段階で動作確認できる。

1. **PassCost**: ✅ 実装済み。移動システムにPropチェックを1箇所追加するだけ
2. **Breakable**: ✅ 実装済み。BreakableInteraction + BreakPropActivityで戦闘システムと接続
3. **DropGroup**: Breakable完成後に追加。既存のアイテムスポーンシステムを再利用する
4. **拾う/設置する**: PropとItemの状態遷移。UIにコマンド追加が必要

## ランダム配置への影響

メカニクスが実装されると、PropPlannerのランダム配置が意味を持つようになる。

- 木箱/樽: 壊すとアイテムがドロップする。探索のインセンティブ
- 瓦礫: 移動コスト増加で戦術的な地形効果を生む
- ランタン/蝋燭: 光源として暗闇フロアで重要。壊すと暗くなる。拾って持ち運べる

## コメント

- Breakableの攻撃対象選択UIは別途設計が必要である。現在の攻撃システムは敵NPCのみを対象としている
- セーブ/ロードにBreakable.HPの永続化が必要になる
- PropPlannerの再有効化はPassCost実装後が適切である
- 拾う/設置する操作はECSのコンポーネント付け替えで実現できる。大きなアーキテクチャ変更は不要である
