# worldhelperパッケージの責務分割

## 背景

`internal/worldhelper` は108個の公開関数を25ファイルに分散し、53種類のコンポーネントにアクセスする。コードベース全体の92ファイルから参照されている。典型的なshallow moduleで、インターフェースの複雑さ（108関数）が実装の深さ（各関数は平均25行）を大幅に上回る。全関数が `world w.World` を受け取るため、実際に何が必要かが隠蔽されている。

「アイテム移動のコードはどこ？」と探すとき108関数から選ぶ必要があり、変更の影響範囲も見えない。

### 要件

- worldhelperを責務ごとに分割し、各パッケージのインターフェースを狭くする
- 循環依存を発生させない
- 全テストが通る状態を維持する
- 段階的に移行可能な構造にする

### 要件外

- 関数シグネチャの変更。パッケージ移動のみ行い、APIは変えない
- worldhelperの互換レイヤー。re-exportは作らず、呼び出し元のimportを全て書き換える

## 設計

### 方針: 操作の性質による3パッケージ分割

worldhelperの108関数を**操作の性質**で3パッケージに分割する。コンポーネントのドメイン（アイテム系、戦闘系など）で分けると、`GetWeapons` のように複数ドメインにまたがる関数の帰属が曖昧になる。代わりに「何をする操作か」で分類する。

- **query**: 状態の読み取り・単純な状態変更・純粋計算
- **lifecycle**: エンティティの生成・削除・配置
- **action**: query + lifecycle を組み合わせた複合的なゲームロジック

配置先は `internal/world/` のサブパッケージとする。`internal/world/world.go` の `World` 型を各サブパッケージがimportする方向で、逆方向のimportはないため循環しない。

### 分類ルール

各パッケージへの分類は以下のルールで判定する。

1. 関数の実装が `NewEntity` / `DeleteEntity` / `AddEntities` を呼ぶか、エンティティの配置（Location系コンポーネントの排他切り替え）を行う → **lifecycle**
2. lifecycle関数と密結合している（lifecycle関数から呼ばれ、分離すると循環依存になる）→ **lifecycle**
3. 複数の query/lifecycle 操作を組み合わせてゲームルールを実現する → **action**
4. それ以外（コンポーネントの読み取り・単一コンポーネントの変更・純粋計算）→ **query**

ルール2は循環依存回避のための実用的な判定。例えば `FullRecover` はコンポーネント変更のみだが、`SpawnPlayer`（lifecycle）から呼ばれるため lifecycle に配置する必要がある。

### 内部依存の分析

分割にあたり、worldhelper内の関数間呼び出しを網羅的に洗い出した。

エンティティ生成チェーン（lifecycle内で閉じる）:
```
ChangeStackableCount → SpawnBackpackItem → MoveToBackpack → mergeStackableItems → GetEntityCount, ChangeItemCount
SpawnPlayer → FullRecover → setMaxStats, CalculateMaxActionPoints
SpawnEnemy → FullRecover, CalculateMaxActionPoints
```

ゲームロジックチェーン（action → query + lifecycle）:
```
BuyItem → HasCurrency, ConsumeCurrency (query) + ChangeStackableCount, SpawnBackpackItem (lifecycle)
SellItem → GetItemValue (query) + ChangeItemCount (lifecycle) + AddCurrency (query)
Craft → GetPlayerEntity (query) + SpawnBackpackItem (lifecycle)
ApplyDamage → GetEntityName, GetGameLog (query)
ApplyHealing → SpawnVisualEffect (lifecycle)
ApplyProfession → FullRecover (lifecycle) + SpawnBackpackItem (lifecycle) + MoveToEquip (lifecycle)
PreviewEndRun → MoveToBackpack (lifecycle) + GetEntityCount, CalculateSellPrice (query)
ExecuteEndRun → ApplyProfession (action, 同パッケージ内)
```

クエリチェーン（query内で閉じる）:
```
CanPlayerAct → GetTurnState, GetPlayerEntity
RestoreAllActionPoints → CalculateMaxActionPoints, CalculateSpeed
GetVisibleEnemies → GetPlayerEntity, IsInVision → GetDungeon
FormatItemName → GetEntityCount
GetEntityWeight → GetEntityCount
UpdateWeightCapacity → GetEntityWeight
LockAllDoors → CloseDoor
UnlockAllDoors → OpenDoor
```

### 循環依存の回避

分類の境界で循環依存が発生するケースを特定し、対策した。

**FullRecover の配置**: FullRecover はコンポーネント変更のみ（HP/APの再設定）でCRUDではないが、`SpawnPlayer`, `SpawnEnemy`, `SpawnNeutralNPC`（いずれもlifecycle）から呼ばれる。action に移すと `BuyItem`（action）→ `SpawnBackpackItem`（lifecycle）の方向と合わせて lifecycle ⇄ action の循環が発生するため、lifecycle に配置する（ルール2を適用）。

**ApplyProfession の配置**: lifecycle 関数（FullRecover, SpawnBackpackItem, MoveToEquip）を組み合わせるが、lifecycle 側からは呼ばれない。action に安全に配置できる。

**ドア操作の配置**: OpenDoor, CloseDoor はコンポーネント変更のみ（Door, BlockPass, BlockView, SpriteRender）でCRUDではない。lifecycle 側（SpawnDoor等）から呼ばれていないため query に配置する。LockAllDoors, UnlockAllDoors も同様。

### 依存グラフ

3パッケージ間の依存はリニアチェーンを形成し、循環しない。

```
world/query (0 deps)
    ↑
world/lifecycle (→ query)
    ↑
world/action (→ query + lifecycle)
```

lifecycle → query の依存:
- FullRecover → CalculateMaxActionPoints
- SpawnEnemy → CalculateMaxActionPoints
- ChangeStackableCount → FindStackableInInventory, GetEntityCount
- mergeStackableItems → GetEntityCount, ChangeItemCount

action → query の依存:
- ApplyDamage → GetEntityName, GetGameLog, AppendNameWithColor
- BuyItem → HasCurrency, GetCurrency, ConsumeCurrency
- Craft → GetPlayerEntity, FindStackableInInventory, GetEntityCount
- PreviewEndRun → GetEntityCount, CalculateSellPrice
- ApplyProfession → CalculateMaxActionPoints（FullRecover経由）

action → lifecycle の依存:
- BuyItem → SpawnBackpackItem, ChangeStackableCount
- SellItem → ChangeItemCount
- Craft → SpawnBackpackItem, ChangeStackableCount
- ApplyHealing → SpawnVisualEffect
- ApplyProfession → FullRecover, SpawnBackpackItem, MoveToEquip
- PreviewEndRun → MoveToBackpack
- ExecuteEndRun → DeleteEntity（直接）

### パッケージ詳細

#### 1. `internal/world/query/` — 状態の読み書き・純粋計算（~50関数）

#### 2. `internal/world/lifecycle/` — エンティティの生成・削除・配置（~21関数）

エンティティの作成（AddEntities, NewEntity）、削除（DeleteEntity）、配置（Location系コンポーネントの排他的切り替え）を担う。queryに依存する（CalculateMaxActionPoints, FindStackableInInventory, GetEntityCount等）。

| 移動元ファイル | 関数 |
|---|---|
| spawn.go（キャラクター部分） | SpawnTile, SpawnPlayer, SpawnNeutralNPC, SpawnEnemy, SpawnEnemyOption, WithBoss |
| spawn.go（アイテム部分） | SpawnBackpackItem, SpawnFieldItem, SpawnStorageItem, FullRecover |
| spawn_prop.go（生成・削除部分） | SpawnProp, SpawnDoor, DeleteDoorLockTriggers |
| damage.go（生成部分） | SpawnVisualEffect |
| state_event.go | RequestStateChange, ConsumeStateChange |
| location.go（配置部分） | MoveToBackpack, MoveToEquip, MoveToField, MoveToStorage |
| inventory.go（変更部分） | ChangeItemCount |
| stackable.go | ChangeStackableCount |

FullRecover はCRUD操作ではないが、SpawnPlayer / SpawnEnemy / SpawnNeutralNPC から呼ばれるため lifecycle に配置する（循環依存回避）。

RequestStateChange（NewEntity）と ConsumeStateChange（DeleteEntity）はイベント機構だが、エンティティのCRUDを行うため lifecycle に配置する。

#### 3. `internal/world/action/` — ゲームロジック（~19関数 + 2型）

query と lifecycle を組み合わせた複合的なゲームルールの実装を担う。query + lifecycle の両方に依存する。

| 移動元ファイル | 関数 |
|---|---|
| damage.go（ロジック部分） | ApplyDamage, ApplyHealing |
| shop.go（ロジック部分） | BuyItem, SellItem, GetShopInventory |
| craft.go | Craft, CanCraft |
| run.go | PreviewEndRun, ExecuteEndRun, AutoSellResult（型）, SoldItem（型） |
| profession.go | ApplyProfession |

ApplyProfession は FullRecover（lifecycle）+ SpawnBackpackItem（lifecycle）+ MoveToEquip（lifecycle）を組み合わせる。lifecycle 側から呼ばれていないため action に安全に配置できる。

CanCraft は読み取り専用だが、Craft と private ヘルパー（requiredMaterials, consumeMaterials）を共有するため action に配置する。

### 移行手順

パッケージ単位で移行する。worldhelperは1パッケージなので、ファイル単位の移行では中間状態でビルドが壊れる。依存の上流（query）から順にパッケージ単位で一気に移行し、各ステップでビルドとテストを実行する。

1. 3パッケージのディレクトリとdoc.goを作成する
2. query を移行する（他パッケージへの依存がないため最初に着手する）
3. lifecycle を移行する（queryに依存。query移行完了後に着手する）
4. action を移行する（query + lifecycleに依存。両方の移行完了後に着手する）
5. worldhelper ディレクトリを削除する
6. `make check` で全テスト・lintが通ることを確認する

各ステップの移行手順:
1. 新パッケージに該当関数をコピーする
2. 92ファイルのimportを新パッケージに切り替える
3. worldhelper側の元関数を削除する
4. ビルド・テストを実行する

### 変更箇所

| ファイル | 内容 |
|---|---|
| `internal/world/query/*.go` | 新規: 状態読み書き・純粋計算・ドア操作 |
| `internal/world/lifecycle/*.go` | 新規: エンティティ生成・削除・配置 |
| `internal/world/action/*.go` | 新規: ゲームロジック・職業適用 |
| `internal/worldhelper/` | 削除 |
| 92ファイルのimport | worldhelper → 各新パッケージへ変更 |

### 採用しなかった方法

| 方法 | 不採用の理由 |
|---|---|
| コンポーネントドメインで5分割（item, combat, commerce等） | `GetWeapons` のように複数ドメインにまたがる関数の帰属が曖昧になる。「アイテム系だからitem」という分類は「worldhelperに関係するからworldhelper」と同じ思考パターンに陥る |
| lifecycle を CRUD のみに厳密に限定 | FullRecover（コンポーネント変更のみ）をactionに移すと、SpawnPlayer（lifecycle）→ FullRecover（action）と BuyItem（action）→ SpawnBackpackItem（lifecycle）で lifecycle ⇄ action の循環依存が発生する。lifecycle側から呼ばれる密結合関数は lifecycle に残す必要がある |
| re-exportによる互換レイヤー | CLAUDE.mdの方針に反する。呼び出し元のimport変更は機械的に可能で、中間レイヤーは不要な複雑さ |
| worldhelperを残してファサードにする | 108関数のre-exportは元の問題（shallow module）を解決しない |
| ファイル単位の段階的移行 | worldhelperは1パッケージなので、1ファイルを移動するだけで同パッケージ内の他関数への参照が壊れる。パッケージ単位の移行が必要 |

## コメント

- queryパッケージの関数数（~50）は多いが、「エンティティのCRUD/配置を行わない」という明確なルールで一貫している。将来的にqueryが肥大化した場合、query内をさらにサブパッケージに分割する余地がある
- `CalculateBuyPrice`, `CalculateSellPrice` は `world w.World` すら受け取らない純粋関数。将来的に既存の `formula` パッケージへの移動を検討してもよい
- lifecycle のルール2（密結合による例外配置）は FullRecover のみに適用される。他の関数はルール1（CRUD/配置）で自然に分類できる
- `addCurrency` を `AddCurrency` として公開することで、`ConsumeCurrency`（通貨減算）と `AddCurrency`（通貨加算）が対称的にquery内に揃う
- ApplyProfession は5分割設計では spawning に配置していたが、3分割では action が適切。lifecycle関数を複数組み合わせる操作であり、lifecycle側から呼ばれていない
- 操作の性質による分類は、新しい関数を追加する際の判断が容易になる。「エンティティを作る/消す/動かす？→ lifecycle」「複数操作を組み合わせる？→ action」「それ以外？→ query」
